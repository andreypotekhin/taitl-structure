### D06172607 Join Semantics

Challenges.md point C9, "Join Semantics Need Sharper Definitions", is resolved by making joins explicit, symbolic, and
cardinality-aware in v1, with row-multiplying `join_many(...)` deferred to v2. The resolved design contract lives in
`docs/dev/design/specifications/JoinSemantics.spec.md`, with the implementation plan in
`docs/dev/planning/P06172602.Join-semantics-definitions.plan.md`.

Join calls require an explicit `how` argument. Structure supports `Join.LEFT` and `Join.INNER` in the v1 compiled DSL,
while right, full, cross, semi, and anti joins are deferred until the row-centric schema model has precise semantics for
them. This keeps every output row explainable as the current row plus optional joined scopes.

`join_one(...)` is a cardinality promise, not a deduplication request. Duplicate right-side rows are a data-quality
problem because Spark would multiply the current row; Structure must not silently choose an arbitrary row. If the right
key is not proven unique, v1 compiles with a warning by default and points users toward `primary_key=True`, future
unique-key metadata, or v2 `join_many(...)` when multiplication is intended.

Key behavior remains visible in source. Composite keys are expressed with equality pairs joined by `&`. Null matching
uses explicit `null_safe_eq(...)`; nullable fields alone do not imply null-safe equality. Case-insensitive or otherwise
normalized keys use compileable expression helpers such as `lower(trim(...))` in the join condition rather than a hidden
join option. This keeps business-key semantics reviewable in generated PySpark and lineage.

Generated joins preserve source order, use deterministic aliases, qualify field references to avoid name collisions,
and project only the right-side fields needed by keys, filters, output projection, lineage, or diagnostics. Broadcast
hints apply to the joined right side only and remain advisory; they must not change row semantics.
