### D06202604 Backend Capability Interface

Challenges.md point C23, "Backend Adaptability Needs a Capability Interface Before More Features", is resolved by
adding a static backend capability interface and PySpark v1 capability profile.

The compiler, online runner, generated PySpark emitter, streaming compatibility checks, and future explain output ask a
`BackendCapabilities` object whether a `CapabilityRequirement` is supported. The requirement names a feature group such
as expression, join, validation, streaming, imports, or backend, plus the feature name inside that group. Supported
requirements return a successful decision. Unsupported requirements fail through `BACKEND-E2401` for unknown backend
targets or `BACKEND-E2402` for unsupported backend capabilities.

The interface is deliberately internal. It is not a new public user API. Public behavior is the clearer diagnostic and
the guarantee that backend support rules are centralized.

Capability selection is Spark-free. It must not import PySpark, start Java, create a Spark session, connect to a Spark
cluster, or inspect the installed runtime. PySpark version behavior is represented as static source metadata selected
from configuration.

This keeps v1 ordinary PySpark focused while leaving room for v2 analytical features, v3 streaming orchestration, and
v4 Spark Connect to add capability profiles without scattering target checks across the codebase.
