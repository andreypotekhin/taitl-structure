### D06212601 Analytical Join Coverage

Challenges.md point C27, "Analytical Join Coverage Is Still Narrow", is resolved by preserving v1 `join_one(...)` as a
narrow lookup primitive and creating a separate v2 analytical join family. The resolved design contract lives in
[AnalyticalJoinCoverage.md](../../../specifications/AnalyticalJoinCoverage.md), with supporting design notes in
[AnalyticalJoinCoverage.md](../AnalyticalJoinCoverage.md) and the implementation plan in
[P06212601.Analytical-join-coverage.plan.md](../../planning/P06212601.Analytical-join-coverage.plan.md).

The v2 join family should be staged by frequency and semantic risk: existence predicates for semi and anti joins,
`join_many(...)` for intentional row multiplication, deterministic dedupe policies for lookup joins, temporal validity
lookups, and as-of lookups. Right, full, and cross joins remain deferred because they do not fit the current
row-centric transform model cleanly.

The first implemented slice admits existence joins, `join_many(...)`, and deterministic deduped `join_one(...)` in the
default PySpark profile. Temporal validity and backward as-of joins remain staged follow-up work.

No analytical join form may bypass the compiler-visible path. Each form must have source syntax, IR representation,
backend capability checks, shared PySpark recipes, diagnostics, traceability, and online/generated parity tests before it is
considered supported. Hooks remain the explicit escape hatch for unsupported join shapes.

Policies that select one right-side row must be deterministic. Structure must not silently use nondeterministic
`first(...)`, `dropDuplicates(...)`, or arbitrary row selection to make lookup joins convenient.
