### D06202601 Online Generated Semantic Contract

Structure resolves C24 by making online PySpark execution and generated PySpark emission consume a shared semantic
target plan. Checked backend-neutral `TransformPlan` IR plus `PySparkCapabilities` lowers to deterministic PySpark
execution recipes. The online runner interprets those recipes with live PySpark DataFrame and Column objects. The
generator renders those same recipes as source text.

The generator owns text concerns such as imports, formatting, file headers, comments, generated paths, and
write-if-changed behavior. The online runner owns live runtime concerns such as DataFrame binding, hook invocation,
and runtime diagnostics. Neither path owns independent decisions about projection order, filter order, expression
mapping, literal typing, join aliases, hook order, validation placement, schema projection, or compiled-path
performance guardrails.

The shared target plan is internal before 1.0. Recommended implementation names are `PySparkExecutionPlan`,
`PySparkStepRecipe`, `PySparkExpressionRecipe`, `PySparkJoinRecipe`, `PySparkValidationRecipe`,
`PySparkHookRecipe`, and `PySparkCapabilities`. Implementations may adjust names to fit local code structure, but not
the responsibility split.

Every new compiled operation must add a recipe shape and an online/generated parity test before the operation is
considered supported. Generated-code snapshots remain useful for review, but runtime parity is the authority for
semantic correctness.
