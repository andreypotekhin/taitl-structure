### D06202602 Data Quality Constraints Boundary

Challenges.md point C26, "Data Quality Constraints Stop at Schema Shape", is resolved by making the validation boundary
explicit. Structure v1 validation is schema-first. Intermediate validation defaults to `schema_only`, which checks
DataFrame shape and must not scan rows.

Richer data-quality constraints are a deferred, opt-in model. Accepted values, ranges, regex-like string checks,
decimal domain rules, uniqueness, referential checks, freshness, and row-count policies must be represented separately
from base schema shape. Any check that can trigger Spark work must be explicit in source or configuration and must
surface its cost in diagnostics.

Generated PySpark schema constants remain ordinary `StructType` values and are supported caller-facing artifacts.
Callers may import them for `spark.read.schema(...)`, runtime validation, and projection before their own writes. Online
execution must expose equivalent materialized Spark schemas after `run(session)`, for example through
`result.schema.enriched`, without requiring generated files to exist. Structure does not add v1 write orchestration;
callers own `write`, `writeStream`, table creation, partitioning, checkpoints, and storage options.

Input, intermediate, and output validation should have symmetric phase modes:
`input_validation_mode`, `intermediate_validation_mode`, and `output_validation_mode`. Future constraints should also
bind to validation phases. A constraint runs only when it is bound to the current phase and that phase allows
`schema_and_constraints`.

Future constraint support should generate separate constraint metadata or runtime helpers. It must not silently change
the meaning of existing `*_SCHEMA` constants.
