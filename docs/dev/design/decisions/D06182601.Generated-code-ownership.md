### D06182601 Generated Code Ownership

Challenges.md point C7, "Generated Code Ownership Rules Need to Be Explicit", is resolved by treating generated
PySpark as committed, reviewable build output owned by the Structure compiler. Generated files are part of the
repository because Structure's value depends on making Spark behavior visible to reviewers, operators, and CI.

Developers must not edit generated files by hand. A desired generated-code change must be made in Structure source,
configuration, or generator implementation, then reproduced by `structure compile`. Manual edits would make the
generated directory a second source of truth and would hide compiler defects that should be fixed at the source.

Generated output must be deterministic and formatted so diffs stay small, stable, and meaningful. Generated-code
changes are reviewed like other build artifacts: reviewers should check shape, imports, Spark APIs used, validation,
and absence of forbidden compiled-path features such as UDFs, RDD operations, `collect`, and `toPandas`.

CI should run `structure compile --fail-on-diff` for projects that commit generated artifacts. A failing diff means
source, configuration, generator behavior, or checked-in generated files are out of sync, and the fix is to regenerate
or correct the generator rather than patching generated files manually.
