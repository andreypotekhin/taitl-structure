### D06182606 No Spark Dependency During Compile

Challenges.md point C15, "Need a No Spark Dependency During Compile Rule", is resolved by making Spark-free
compilation a hard compiler contract. `structure check`, `structure compile`, `structure compile --fail-on-diff`, and
compiler tests must run without a SparkSession, Spark cluster, Java runtime, or `pyspark` import.

The compiler owns discovery, source inspection, schema extraction, symbolic execution, compileability checks,
backend-neutral IR construction, PySpark text emission, compiler traceability metadata emission, and generated-file diff
checking. Those phases use Structure DSL types and compiler-owned capability metadata. They may emit PySpark source
text, but they must not import PySpark to decide whether source is valid.

Spark belongs to runtime and integration boundaries. Generated PySpark modules may import PySpark. Generated-code import
tests may require PySpark. PySpark execution tests may require Java and a local Spark runtime. Those tests must be
separate from compiler tests so ordinary library builds and CI checks can validate Structure source quickly.

PySpark version differences are modeled through an internal emitter capability registry or equivalent static metadata,
not through probing a live Spark installation during compile. Any diagnostic that depends on configured target PySpark
behavior should cite the target version/configuration, not the developer machine's installed Spark runtime.

This decision keeps compiler feedback fast, makes CI easier to provision, avoids local Java/Spark setup friction, and
prevents discovery or checks from accidentally executing Spark-side effects. It also forces a cleaner boundary: runtime
Spark behavior is verified in generated-code and PySpark execution tests, while compiler correctness is verified without
Spark.
