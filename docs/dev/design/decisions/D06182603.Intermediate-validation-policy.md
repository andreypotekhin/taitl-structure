### D06182603 Intermediate Validation Policy

Challenges.md point C10, "Intermediate Validation May Be Expensive", is resolved by keeping intermediate validation
enabled by default while making its runtime cost explicit and configurable.

Generated code validates intermediate subtransform boundaries by default because method return annotations declare
real pipeline contracts. The default mode is `intermediate_validation_mode = "schema_only"`, which checks schema shape
only: column names, data types, nullable flags where Spark exposes them reliably, nested struct shape, and missing or
extra columns. This default must not trigger row scans.

Projects may set `intermediate_validation_mode = "schema_and_constraints"` when they intentionally want fuller
runtime checks, including row-level constraints once Structure supports them. This mode is opt-in because it may add
Spark actions or otherwise increase runtime work.

Projects may set `validate_intermediate = false` to disable intermediate schema validation at generated subtransform
boundaries. This setting does not disable compile-time schema flow checks, field existence checks, or type checks.
Class-level and method-level validation overrides remain available for narrower exceptions, but project configuration
is the blunt instrument for very large or already externally validated pipelines.
