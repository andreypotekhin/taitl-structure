### D06202603 Diagnostics Registry Contract

Structure resolves C29 by treating diagnostics as a registry-backed public contract. Each diagnostic has a stable code,
severity, title, component prefix, lifecycle status, suggested fix, and documentation link. Feature phases supply
concrete context such as source location, transform name, config setting, hook, join, generated path, or runtime input.

Codes use `{component}-{severity}{number}`, for example `CONF-E0101`, `DSL-W0403`, `GEN-E0901`, or `ONLINE-E1001`.
Component prefixes are assigned by feature area in `docs/specifications/Diagnostics.md`. The public documentation
target for released codes is `docs/Diagnostics.md#conf-e0101`, with deeper links to feature specifications when needed.

The registry must exist before broad diagnostic implementation so duplicate codes, missing docs links, malformed codes,
and unknown component prefixes can fail tests early. Renderers such as the CLI, runtime exceptions, CI annotations, and
future IDE integrations may format diagnostics differently, but they must preserve code, severity, title, problem,
suggested fix, and documentation link.

Published active codes must not be reused for a different meaning. If behavior changes after release, the old code is
deprecated or retired with a stable anchor, and the new behavior receives a new code.
