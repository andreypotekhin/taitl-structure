### D06302601 Alternative Backend Extension Boundary

Structure will keep PySpark as the v1 target while designing the compiler boundary so future backends can be added by
target adapter, not by PySpark conditionals scattered through discovery, symbolic execution, IR validation, CLI,
StructureTools, or diagnostics.

The portability promise applies to compiler-visible Structure source only: schemas, compiled subtransforms, expression
helpers, joins, filters, validation, and backend-neutral IR. Hooks are target-specific opaque runtime code. Hook
decorators may declare `target_backend`, and unmarked hooks inherit a configurable `hook_target_default`. Runtime
execution must never invoke a hook against a backend outside its effective target set.

Future targets should be admitted through capability profiles and target adapters. The roadmap is Python-hosted: v2
prioritizes PySpark-family targets such as Spark SQL and typed PySpark DataFrame patterns, v3 adds Polars LazyFrame and
DuckDB, and v4 adds Ibis. Other targets should come through Ibis when practical. Dask DataFrame and Ray Dataset remain
out of scope until after the relational core is stable.

Compatibility checks belong in both CLI and `StructureTools`. Active-target unsupported features fail fast before
online execution or generation. Multi-target checks produce structured reports with supported, unsupported, degraded,
opaque, and unknown features.
