### D06172606 Nullability and Type Coercion

Challenges.md point C3, "Nullability and Type Coercion Rules Are Missing", is resolved by adopting Spark-aligned
nullability and assignment rules for v1. Structure records Spark SQL assumptions with Spark-native dotted keys directly
under `[tool.structure]`:

```toml
spark.sql.ansi.enabled = true
spark.sql.storeAssignmentPolicy = "ANSI"
```

The default policy accepts Spark-ANSI-compatible widening and typed literals when they do not hide business intent, but
requires explicit helpers for semantic parsing conversions such as string-to-decimal and string-to-timestamp. Nullable
expressions cannot feed non-nullable output fields unless a visible guard or expression, such as `where(...)` or
`coalesce(...)`, repairs the nullability. Structure source may use ordinary Python literals in expression contexts, and
generated PySpark may lower those literals to `F.lit(...)` where needed. Structure includes both `Float()` and
`Double()` schema types; Python `float` literals infer `Double()` by default because Python floats are double precision.

This balances PySpark ergonomics with readable data-quality intent. Full semantics are specified in
[NullabilityAndTypeCoercion.md](../../../specifications/NullabilityAndTypeCoercion.md); the implementation plan is
[P06172601.Nullability-and-type-coercion-rules.plan.md](../../planning/done/P06172601.Nullability-and-type-coercion-rules.plan.md).
