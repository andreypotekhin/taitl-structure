# Extend Plain Python Expression Support

This ExecPlan is a living document. Keep `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` up to date as work proceeds.

Follow `docs/dev/auto/Plans.md` from the repository root.

## Purpose / Big Picture

Structure already lets users write compact compiler-visible expressions such as `is_large=total > 1000`. That support
is useful but incomplete: the specs mention `upper(...)`, `when(...).otherwise(...)`, and broader comparisons, while
the implementation currently supports only a narrow subset.

After this change, users can write richer row-local derived fields without falling back to PySpark hooks, for example:

    tier=when(total >= 1000, "large").otherwise("standard")
    is_discounted=discount > 0
    gross_total=total + tax
    customer_id=upper(trim(order.customer_id))

The observable result is that these expressions compile to shared PySpark recipes, render into generated PySpark, run
online through the same recipe layer, and remain Spark optimizer-visible.

## Surprises & Discoveries

- Observation: plain Python expression support is operator overloading on `Expression`, not AST parsing.
  Evidence: `src/structure/app/dsl/model/expr/Expression.py` implements the symbolic operators.

- Observation: the implementation already has a strong shared semantic target plan.
  Evidence: `PySparkExpressionMapper`, `RenderPySparkExpression`, and `PySparkExpressionEvaluator` all consume
  `PySparkExpressionRecipe`.

- Observation: the documented surface is ahead of implementation.
  Evidence: `docs/specifications/SymbolicExecution.md` and `docs/specifications/NullabilityAndTypeCoercion.md`
  mention `when(...)`; `docs/dev/project-management/sprints/Sprint03_SymbolicExpressionsFilteringHelpers.md` lists
  `upper` and `when` as in-scope.

## Decision Log

- Decision: Complete the already-specified v1 expression surface before adding broad PySpark-like helpers.
  Rationale: This follows the project's "small Structure semantics, not a PySpark mirror" design and minimizes risk.
  Date/Author: 2026-07-01 / Codex

- Decision: Add `upper`, `when(...).otherwise(...)`, remaining ordinary comparisons, and basic `+` and `*`.
  Rationale: These unlock common classification, flags, and numeric derived fields while staying row-local and
  optimizer-visible.
  Date/Author: 2026-07-01 / Codex

- Decision: Defer division, modulo, unary numeric operators, string predicates, and date/timestamp parsing.
  Rationale: Those need more specific result typing, null semantics, or helper design to avoid accidental PySpark API
  cloning.
  Date/Author: 2026-07-01 / Codex

## Outcomes & Retrospective

Implemented the v1 plain expression extension. Structure now captures `upper(...)`, `when(...).otherwise(...)`, `<`,
`<=`, `>=`, `+`, and `*` as compiler-visible expressions, validates `where(...)` and `join_one(on=...)` predicates as
boolean expressions, and lowers the new expression forms through the shared PySpark recipe layer for online and
generated execution.

Focused symbolic, generated-rendering, online-evaluator, and API snapshot tests pass. `make rigidity` and `make build`
also pass. Deferred follow-up remains the larger expression families listed below: division, modulo, string predicates,
date/timestamp parsing, and collection higher-order helpers.

## Context and Orientation

The expression pipeline starts in `src/structure/app/dsl/model/expr/Expression.py` and
`src/structure/app/dsl/model/expr/expressions.py`.

A compiled subtransform is symbolically executed by
`src/structure/app/compiler/frontend/commands/CompileTransform.py`. During symbolic execution, schema row parameters
are represented by `RowScope` or `InputScope`. Field access returns `Expression` objects. The compiler later turns
returned schema objects into projection assignments.

The PySpark target path is shared:

- `src/structure/app/target/pyspark/logic/mapping/PySparkExpressionMapper.py` maps backend-neutral expressions to
  `PySparkExpressionRecipe`.
- `src/structure/app/target/pyspark/commands/RenderPySparkExpression.py` renders generated PySpark source.
- `src/structure/app/runtime/execution/online/logic/PySparkExpressionEvaluator.py` evaluates the same recipes with live
  PySpark functions and columns.
- `src/structure/app/target/capabilities/logic/rules/PySparkCapabilityRules.py` declares supported backend
  capabilities.

The public DSL exports from `src/structure/__init__.py` through `structure.app.dsl.api`.

## Plan of Work

Extend the expression object model in `Expression.py` with comparison operators `lt`, `le`, `ge`, arithmetic operators
`add` and `mul`, and reverse literal forms for addition, subtraction, and multiplication. Keep division, modulo, and
unary numeric operators deferred.

Extend public helpers in `expressions.py` with `upper(value)` and `when(condition, value).otherwise(fallback)`. The final
conditional expression kind is `when`, with arguments `(condition, value, fallback)`. A bare `when(...)` builder is not
an expression and must not lower to PySpark.

Validate predicates earlier. `where(...)` and `join_one(on=...)` must require boolean Structure expressions, raising
`TypeError` so `CompileTransform` wraps user mistakes in the existing `DSL-E0401` diagnostic path.

Wire every new expression kind through `PySparkExpressionMapper`, `RenderPySparkExpression`, and
`PySparkExpressionEvaluator`. The generated and online paths must consume the same recipe shape.

Export `upper` and `when` from `structure.app.dsl.model.expr`, `structure.app.dsl.api`, and root `structure`.

Add tests for symbolic capture, generated PySpark rendering, online evaluator behavior, and negative predicate cases.
Update docs so the public expression surface matches implementation.

## Validation and Acceptance

Implementation is accepted when all of these are true:

- A transform can compile `upper(trim(order.customer_id))`.
- A transform can compile `when(total >= 1000, "large").otherwise("standard")`.
- A transform can compile `<`, `<=`, `>=`, `+`, and `*` expressions.
- Generated PySpark renders the new expressions using `F.upper`, `F.when(...).otherwise(...)`, and Column operators.
- Online execution evaluates the same `PySparkExpressionRecipe` shapes as generated code.
- `where(...)` rejects non-boolean expressions before target lowering.
- `join_one(on=...)` rejects non-boolean expressions before target lowering.
- Public docs list the supported expression surface accurately.
- Focused tests pass.
- `make build` passes.

Expected representative generated snippets:

    F.upper(F.trim(F.col("orders.customer_id")))
    F.when((F.col("orders.total") >= F.lit(1000)), F.lit("large")).otherwise(F.lit("standard"))
    (F.col("orders.total") + F.col("orders.tax"))
    (F.col("orders.price") * F.col("orders.quantity"))

## Idempotence and Recovery

All code changes are additive or narrowly local. If a test fails, inspect the failing expression kind and follow the
pipeline in order:

1. DSL expression creation.
2. CompileTransform projection/filter/join validation.
3. PySparkExpressionMapper recipe creation.
4. RenderPySparkExpression generated output.
5. PySparkExpressionEvaluator online output.

Do not revert unrelated working-tree changes. If generated examples change unexpectedly, compare whether the source
fixture changed. If source did not change, the renderer likely changed existing output and needs investigation before
updating goldens.
