## Purpose / Big Picture

Structure already records desired behavior through design docs, specifications, user stories, generated fixtures, and
tests. This plan adds a behavior-rigidity layer that makes drift visible from more angles: public examples, checked-in
golden generated output, API snapshots, invariant specifications, concept coverage, metamorphic tests, property-based
tests, differential tests, and compatibility consistency checks.

After this change, a developer can run focused make targets such as `make golden`, `make differential`,
`make metamorphic`, `make concepts`, and `make rigidity` to verify that public examples, generated code, public API
shape, concept coverage, and compatibility commitments still match the intended behavior.

## Surprises & Discoveries

- Observation: Local PySpark is not installed in the ordinary development environment.
  Evidence: `python -c "import importlib.util; print(importlib.util.find_spec('pyspark') is not None)"` printed
  `False` with `PYTHONPATH=src;examples`.

- Observation: `PySpark.render.project()` already returns an ordered path-to-text mapping.
  Evidence: The examples golden output was generated directly from that API, so no separate code-generation path was
  needed for golden tests.

## Decision Log

- Decision: Store example source at `examples/<package>` and package-owned generated output at
  `examples/structure_generated/<package>`.
  Rationale: This avoids duplicated source package names such as `examples/orders/orders` while keeping generated
  output beside the public examples.
  Date/Author: 2026-06-30 / Codex

- Decision: Put generated-output golden tests in `tests/golden`.
  Rationale: Golden tests are a distinct reviewability mechanism and should be runnable directly with `make golden`.
  Date/Author: 2026-06-30 / Codex

- Decision: Use `make golden`, `make differential`, `make metamorphic`, `make concepts`, and `make rigidity`.
  Rationale: Short make targets are easier to remember than `test-*` variants and match existing target style.
  Date/Author: 2026-06-30 / Codex

- Decision: Keep live Spark behavior in integration tests and make the first differential suite Spark-free.
  Rationale: Ordinary `make rigidity` should remain usable without PySpark, Java, Spark startup, or Docker. Live
  behavior remains covered by `make integration BACKEND=pyspark35` and `make integration BACKEND=pyspark40`.
  Date/Author: 2026-06-30 / Codex

## Outcomes & Retrospective

The initial behavior-rigidity layer is additive. It introduces public example layout, golden output comparison, API
surface snapshots, compatibility consistency checks, and named test directories without changing Structure runtime
behavior. Future work can deepen differential tests with live PySpark when that dependency is available in a dedicated
suite.

## Context and Orientation

`examples/` is now a public example source root. The first example is the `orders` package under `examples/orders`.
Its expected generated files are checked in under `examples/structure_generated/orders`. The internal model fixtures under
`res/testing/model/v1` and `res/testing/model/v2` remain orders-focused compiler fixtures and are not the home for all
public examples.

`tests/golden` compares generated source to checked-in example golden files. `tests/differential` compares behavior or
behavioral contracts against independently written references. `tests/metamorphic` checks behavior relationships that
should hold across equivalent or repeated operations. `tests/concepts` maps public concepts to representative
end-to-end tests.

## Plan of Work

Add the first public example package, generate checked-in output with the existing PySpark renderer, and add golden
tests that fail with unified diffs when generated files drift.

Add first-pass differential and metamorphic tests that are Spark-free, so ordinary local development can run them
without live infrastructure. Keep live DataFrame execution in the existing integration matrix.

Add a public API snapshot for the root `structure` package. The snapshot records exported names, kind, module,
qualname, signatures, enum members, and dataclass fields where those are inspectable.

Add an invariant specification describing the checks Structure should enforce at schema, transform-plan, PySpark-plan,
generated-file, and runtime-result boundaries.

Update `Makefile` and `docs/dev/Testing.md` so contributors can find and run the new behavior-rigidity suites.

## Validation and Acceptance

Acceptance is met when `docs/dev/Testing.md` documents the new test layers, `make rigidity` passes without live Spark,
golden tests fail with readable diffs when generated example output changes, API snapshot tests fail on public export
drift, and compatibility tests prove docs, Compose defaults, and the integration runner agree on PySpark 3.5.x and
4.0.x.

## Idempotence and Recovery

All added tests are safe to rerun. Golden tests do not rewrite checked-in files. Regenerate example golden output only
as an intentional source change after reviewing the diff. If the public API snapshot fails, either restore the
accidental API change or regenerate the snapshot as part of an intentional compatibility change.
