# Introduce Integration and Cross Libraries

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can design, document, implement,
and validate the `structure.lib.integration` and `structure.lib.cross` libraries without needing prior conversation
history.

## Purpose / Big Picture

Structure is growing separate compiler, runtime, backend, diagnostics, configuration, and generated-code features. Some
code will be reusable because it talks to an external system such as Spark. Other code will be reusable because it is a
cross-cutting concern such as errors, diagnostics, or logging. Today those boundaries are implicit, so reusable pieces
can drift into feature apps such as `structure.app.backend` or into overly general helper packages.

After this work, contributors have two explicit library homes:

- `structure.lib.integration`: adapters and helpers for external systems, organized by system name.
- `structure.lib.cross`: cross-cutting concepts shared by apps, such as structured errors and logging.

The observable result is not a user-facing CLI feature yet. The result is a package boundary that can be verified by
tests: compiler-safe imports stay free of PySpark, backend capability code uses the new Spark integration package, and
diagnostic/error primitives can be reused outside one app.

## Outcomes & Retrospective

This plan defines the desired architecture and the implementation milestones. It has not yet created the new library
packages or migrated code. The first implementation slice should be intentionally small: package skeletons, boundary
tests, and moving reusable backend diagnostic primitives only after tests protect current behavior.

## Context and Orientation

Structure is a Python package rooted at `src/structure`. The project currently divides code into:

- `src/structure/app`: feature apps and their orchestration or business logic. For example,
  `src/structure/app/backend` owns backend capability behavior.
- `src/structure/lib`: reusable library code. It currently contains small `app` and `common` packages.
- `tests/specs`: tests for specification-backed behavior.
- `docs/specifications`: implementation-ready behavior contracts.
- `docs/dev/design`: design explanations and tradeoffs.
- `docs/dev/planning`: living ExecPlans.

In this plan, an external system means software outside Structure that has its own runtime, API, versioning, and failure
modes. Spark is the main example. PySpark is the Python API for Spark. Spark Connect is a future Spark client mode that
must not distort the current ordinary PySpark contract.

A cross-cutting concern means a concept used by many features, where placing it inside one app would make other apps
depend on the wrong owner. Diagnostics, structured errors, logging, documentation links, and future metrics are
cross-cutting concerns.

The intended boundaries are:

- `structure.lib.helper` or the current `structure.lib.common`: general-purpose helpers with no external-system focus,
  such as text, collection, path, or import utilities when they are genuinely reusable.
- `structure.lib.integration.<system>`: helpers and adapters for an external system such as Spark. This package may
  know about external API names, version ranges, optional imports, generated imports, and compatibility probes. It must
  not know Structure business rules such as transform semantics, schema inheritance, join semantics, or user stories.
- `structure.lib.cross.<concern>`: shared concerns used across apps. This package may define generic diagnostic,
  logging, error, tracing, or metrics primitives. It must not depend on feature apps.
- `structure.app.<feature>`: Structure business logic. Apps may depend on `structure.lib.cross`,
  `structure.lib.integration`, and helper libraries, but libraries must not depend on apps.

## Plan of Work

Start by documenting the boundary. Add `docs/specifications/LibraryBoundaries.md` to define the package ownership rules,
dependency direction, and acceptance tests. Add `docs/dev/design/LibraryBoundaries.md` to explain why `integration` and
`cross` are separate from helpers and apps. Update `docs/dev/Architecture.md` with a short section that links to both
documents.

Add package skeletons:

    src/structure/lib/integration/__init__.py
    src/structure/lib/integration/spark/__init__.py
    src/structure/lib/integration/spark/target.py
    src/structure/lib/integration/spark/imports.py
    src/structure/lib/cross/__init__.py
    src/structure/lib/cross/errors/__init__.py
    src/structure/lib/cross/errors/Diagnostic.py
    src/structure/lib/cross/errors/DiagnosticRegistry.py
    src/structure/lib/cross/errors/StructureError.py
    src/structure/lib/cross/logging/__init__.py
    src/structure/lib/cross/logging/loggers.py

The names above are prescriptive for the first slice. Later implementation may split files if they grow, but the public
package boundary should remain stable.

Move reusable backend diagnostic code out of `src/structure/app/backend/capabilities/logic/model` only after adding tests that
capture current behavior. `BackendDiagnostic`, `BackendCapabilityError`, and diagnostic-code constants can either wrap
generic cross primitives or become thin backend-specific subclasses. The key rule is that backend capability behavior
must remain unchanged for the tests in `tests/specs/backend_capabilities/test_backend_capabilities.py`.

Move Spark-specific static metadata from backend rules into `structure.lib.integration.spark` when it is not actually a
backend business rule. Good candidates are the default PySpark target range, generated import names, and helpers that
describe optional PySpark imports. Do not move capability decisions themselves if they express Structure semantics such
as which DSL feature is supported in v1; those remain backend app rules.

Add a small logging cross-cut only if it stays tiny. It should use Python's standard `logging` module and expose a
stable way to name Structure loggers, such as `structure_logger("app.backend")`. Do not add a logging framework or
configuration system in this plan. Logging configuration belongs to callers or a future CLI/runtime app.

Add dependency-boundary tests. Use Python source inspection rather than runtime monkey-patching for the first slice.
The tests should prove:

- modules under `src/structure/lib/cross` do not import `structure.app`;
- modules under `src/structure/lib/integration` do not import `structure.app`;
- importing `structure.lib.integration.spark` does not import `pyspark`;
- importing `structure.app.backend.capabilities.api.capabilities` still does not import `pyspark`;
- backend capability behavior is unchanged.

## Milestones

Milestone 1 establishes the documented contract and empty package boundary. At the end of this milestone,
`docs/specifications/LibraryBoundaries.md`, `docs/dev/design/LibraryBoundaries.md`, and package `__init__.py` files
exist. Run `poetry run pytest tests/specs/backend_capabilities` and expect current backend tests to keep passing. This
milestone proves the new boundary is visible without changing behavior.

Milestone 2 introduces cross-cut errors. At the end of this milestone, `structure.lib.cross.errors` owns the generic
diagnostic value, registry value, and base `StructureError`. Backend capability errors either subclass or compose those
primitives. Run `poetry run pytest tests/specs/backend_capabilities` and the new cross-error tests. Existing assertions
for `BACKEND-E2401` and `BACKEND-E2402` must still pass.

Milestone 3 introduces Spark integration metadata. At the end of this milestone,
`structure.lib.integration.spark.target` owns static Spark target names and PySpark target-range constants, while
backend capability rules still own Structure feature decisions. Run the import-safety tests and prove that PySpark is
not present in `sys.modules` after importing compiler/backend capability modules.

Milestone 4 introduces tiny cross-cut logging. At the end of this milestone, apps can ask for named Structure loggers
through `structure.lib.cross.logging` without configuring handlers or changing caller logging behavior. Run the logging
tests and full pytest suite. No user-visible log output should appear by default.

Milestone 5 wires documentation and indexes. At the end of this milestone, architecture, diagnostics, backend
capability, and project-management docs link to the new boundary documents where relevant. Run the full test suite and
read the updated docs as plain Markdown to confirm line lengths and naming are consistent.

## Validation and Acceptance

Acceptance is met when a human can verify all of the following from the repository root:

- `poetry run pytest tests/specs/backend_capabilities` passes with the same observable backend capability behavior as
  before the library split.
- `poetry run pytest tests/specs/library_boundaries` passes and proves library packages do not depend on apps.
- `poetry run pytest tests/specs/integration_spark` passes and proves importing static Spark integration helpers does
  not import PySpark.
- `poetry run pytest tests/specs/cross_errors` passes and proves diagnostics/errors can be created without importing a
  feature app.
- `poetry run pytest tests/specs/cross_logging` passes and proves logger naming is stable and silent by default.
- `poetry run pytest` passes, or any unrelated pre-existing failure is recorded with enough detail to reproduce.

The change is complete only when docs and tests explain the boundary well enough that a future contributor knows
whether a new reusable helper belongs in helper/common, integration, cross, or an app.

## Idempotence and Recovery

The work is additive at first. Re-running tests is safe. If a migration step breaks backend capability behavior, revert
only the migration edits for that step and keep the package skeleton and docs. Do not move all diagnostics at once.
Migrate one reusable primitive, run tests, then migrate the next.

If importing `structure.lib.integration.spark` accidentally imports PySpark, move that import behind a function such as
`import_pyspark()` and test that the module-level import remains clean. If a cross library needs app-specific context,
the design is wrong for `cross`; keep the generic primitive in `cross` and place the app-specific wrapper in the app.

## Artifacts and Notes

The package dependency direction should be:

    structure.app.* -> structure.lib.cross
    structure.app.* -> structure.lib.integration
    structure.app.* -> structure.lib.common or structure.lib.helper
    structure.lib.cross -> standard library and helper/common only
    structure.lib.integration -> standard library, helper/common, optional external systems behind lazy imports

Forbidden dependency direction:

    structure.lib.* -> structure.app.*

The first Spark integration package should prefer static metadata over runtime probing. Runtime probing is allowed only
inside clearly named functions that callers invoke intentionally, because normal compiler commands must remain Spark
free.

## Interfaces and Dependencies

Use only the Python standard library for `structure.lib.cross.errors`, `structure.lib.cross.logging`, and static parts
of `structure.lib.integration.spark`. Do not add a PySpark dependency. Do not introduce a logging framework.

The first implementation should expose these interfaces:

    structure.lib.cross.errors.Diagnostic
    structure.lib.cross.errors.DiagnosticRegistry
    structure.lib.cross.errors.StructureError
    structure.lib.cross.logging.structure_logger(name: str) -> logging.Logger
    structure.lib.integration.spark.DEFAULT_TARGET_PYSPARK
    structure.lib.integration.spark.PYSPARK_BACKEND_ID
    structure.lib.integration.spark.generated_imports()

Backend-specific constants such as `BACKEND-E2401` and `BACKEND-E2402` may remain in
`structure.app.backend.capabilities.logic.model` or move into a backend-specific registry module. They should use cross-error
primitives but keep the backend app as the owner of backend capability meaning.
