# Add Compiler Provenance and Static Dataflow Lineage

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement compiler

## Purpose / Big Picture

Structure is a compiler for Python-like schema transform code that emits PySpark code. The most useful v1 lineage is
therefore compile-time lineage: it should explain where generated PySpark came from and how data flows through
transforms, tables, and columns. After this work, a developer can run `structure check`, `structure compile`, or
`structure explain` and see source-to-generated provenance plus static dependencies inferred from the compiler IR.

The observable outcome is better troubleshooting. When the compiler reports an unsupported expression, schema mismatch,
unsafe join, or generated-code diff, the diagnostic can point to the source node, the IR node, the generated PySpark
node, and the upstream table or column dependencies involved.

Runtime LDJSON lineage is intentionally out of scope for v1. It is useful as optional transform-run telemetry, but it
belongs after v4 because it records runtime facts rather than compiler understanding.

## Context and Orientation

Structure lets developers write typed Python transform classes and compiles them to explicit PySpark DataFrame code.
The compiler pipeline is documented in `docs/dev/Architecture.md` and `docs/dev/design/ExecutionDataFlows.md`.

The relevant compiler phases are:

- discovery, which finds schema and transform source objects;
- symbolic execution, which runs transform methods with symbolic schema objects rather than real DataFrames;
- IR generation, which records transform semantics in backend-neutral data structures;
- compileability checks, which reject unsupported source patterns before code generation;
- PySpark code generation, which lowers IR to generated PySpark files;
- diagnostics and `structure explain`, which tell developers what the compiler understood.

Terms used in this plan:

- Source node: a source-code construct the compiler can identify, such as a transform class, input declaration,
  subtransform method, schema constructor argument, `where(...)` call, join condition, or hook declaration.
- IR node: a compiler model object created from source, such as `TransformPlan`, `StepPlan`, `Filter`, `Project`,
  `Join`, `HookCall`, `ValidateSchema`, or an expression node.
- Generated PySpark node: an emitted code construct, such as a generated class, `run(...)` parameter, `select(...)`
  entry, `where(...)` expression, `join(...)` call, hook call, validation call, or output file span.
- Compiler provenance: the mapping from source node to IR node to generated PySpark node.
- Static dataflow lineage: dependency information inferred without running Spark, such as "output column
  `customer_name` depends on input table `customers` column `name` through step `add_customer`."
- Runtime lineage emitter: optional runtime support that writes transform-run facts as LDJSON. This is out of scope for
  this plan and belongs in `docs/dev/project-management/NiceToHave.md`.

## Plan of Work

This plan has two layers. The documentation layer is complete and should now be maintained as project docs evolve. The
compiler implementation layer remains open and starts below.

Start implementation with a small provenance identity model. Every source node, IR node, and generated PySpark node
that participates in diagnostics should have a stable ID within one compile run. IDs must be deterministic for
unchanged source so tests, snapshots, and generated-code review remain stable. Source IDs should include source module,
class, member, and source span when available. IR IDs should include transform, step, operation kind, and ordinal.
Generated IDs should include generated module, construct kind, and emitted span or symbolic path.

Add a `ProvenanceMap` compiler artifact. It should answer three questions:

- which source node produced this IR node;
- which IR node produced this generated PySpark node;
- given any one node, what nearby source, IR, and generated nodes should a diagnostic display.

Attach provenance incrementally. Discovery should create source nodes for transform classes, inputs, schemas, and
members. Symbolic execution should attach source nodes to expression calls, filters, joins, projections, and hooks. IR
construction should copy source provenance onto `TransformPlan`, `StepPlan`, operation, and expression nodes. PySpark
generation should record generated nodes for emitted code blocks and point them back to the IR nodes being lowered.

Then add a static dataflow model inferred from IR. It should not parse generated PySpark text. It should walk
`TransformPlan` and produce dependencies at these levels:

- transform depends on named inputs;
- step depends on the prior step and any joined inputs;
- output schema depends on the final step;
- output field depends on source fields, literals, expression helpers, filters, joins, and hook boundaries;
- join output depends on join key fields, joined input fields used later, join type, and hint;
- hook boundary depends on the incoming DataFrame and is marked opaque unless the compiler later understands the hook.

Represent unresolved or opaque dependencies explicitly. Hooks are the most important case: a hook may add, drop, or
modify columns using arbitrary PySpark. Static dataflow should show the hook as a boundary rather than pretending to
know its internals. Diagnostics should say which columns are known before the hook, which declared schema is expected
after it, and why the hook boundary limits static certainty.

Expose the models through diagnostics and `structure explain`. Diagnostics should include source location first,
followed by the IR operation and generated-code location when available. `structure explain` should gain a lineage view
that prints compile-time provenance and static dependencies for a transform. Keep the default explain output concise,
and add a flag or subview for column-level details if the normal output would become noisy.

Finally, add tests. Keep them focused on behavior rather than internal implementation shape. Use a transform fixture
with projection, filtering, a helper expression, a left `join_one(...)`, and an after hook. Assert that source-to-IR
and IR-to-generated mappings exist for each step, and assert that output fields report the expected static
dependencies. Add regression tests for deterministic IDs by compiling the same fixture twice and comparing the emitted
provenance/dataflow artifact or explain output.

    @transform
    class EnrichOrders(Transform):

        orders = input(OrderRaw)
        customers = input(Customer)

        @expr_fn
        def clean_id(value):
            return lower(trim(value))

        def normalize(self, order: OrderRaw) -> OrderNormalized:
            where(order.id.is_not_null())
            return OrderNormalized(id=order.id, customer_id=self.clean_id(order.customer_id))

        def add_customer(self, order: OrderNormalized) -> OrderWithCustomer:
            customer = self.customers.join_one(
                on=self.customers.id == order.customer_id,
                how=Join.LEFT,
            )
            return OrderWithCustomer(id=order.id, customer_name=customer.name)

        @after(add_customer)
        def add_quality(self, *, df, spark, ctx):
            return df

    OrderWithCustomer.id <- OrderNormalized.id <- orders.id
    OrderWithCustomer.customer_name <- customers.name through join add_customer

## Idempotence and Recovery

This work should be additive. Re-running `structure check`, `structure compile`, or tests must not mutate source files
except for explicit generated-code output during compile. Provenance IDs must be deterministic so regenerated artifacts
do not churn.

If generated-code spans are hard to record during early code generation, first record stable symbolic generated paths
such as `EnrichOrdersGenerated.run.step[1].join[0]`. Add physical file spans later when the emitter can report them
without making generation brittle.

If hook internals are unavailable, keep the boundary opaque. Do not inspect arbitrary hook source with ad hoc parsing
unless a separate design contract makes hook analysis compiler-visible.

If the dataflow graph becomes too noisy, keep full detail in an internal artifact and make `structure explain` default
to transform and step summaries with an opt-in column view.

## Interfaces and Dependencies

The implementation must stay Spark-free in compiler phases. Do not import PySpark from provenance or static dataflow
model code.

Required compiler-facing artifacts:

- a provenance model that links source nodes, IR nodes, and generated PySpark nodes;
- a static dataflow model inferred from IR;
- diagnostics that can ask provenance and dataflow for context;
- `structure explain` output that can show provenance and static dependencies;
- tests that prove deterministic IDs and dependency inference.

Required static dataflow entities:

- transform dependencies;
- step dependencies;
- named input dependencies;
- output schema dependencies;
- output field dependencies;
- join key and joined-field dependencies;
- filter dependencies;
- expression helper dependencies;
- opaque hook boundaries.
