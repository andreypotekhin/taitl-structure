# Implement the Data Quality Constraints Boundary

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement the future data
quality constraint model without prior conversation history.

## Purpose / Big Picture

Structure users need to know what "schema-enforced" means. After this work, v1 validation remains fast and predictable
because schema-only validation checks DataFrame shape without scanning rows. Future data-quality checks become explicit
and cost-aware, so users can opt into stronger guarantees only when the added Spark work is intentional.

The observable result is twofold: generated schema constants can be imported by caller code for reads and pre-write
validation, and future constraint validation can prove through tests that action-triggering checks never happen by
surprise.

## Surprises & Discoveries

- Observation: The schema model already reserved `SchemaDef.constraints`, but no specification owned those semantics.
  Evidence: `docs/specifications/SchemaModel.md` lists `constraints` and says row-level constraint validation belongs
  outside the base schema model.

- Observation: Configuration already exposes `intermediate_validation_mode = "schema_and_constraints"`, but the
  future constraint boundary was not documented.
  Evidence: `docs/Configuration.md` names the mode while saying row-level constraints may be added later.

## Decision Log

- Decision: Keep v1 intermediate validation schema-only by default.
  Rationale: Schema-only validation is cheap and predictable because it compares metadata rather than scanning data.
  Date/Author: 2026-06-20 / Codex

- Decision: Require action-triggering data checks to be explicit.
  Rationale: Counts, uniqueness checks, referential checks, and freshness checks can add Spark jobs. Users need to
  choose that cost deliberately.
  Date/Author: 2026-06-20 / Codex

- Decision: Keep generated `*_SCHEMA` constants shape-only.
  Rationale: Callers should be able to import generated schemas as normal PySpark `StructType` values without also
  importing executable data-quality policy.
  Date/Author: 2026-06-20 / Codex

- Decision: Expose online materialized schemas through transform invocations.
  Rationale: Online execution is the default, so callers need access to the final output `StructType` even when they do
  not run `structure compile` or import generated files.
  Date/Author: 2026-06-20 / Codex

- Decision: Use symmetric validation phase modes plus phase-bound constraints.
  Rationale: Phase modes give operators a project-level cost guard, while phase-bound constraints express local source
  intent.
  Date/Author: 2026-06-20 / Codex

## Outcomes & Retrospective

The v1 schema-validation boundary is implemented: generated Spark schemas are reusable, online execution exposes
materialized schemas, and input/intermediate/output validation modes are configured separately. Row-level
data-quality constraints remain v2+ work because they can trigger Spark actions and need explicit cost policy.

## Context and Orientation

Structure schemas declare DataFrame shape. Generated schema constants such as `ORDER_ENRICHED_SCHEMA` are PySpark
`StructType` values emitted under `generated/structure_generated/.../pyspark/schemas/`. Online execution must expose
equivalent materialized schemas through the transform invocation after `run(session)`. Runtime helpers such as
`assert_schema(...)` and `project_schema(...)` validate and project DataFrames according to those shapes.

Data quality constraints are different. They inspect values or datasets: examples include "status is one of these
values", "total is non-negative", "customer_id is unique", "every order customer exists in customers", "data is fresh
within one day", and "the result has at least one row". Some are row-local filters. Others require aggregation or joins.

## Plan of Work

First, preserve the current v1 behavior. Confirm that `schema_only` validation is implemented as metadata comparison
against `DataFrame.schema` and does not use `count`, `collect`, `take`, aggregations, joins, or row filters.

Next, add an internal constraint model. Use immutable records or the local equivalent. The model must capture the
constraint name, owner, target, validation phases, kind, expression, severity, cost class, streaming compatibility, and
source location. Do not export these records from the public `structure` root package before the public DSL is
finalized.

Then add cost classification. Every constraint must be classified as `compile_time`, `schema_only`, `row_local`,
`aggregation`, or `join`. Batch-only and streaming-compatible classification should derive from this cost unless a
narrower rule proves a check safe for streaming.

Generate constraint metadata separately from `*_SCHEMA` constants. The generated schema module may import only PySpark
types and define `StructType` constants. Constraint metadata can live in a sibling generated module or in traceability-like
metadata once the format is specified.

Expose online schemas from the transform result. The caller should write:

    transform = EnrichOrders(orders=orders_df, customers=customers_df)
    result = transform.run(session)
    output_schema = result.schema.enriched

The online schema must come from the same checked schema model as generated schema constants and must not require
generated files.

Finally, lower enabled constraints into shared validation recipes consumed by online execution and generated code.
Generated code must make action-triggering checks visible. Online execution must run the same recipes.

## Validation and Acceptance

The documentation phase is accepted when:

- `docs/specifications/DataQualityConstraints.md` exists;
- `docs/dev/design/DataQualityConstraints.md` exists;
- `docs/dev/design/decisions/D06202602.Data-quality-constraints-boundary.md` exists;
- C26 in `docs/dev/design/Challenges.md` points to those documents;
- configuration, usage, generated PySpark, schema model, roadmap, implementation, and sprint docs reference the
  boundary where relevant.

The implementation phase is accepted when tests prove:

- `schema_only` validation compares schema metadata and does not trigger Spark row scans;
- `schema_and_constraints` can enable implemented constraints only when explicit;
- action-triggering constraints are visible in recipes, generated code, diagnostics, and traceability metadata;
- generated `*_SCHEMA` constants remain deterministic `StructType` values;
- online execution exposes equivalent materialized Spark schemas after `run(session)`;
- generated constraint metadata does not change generated schema constants;
- callers can import generated schemas for reads and pre-write validation/projection;
- callers can use online-exposed schemas for pre-write validation/projection;
- online and generated execution run the same enabled constraint recipes;
- streaming compatibility rejects batch-only constraints with clear diagnostics.

## Idempotence and Recovery

The implementation should be additive. Keep generated schema constants stable while adding constraint metadata beside
them. If a test reveals that adding constraints changes existing schema snapshots, stop and move the new data to a
separate artifact.

If PySpark is unavailable locally, run no-Spark compiler and generation tests, then record that Spark runtime tests were
not run. Do not weaken the no-Spark compile boundary.
