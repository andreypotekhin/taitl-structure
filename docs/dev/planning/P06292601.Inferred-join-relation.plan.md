# Infer `join_one(...)` Relations from Join Conditions

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [Plans.md](../auto/Plans.md). It is self-contained so a contributor can implement inferred
lookup-join relation selection without relying on prior conversation history.

## Purpose / Big Picture

After this change, Structure users can write lookup joins without repeating the right-side relation argument when the
`on` condition already names that relation clearly:

    customer = join_one(
        on=order.customer_id == self.customers.id,
        how=Join.LEFT,
    )

The compiler will infer that `self.customers` is the joined relation because it is the only unjoined relation scope in
the join condition. The existing explicit form remains valid for ambiguous or intentionally extra-clear cases:

    customer = join_one(
        self.customers,
        on=order.customer_id == self.customers.id,
        how=Join.LEFT,
    )

The observable result is that tests can compile transforms using both forms and receive identical join IR, generated
PySpark, online execution behavior, diagnostics, static traceability, and uniqueness warnings. Ambiguous predicates
must fail early with a diagnostic that tells the developer to pass `relation=...` or the positional relation argument.

## Surprises & Discoveries

- Observation: The current compiler validates joins by using the explicit relation argument as the declared joined
  input, then checking that each equality pair references that input on exactly one side.
  Evidence: `CompileTransform._validate_join_pair(...)` computes `left_has_input` and `right_has_input` from
  `join.input_name`.

- Observation: Current field expressions carry a string scope name but no explicit scope kind.
  Evidence: `RowScope.__getattr__(...)` stores `data={"scope": self._structure_scope_name, ...}` in field
  expressions. The first implementation can infer with names plus active context bindings, but durable design should
  prefer richer scope metadata over string-only classification.

- Observation: The current public function already defaults `how=Join.LEFT`, while the specification text says `how`
  is required in v1.
  Evidence: `InputScope.join_one(...)` defines `how: Join = Join.LEFT`; `JoinSemantics.md` says source should show
  whether unmatched rows are kept or removed. This plan preserves the current default and does not broaden the task
  into a join-type API migration.

## Decision Log

- Decision: Add inferred relation selection as an additional public form, not a replacement for explicit relation
  selection.
  Rationale: Inference is excellent when exactly one unjoined relation appears in the condition. Explicit relation
  selection is still the right escape hatch for ambiguous predicates, migration clarity, and future join families.
  Date/Author: 2026-06-29 / Codex.

- Decision: Infer only from accepted v1 equality key predicates.
  Rationale: `join_one(...)` is intentionally a lookup equi-join. Inference should not add a second, looser grammar
  for arbitrary predicates.
  Date/Author: 2026-06-29 / Codex.

- Decision: The inferred candidate must be exactly one currently unjoined relation scope across the whole flattened
  `on` expression.
  Rationale: This keeps the rule memorable and prevents the compiler from guessing when a condition mentions two
  candidate right sides or none.
  Date/Author: 2026-06-29 / Codex.

- Decision: Later joins may compare an unjoined relation to the driving row or to an earlier joined scope.
  Rationale: Existing join semantics allow serial enrichment, and inferred syntax should not make serial joins weaker.
  Date/Author: 2026-06-29 / Codex.

## Context and Orientation

Structure is a Python-to-PySpark compiler. Users write typed transform methods with symbolic schema rows. The compiler
executes those methods with symbolic objects instead of real rows and records filters, joins, and projections into
backend-neutral intermediate representation, usually called IR. IR means a Python data model that describes what the
pipeline should do before it is lowered to online PySpark execution or generated PySpark source.

The current public v1 lookup join is `join_one(relation, *, on, how=Join.LEFT, hint=None)`. It is implemented in
`src/structure/app/dsl/model/expr/InputScope.py`. The `relation` argument can be a transform input scope such as
`self.customers` or a schema relation parameter such as `customer` in a multi-parameter subtransform. The function
records a `JoinPlan` from `src/structure/app/compiler/ir/model/JoinPlan.py` and returns a joined `RowScope`.

The compiler validates joins in `src/structure/app/compiler/frontend/commands/CompileTransform.py`. The relevant
methods are `_validate_joins(...)`, `_join_conditions(...)`, `_validate_join_pair(...)`, `_unique_join(...)`, and
`_scopes(...)`. They already flatten `&`, accept `==` and `null_safe_eq(...)`, reject unsupported predicates, preserve
source order, check type compatibility, and warn when `join_one(...)` uniqueness is not proven.

A scope is the symbolic owner of a field expression. Examples are the driving row `order`, an unjoined relation
parameter `product`, a transform input scope `self.customers`, and the joined scope returned by `join_one(...)`.
Today, field expressions record only the scope name string in expression data. This plan uses the active compile
context and step input bindings to classify those names as current, already joined, or unjoined relation scopes.

Update `docs/specifications/SymbolicExecution.md` and `docs/specifications/DSL.md` to describe relation inference
during join capture. The symbolic engine records the same `JoinPlan` whether the relation was explicit or inferred.
No online or generated backend should be able to observe which source spelling the user chose, except through
diagnostics or source metadata.

Add tests before implementation. In `tests/specifications/multiple-schema-parameters/test_multiple_schema_parameters.py`
or a new focused `tests/specifications/join-semantics/test_inferred_join_relation.py`, add positive tests showing that
`join_one(on=product.id == order.product_id)` compiles and records `step.joins[0].source == "products"`. Add the same
test with reversed operands, composite `&`, `null_safe_eq(...)`, expression helpers, a class input scope
`self.products`, and a serial join where the second join compares an unjoined relation to the first joined scope.

Add negative diagnostics in `tests/specifications/compiler-diagnostics/test_v1_negative_compiler_diagnostics.py`.
Cover these cases:

    join_one(on=order.customer_id == order.account_id)

This has no unjoined relation candidate and must tell the user to pass an explicit relation or compare against a
declared input/relation parameter.

    join_one(on=self.customers.id == self.accounts.customer_id)

This has two unjoined relation candidates and must ask for `join_one(relation=..., on=...)` or
`join_one(self.customers, on=...)`.

    join_one(
        on=(order.customer_id == self.customers.id)
        & (order.product_id == self.products.id),
    )

This mixes two candidate right sides in one join and must fail. The user should split the logic into two join calls.

    join_one(on=self.customers.parent_id == self.customers.id)

This compares one unjoined relation to itself and must fail with the existing "one side must be non-joined" style of
message.

Refactor `join_one(...)` capture in `InputScope.py`. Change the signature to accept an optional first argument:

    def join_one(
        relation: Relation | None = None,
        *,
        on: object,
        how: Join = Join.LEFT,
        hint: JoinHint | None = None,
    ) -> Relation:

Keep positional `join_one(self.customers, on=...)` valid. Also support keyword `relation=self.customers` if practical,
because diagnostics can recommend that spelling without depending on positional ordering. Reject any non-`InputScope`
relation with the existing clear error. If `relation` is omitted, call a small inference component rather than placing
the full algorithm inside the public function.

Implement relation inference as focused logic, preferably in a small helper class or functions near symbolic
execution or compiler frontend code. It needs the active `CompileContext`, the `on` expression, and the set of scopes
available at this point. If the current `CompileContext` does not yet know the current row scope, bound relation
scopes, and already joined scopes, extend it minimally during `CompileTransform._steps(...)` before invoking the user
method. Avoid relying on Python local variable names.

The inference algorithm is:

1. Flatten `on` through `and` expressions left to right, exactly as `_join_conditions(...)` does.
2. Require every leaf to be `eq` or `null_safe_eq`. If not, let the existing unsupported-condition diagnostic win.
3. For each equality pair, recursively collect field scope names from the left and right expression.
4. Classify each scope as current row, already joined, or unjoined relation.
5. A pair contributes candidate right scopes from its unjoined relation scopes.
6. Across all pairs, require exactly one candidate right scope.
7. Require every pair to reference that candidate on exactly one side.
8. Require the opposite side of every pair to contain at least one current-row or already joined scope.
9. Return the `InputScope` metadata for that candidate and build the same `JoinPlan` as the explicit form.

If richer metadata is cheap, introduce it now. `RowScope` can carry optional scope kind and source fields, and field
expressions can include data such as `scope_kind`, `source`, and `occurrence`. If that becomes invasive, defer it and
use a context-level scope registry keyed by the existing scope name string. Record the choice in the `Decision Log`.

Update relation-read validation. `_validate_relation_reads(...)` currently treats relation parameters as joined if
their `join.input_name` appears in captured joins. This should still work for inferred joins if the captured `JoinPlan`
is identical. Add a regression where a relation parameter is used in projection before any inferred join and still
fails.

Preserve all downstream behavior. The explicit and inferred forms should produce identical `JoinPlan` values. Because
generated PySpark, online execution, static traceability, and uniqueness warnings already consume `JoinPlan`, they
should need little or no change. Add tests that compare important fields rather than adding a second lowering path.

Update examples and docs. In `Readme.md`, `docs/Basics.md`, `docs/QuickRef.md`, `docs/Overview.md`, and
`docs/dev/Terminology.md`, prefer the inferred form in simple examples and show the explicit form as the ambiguity
escape hatch. Do not update every historical plan in `docs/dev/planning/done`; those are archival records.

## Validation and Acceptance

The feature is complete when all of these behaviors are observable:

- Existing `join_one(relation, on=...)` source compiles unchanged.
- New `join_one(on=...)` source compiles when the condition contains exactly one unjoined relation.
- Inferred and explicit forms produce equivalent `JoinPlan` values.
- Operand order does not matter.
- Composite keys preserve source order after flattening `&`.
- `null_safe_eq(...)` works with inference.
- Expression helpers around key fields do not prevent inference.
- A later inferred join may reference an earlier joined scope.
- A condition with no unjoined relation fails with a direct diagnostic.
- A condition with two unjoined relations fails and asks for explicit relation selection.
- A composite condition that mixes right-side candidates fails and asks the user to split joins or select explicitly.
- A condition comparing a relation only to itself fails.
- Relation parameters still cannot be projected or filtered before joining.
- Unproven `join_one(...)` uniqueness warnings still appear.
- Generated PySpark for inferred joins matches explicit-join behavior.
- Online execution for inferred joins matches generated execution.
- Static traceability records the same join dependencies as explicit joins.
- Public docs show inferred syntax as the simple form and explicit syntax as the ambiguity escape hatch.
- `make build` passes.

## Idempotence and Recovery

This change should be additive. Do not remove explicit `join_one(relation, ...)`, and do not change the v1 accepted join
condition grammar. If inference touches downstream generated-code snapshots unexpectedly, first compare the explicit
and inferred `JoinPlan` values. If they differ, fix inference rather than adding target-layer special cases.

If scope names collide during implementation, stop and add richer scope identity instead of guessing by string name.
The project already values deterministic aliases and traceability; a slightly stronger scope model is better than a
fragile inference rule.

If the full build fails for an unrelated existing issue, record the command and failure in `Surprises & Discoveries`
and run the focused tests that prove this feature. Do not weaken tests or diagnostics to make the build pass.

Preserve unrelated worktree changes. Do not rewrite generated fixtures wholesale unless the changed fixture is owned by
an example intentionally updated for inferred joins.

## Artifacts and Notes

Preferred simple syntax:

    customer = join_one(
        on=order.customer_id == self.customers.id,
        how=Join.LEFT,
        hint=JoinHint.BROADCAST,
    )

Still-supported explicit syntax:

    customer = join_one(
        self.customers,
        on=order.customer_id == self.customers.id,
        how=Join.LEFT,
    )

Preferred relation-parameter syntax:

    def add_product(self, order: OrderRaw, product: Product) -> OrderWithProduct:
        product = join_one(
            on=product.id == order.product_id,
            how=Join.LEFT,
        )
        return OrderWithProduct(id=order.id, product_name=product.name)

Ambiguous syntax that must fail:

    joined = join_one(
        on=self.customers.account_id == self.accounts.id,
        how=Join.LEFT,
    )

Expected diagnostic shape:

    Cannot infer joined relation for join_one(...).
    The join condition references multiple unjoined relations: customers, accounts.
    Use join_one(self.customers, on=...) or join_one(relation=self.customers, on=...) to choose one.

## Interfaces and Dependencies

The public interface at completion is:

- `join_one(on=expression, how=Join.LEFT, hint=None) -> RowScope`
- `join_one(relation, *, on=expression, how=Join.LEFT, hint=None) -> RowScope`
- optionally `join_one(relation=relation, on=expression, how=Join.LEFT, hint=None) -> RowScope`

The internal interfaces at completion are:

- a relation-inference helper that receives an `Expression` and active symbolic scope metadata;
- active scope metadata that can distinguish current row, unjoined relation, and joined scopes;
- unchanged `JoinPlan` fields unless richer source metadata is explicitly needed;
- unchanged PySpark, online, and traceability consumers that rely on `JoinPlan`, not source spelling.

No new third-party dependency is required. Continue using the existing symbolic expression tree, compiler diagnostics,
and PySpark lowering layers.
