## Purpose / Big Picture

At v1 completion, a developer can define schema-driven transforms, check them without Spark, optionally generate
reviewable PySpark code, run them online through `StructureSession`, and trust that online and generated execution share
the same Spark-visible semantics. Configuration mistakes, unsupported DSL, stale generated output, validation failures,
and generated-mode import problems should produce structured diagnostics with documentation links.

The visible end state is:

    structure check
    structure compile
    structure compile --fail-on-diff
    structure explain testing.model.v1.orders.transforms.order.EnrichOrders
    pytest

The compiler commands must run without PySpark. Runtime parity tests may require PySpark and are separate from the
Spark-free compiler tests.

## Surprises & Discoveries

- Observation: The v1 fixture originally imported PySpark at module import time for hook bodies.
  Evidence: PySpark is not installed in this workspace, and the symbolic compile slice moved hook imports inside hook
  methods so compiler work remains Spark-free.

- Observation: The first shared PySpark recipe layer can validate the v1 lowering contract without importing PySpark.
  Evidence: `tests/specs/v1_pyspark_recipes/test_v1_pyspark_recipes.py` asserts step order, aliases, joins, hooks,
  projection expressions, validation placement, and unchanged `sys.modules` for `pyspark`; the full suite passed with
  `19 passed in 0.36s`.

- Observation: Generated schema fixtures use base-schema composition for inherited schemas instead of expanding every
  inherited field repeatedly.
  Evidence: `ORDER_WITH_CUSTOMER_SCHEMA = T.StructType(ORDER_NORMALIZED_SCHEMA.fields + [...])` and
  `ORDER_PUBLISHED_SCHEMA = T.StructType(ORDER_PUBLICATION_SCHEMA.fields + PUBLICATION_FLAGS_SCHEMA.fields)`.

- Observation: Test runs touch tracked `.pyc` files in this repository.
  Evidence: `git status --short` reported modified files under `src/structure/__pycache__` and
  `src/structure/app/backend/capabilities/logic/model/__pycache__`. A git checkout cleanup was rejected by the execution
  environment quota, so those generated bytecode changes remain workspace hygiene work rather than source work.

- Observation: Expression rendering can stay recipe-first and avoid reinterpreting source DSL objects.
  Evidence: `tests/specs/v1_pyspark_expression_rendering/test_v1_pyspark_expression_rendering.py` renders filters,
  helper calls, arithmetic, comparisons, composite joins, and null-safe equality from `PySparkExpressionRecipe` values
  produced by `lower_pyspark_plan(...)`.

- Observation: The initial step renderer exposed duplicate final validation in the recipe model.
  Evidence: Rendering the `publish` step produced two strict `ORDER_PUBLISHED_SCHEMA` assertions. The fix moved final
  output validation exclusively to `PySparkExecutionPlan.final_validation`, leaving step validations for hooks and
  intermediate boundaries.

## Decision Log

- Decision: Treat shared PySpark execution recipes as the next milestone after symbolic compilation.
  Rationale: `docs/specifications/ExecutionSemanticContract.md` requires online execution and generated code to consume
  one target-specific semantic plan. Implementing recipes first prevents two separate lowerers from drifting.
  Date/Author: 2026-06-21 / Codex.

- Decision: Keep compiler-facing PySpark recipes Spark-free and textual or structural.
  Rationale: `structure check`, `structure compile`, `compile --fail-on-diff`, and `explain` must not import PySpark,
  start Java, create a `SparkSession`, or contact a Spark cluster. Recipes can describe PySpark operations without
  creating live PySpark objects.
  Date/Author: 2026-06-21 / Codex.

- Decision: Record validation placement in the recipe layer before implementing runtime validation helpers.
  Rationale: Validation order is part of the online/generated parity contract. Capturing it now gives generated code
  and online execution the same boundaries later, even before schema checks themselves exist.
  Date/Author: 2026-06-21 / Codex.

- Decision: Track `_structure_local_fields` and `_structure_schema_bases` on schema classes.
  Rationale: Effective fields remain the validation/projection contract, while local fields and schema bases let schema
  rendering emit concise deterministic inherited schema constants matching the generated fixtures.
  Date/Author: 2026-06-21 / Codex.

- Decision: Treat PySpark as a backend target subapp at `structure.app.backend.pyspark`, not as a peer app under
  `structure.app`.
  Rationale: The PySpark recipe and rendering layers are target-specific backend implementation details. Keeping them
  under `backend` makes room for sibling backend concerns such as capabilities and future target implementations
  without implying that PySpark is an end-user application surface.
  Date/Author: 2026-06-21 / Codex.

- Decision: Keep model classes one-per-source-file and let grouped modules act only as import aggregators.
  Rationale: The model layer is easier to navigate and extend when each concept has its own source file. Public type
  constructors such as `String()` are now classes rather than factory functions, preserving author syntax while making
  the model surface explicit.
  Date/Author: 2026-06-21 / Codex.

## Outcomes & Retrospective

Milestone 1 is complete. `lower_pyspark_plan(...)` now turns a checked `TransformPlan` into a Spark-free
`PySparkExecutionPlan` with input validation recipes, source-order step recipes, stable aliases, filters, joins,
projections, hooks, hook-input requirements, validation boundaries, backend capability metadata, and final validation.
This is the shared semantic spine for generated PySpark rendering and online execution.

Milestone 2 is partly complete. `render_pyspark_schema(...)` now renders Spark-free schema constant expressions for
primitive, decimal, array, map, nested struct, and inherited schemas. Runtime materialization of live `StructType`
objects remains for the online execution milestone because it legitimately requires PySpark at runtime.
`render_pyspark_schema_module(...)` now renders full schema module text in memory, including deterministic imports for
nested struct schema constants from other generated schema modules.

The expression renderer is complete for the v1 fixture operations. It renders field references, literals, helper calls,
null checks, boolean combinations, equality, null-safe equality, subtraction, and greater-than comparisons from shared
recipes. Transform module rendering can now compose these expression strings into `.where(...)`, `.join(...)`, and
`.select(...)` blocks.

The PySpark target code now lives under `src/structure/app/backend/pyspark`. Tests import
`structure.app.backend.pyspark.api`, and the old standalone target guidance in the online/generated semantic contract
plan was updated.

Backend recipe classes, DSL type model classes, public DSL type constructor classes, and transform enum classes now
live one class per source file. Compatibility aggregator modules remain class-free for imports that benefit from a
single module.

The generated step renderer now renders source blocks for subtransform comments, aliasing, joins, filters, projections,
hooks, schema assertions, and hook `project_output` projection from `PySparkStepRecipe` values.

## Context and Orientation

The current implementation is a compact compiler core under `src/structure/app/dsl`. `compile_transform(...)` produces
`TransformPlan`, `InputPlan`, `StepPlan`, `ProjectAssignment`, `JoinPlan`, and `HookPlan` values. The v1 fixture at
`res/testing/model/v1/orders/transforms/order.py` exercises the intended v1 DSL:

- `normalize`: filters, expression helpers, decimal conversion, arithmetic, comparison, and a before/after hook.
- `add_customer`: broadcast left `join_one(...)` with composite keys.
- `add_product`: left `join_one(...)` plus a joined-field filter.
- `add_promotion`: left `join_one(...)` with null-safe equality and an after hook that allows extra columns.
- `publish`: schema projection through `Structure.base(...)` and a final after hook with `project_output=True`.

The next internal target is a `PySparkExecutionPlan`. A recipe is not generated source code and not live PySpark; it is
a deterministic description of how PySpark should execute the checked `TransformPlan`. Generated code renders recipes
as Python source text. Online execution interprets the same recipes with live DataFrames and Columns at runtime.

## Plan of Work

Milestone 1 adds Spark-free PySpark recipes. Create focused dataclasses under a PySpark backend target package, such as
`src/structure/app/backend/pyspark/logic/model/recipes.py`, and a lowering action, such as
`src/structure/app/backend/pyspark/logic/actions/LowerPySparkPlan.py`. The lowering should consume `TransformPlan` and
`PySparkCapabilities`, preserve step order, convert expressions into expression recipes, attach joins and hooks, and
create validation recipes in canonical order. Tests should prove the v1 fixture lowers deterministically without
importing PySpark.

Milestone 2 renders schema and expression materialization. Add type-to-Spark-schema recipes and source rendering for
primitive, decimal, array, map, nested struct, and inherited fields. This milestone should still be Spark-free and
validated by snapshot-like string tests.

Milestone 3 implements generated source rendering. Generate package markers, schema modules, runtime helpers,
transform modules, and lineage files in memory. Add write-if-changed after generated text is deterministic. Generated
files may import PySpark because they are runtime artifacts, but the compiler that writes them must not import PySpark.

Milestone 4 implements configuration and CLI compiler commands. Add effective config resolution from defaults,
`structure.toml`, `[tool.structure]`, and CLI flags. Implement `structure check`, `compile`, `compile --fail-on-diff`,
and `explain` on top of the same compiler pipeline. Add `init`, `clean`, and `--profile` after the core pipeline is
stable.

Milestone 5 implements runtime execution. Add `StructureSession`, online runner, generated runner, runtime schema
materialization, hook input namespace, schema-only validation helpers, and output schema exposure. Runtime tests may be
separate when PySpark is not installed locally.

Milestone 6 hardens diagnostics, streaming compatibility, provenance, lineage, and negative tests. Add the diagnostic
registry before routing many errors through it, then migrate compiler/runtime/CLI failures into structured diagnostics.
Add streaming reports, deterministic provenance, static dataflow lineage, and missing/invalid source tests.

Milestone 7 completes v1 traceability. Once behavior is implemented and tested, update `docs/dev/Specification.md` by
prefixing only completed backed user stories with `+`, and add or update troubleshooting entries for issues actually
encountered during implementation.

## Idempotence and Recovery

All compiler milestones should be additive and rerunnable. Generated output should be built in memory before writing and
written only when content changes. `compile --fail-on-diff` must write to a temporary directory and clean it afterward.
`clean` must remove only Structure-owned generated artifacts under the configured generated directory.

If a milestone breaks v0 or the symbolic v1 compile tests, restore the earlier contract first, then add new behavior
through additive fields, adapters, or recipe layers.

## Interfaces and Dependencies

Milestone 1 should introduce an internal API equivalent to:

    lower_pyspark_plan(plan: TransformPlan, capabilities: BackendCapabilities | None = None) -> PySparkExecutionPlan

The recipe model should include at minimum:

- `PySparkExecutionPlan`
- `PySparkInputRecipe`
- `PySparkStepRecipe`
- `PySparkExpressionRecipe`
- `PySparkProjectionRecipe`
- `PySparkJoinRecipe`
- `PySparkHookRecipe`
- `PySparkValidationRecipe`

These classes are internal implementation details, not public DSL API. They must not contain live Spark sessions,
DataFrames, Columns, generated source text, file handles, or runtime hook return values.
