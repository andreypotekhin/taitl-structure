# Add Local Infrastructure and Backend-Version Integration Testing

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [Plans.md](../auto/Plans.md). It is self-contained so a contributor can add Docker Compose based
local infrastructure and opt-in integration tests without needing prior conversation history.

## Purpose / Big Picture

Structure claims compatibility with ordinary PySpark 3.5.x and 4.0.x. Unit tests, compiler tests, generated-code
snapshots, and no-Spark import guards prove many parts of that claim, but they do not prove that an end-to-end
Structure transform actually runs on every claimed runtime line. After this work, a contributor can run a single
opt-in command that starts local Spark infrastructure, executes Structure transforms against each claimed backend
version, and reports which compatibility claims are backed by live evidence.

The default project build must remain fast and Spark-free. `make build` and `poetry run pytest` must not start Docker,
import PySpark, start Java, create a SparkSession, or contact a Spark cluster. Integration testing is explicit:
developers run `make integration`, `make build INTEGRATION=1`, or a focused pytest command with `--run-integration`
when they want live backend evidence.

## Surprises & Discoveries

- Observation: `tests/integration` already exists.
  Evidence: `Get-ChildItem -Path tests -Force` lists `integration`, while `Get-ChildItem -Path tests/integration`
  returns no files.

- Observation: The public compatibility policy already claims PySpark 3.5.x and 4.0.x by default.
  Evidence: [Compatibility.md](../../Compatibility.md) says `target_pyspark = ">=3.5,<4.1"`.

- Observation: The repository already has a hard design decision that compiler commands and ordinary tests must remain
  Spark-free.
  Evidence: [D06182606.No-spark-compile-dependency.md](../design/decisions/D06182606.No-spark-compile-dependency.md) says compiler tests must run without
  PySpark, Java, SparkSession, Spark startup, or a Spark cluster.

- Observation: `pyproject.toml` originally had pytest configuration but no integration marker.
  Evidence: `[tool.pytest.ini_options]` contained `addopts` and `pythonpath`, but no `markers` entry before this
  implementation.

- Observation: The pytest integration option cannot live in a root `tests/conftest.py` because that location does not
  convey the integration context.
  Evidence: The plugin now lives at `tests/integration/pytest_plugin.py` and loads through pytest's `-p` option.

- Observation: The ordinary pytest suite was green during the first implementation pass, but `make build` did not
  complete in that workspace because `poetry run black --check src tests` timed out rather than reporting a formatting
  error.
  Evidence: `poetry run pytest` reports `61 passed in 0.78s`; `make build` timed out after 300 seconds during Black.

- Observation: The runtime did not yet include a built-in live online PySpark executor during the first infrastructure
  pass.
  Evidence: `RunOnlinePySparkTransform` delegates to `session.online_executor`; without one it raises
  `ONLINE-E1202`.

- Observation: Docker Compose configuration rendered, but live Docker-backed execution was initially blocked in this
  workspace because Docker Desktop's Linux engine was not running or reachable.
  Evidence: `docker compose --env-file infra/compose/.env -f infra/compose/docker-compose.yaml config` exits 0;
  `python scripts/run_integration.py --backend pyspark35` fails with `open //./pipe/dockerDesktopLinuxEngine: The
  system cannot find the file specified`.

- Observation: The original PySpark test image used a floating Debian base that no longer provided
  `openjdk-17-jre-headless`.
  Evidence: `make integration` failed while building `python:3.11-slim`; pinning the image to
  `python:3.11-slim-bookworm` restored the Java package used by Spark.

- Observation: Spark 4 creates local artifact directories relative to the process working directory during test
  startup.
  Evidence: the PySpark 4 lane failed on `/workspace/artifacts` because the repository is mounted read-only; running
  pytest from `/tmp` while keeping `/workspace` as pytest root made the lane pass without weakening the read-only
  source mount.

- Observation: Compose relative paths can resolve outside the repository when callers provide a different
  `--project-directory`.
  Evidence: rendering the Compose file with `--project-directory .` resolved `../..` to `/Users/chaos/Files/Dev`;
  rendering with `STRUCTURE_ROOT=/Users/chaos/Files/Dev/Code/taitl-structure` keeps build contexts and bind mounts on
  the repository root. The runner now supplies `STRUCTURE_ROOT` explicitly.

- Observation: Global projection casts make generated PySpark substantially less readable and less human-like.
  Evidence: same-type projections such as `F.col("customers.name").alias("customer_name")` and
  `F.lower(F.trim(F.col("order_raw.id"))).alias("id")` do not need casts; decimal arithmetic such as `net_total`
  still needs `cast(T.DecimalType(12, 2))` because Spark widens decimal subtraction.

## Decision Log

- Decision: Keep integration tests opt-in through pytest flags and Makefile targets, not through default pytest
  discovery.
  Rationale: This preserves the no-Spark compiler contract and keeps ordinary contributors productive without Docker,
  Java, or PySpark installed on the host.
  Date/Author: 2026-06-21 / Codex

- Decision: Put local infrastructure under `infra/compose/docker-compose.yaml` with a tracked
  `infra/compose/.env_example` and an untracked `infra/compose/.env`.
  Rationale: This matches the requested layout, keeps local port/image choices out of source control, and lets CI use
  the same Compose file with CI-specific environment values. The `_example` suffix avoids accidental confusion with a
  live `.env` file while remaining obvious to copy.
  Date/Author: 2026-06-21 / Codex

- Decision: Run version-matrix tests in version-matched containers, not in the host Poetry environment.
  Rationale: A host environment can only import one PySpark version at a time. Separate containers make the PySpark
  version, Spark server version, Python version, and Java runtime explicit and repeatable.
  Date/Author: 2026-06-21 / Codex

- Decision: Launch all currently claimed backend versions by default on distinct services and ports.
  Rationale: The normal integration path should start the whole compatibility matrix once, then run tests against every
  live backend. Focused single-backend runs remain useful for debugging, but the default should prove all-version
  infrastructure rather than one selected lane.
  Date/Author: 2026-06-21 / Codex

- Decision: Treat online/generated parity on live Spark as the first integration evidence.
  Rationale: Structure's strongest runtime claim is that online execution and generated PySpark execution preserve the
  same semantics on supported Spark runtimes. Backend-specific feature breadth can grow after this parity spine exists.
  Date/Author: 2026-06-21 / Codex

- Decision: Supply the repository root to Compose through `STRUCTURE_ROOT` and use long-form bind mounts.
  Rationale: Long-form mounts avoid Windows drive-letter ambiguity in short volume syntax, and an explicit root keeps
  build contexts stable even when Compose is invoked with a different project directory.
  Date/Author: 2026-06-28 / Codex

- Decision: Emit PySpark casts and aliases only when needed for schema correctness.
  Rationale: Generated PySpark should stay close to idiomatic hand-written code; casts are reserved for untyped nulls,
  assignable type changes, and Spark expression cases with known widening behavior such as decimal subtraction.
  Date/Author: 2026-06-28 / Codex

## Context and Orientation

Structure is a Python package rooted at `src/structure`. It lets users write schema-oriented transforms and run them
online through `StructureSession` or compile them into generated PySpark code. The public compatibility policy in
[Compatibility.md](../../Compatibility.md) currently claims Python 3.11 and newer, ordinary PySpark 3.5.x and 4.0.x, and Linux runtime
environments for online and generated PySpark execution.

The project already separates Spark-free compiler behavior from live runtime behavior. Spark-free means code can be
imported, checked, compiled, and tested without importing `pyspark`, starting Java, creating a `SparkSession`, or
contacting a Spark cluster. Live runtime behavior means tests intentionally run with PySpark and Spark available.

Integration tests in this plan are live runtime tests. They belong under `tests/integration`. They are not replacements
for tests under `tests/specs`, `tests/specifications`, `tests/app`, or `tests/lib`. Instead, they are compatibility
evidence: small end-to-end scenarios that exercise the same public paths users rely on.

Docker Compose is the local infrastructure orchestrator. Compose starts related containers on one local network. A
backend lane is one version-specific Spark runtime plus the test settings needed to reach it. The default Compose stack
starts every backend lane that Structure currently claims to support, with distinct service names and host ports.
Focused single-backend runs may still exist for debugging, but they are a convenience over the all-version stack.

## Version Matrix

The initial matrix must cover the currently claimed ordinary PySpark range:

- `pyspark35`: Python 3.11 or 3.12, PySpark 3.5.x, Spark 3.5.x, Java version compatible with Spark 3.5.
- `pyspark40`: Python 3.11 or 3.12, PySpark 4.0.x, Spark 4.0.x, Java version compatible with Spark 4.0.

The plan intentionally says `3.5.x` and `4.0.x` rather than pinning exact patch versions in prose. Exact image tags
belong in `infra/compose/.env_example`, and the first implementation should choose conservative, explicit patch
versions that are available in the selected image source at implementation time. When patch versions are upgraded, the
developer changes the `.env_example` defaults and records the reason in this plan.

Future matrix lanes should be additive:

- `pyspark35-generated`: generated-code import and execution on PySpark 3.5.x if it needs different setup from online.
- `pyspark40-generated`: generated-code import and execution on PySpark 4.0.x if it needs different setup from online.
- `spark-connect`: deferred until Spark Connect becomes part of the compatibility claim.
- `streaming`: deferred until v3 streaming orchestration has an executable contract.

## Plan of Work

Start with infrastructure, then add the test-selection contract, then add one narrow live test, and only then broaden
the matrix. This order keeps the implementation recoverable: if Docker wiring fails, the ordinary test suite still
proves the rest of the project.

Create the local infrastructure directory:

    infra/compose/docker-compose.yaml
    infra/compose/.env_example
    infra/compose/README.md
    scripts/ensure_compose_env.py

The Compose file should define one shared project network, one test-runner image definition, and all currently claimed
backend-version services. The default stack should launch PySpark 3.5.x and PySpark 4.0.x at the same time on distinct
ports. A first implementation may use public Spark/PySpark images if they are stable and explicit. If public images
prove awkward, prefer tiny repository-owned Dockerfiles under `infra/compose/images/pyspark35` and
`infra/compose/images/pyspark40` so the runtime is transparent. The Compose file must mount the repository read-only
where possible, create a writable test output directory for pytest caches and reports, and pass only environment
variables needed for tests.

Create `infra/compose/.env_example` with local defaults for image tags, ports, project name, and optional pytest
arguments. Do not commit `infra/compose/.env`. Update `.gitignore` to ignore `infra/compose/.env`, root `.env`, and
local override files such as `.env.local`. The example file should be enough for a developer to inspect, but ordinary
commands should not require a manual copy. Add `scripts/ensure_compose_env.py` as a tiny standard-library helper that
copies `infra/compose/.env_example` to `infra/compose/.env` only when `.env` is absent. It must never overwrite an
existing `.env`.

    poetry run python scripts/ensure_compose_env.py

Do not put secrets in either file. Local Spark integration should not need credentials.

Add pytest selection support in `tests/integration/pytest_plugin.py`. Register the `integration` marker and add
options:

    --run-integration
    --integration-backend=pyspark35|pyspark40|all

When `--run-integration` is absent, tests marked `integration` must be skipped with a clear reason. This skip behavior
is better than relying only on `-m` selection because `poetry run pytest` remains harmless even if a developer creates
new integration tests and forgets a marker expression. The marker itself remains required so contributors can run
focused selections such as `poetry run pytest -m integration --run-integration`.

Add Makefile targets without changing the default build behavior:

    make integration
    make integration BACKEND=pyspark35
    make integration BACKEND=pyspark40
    make build INTEGRATION=1

`make integration` should run both initial backend lanes. `BACKEND=pyspark35` or `BACKEND=pyspark40` should run one
lane's pytest selection against the all-version infrastructure. `make build INTEGRATION=1` should run the ordinary
build first, then the integration matrix. Plain `make build` must keep its current behavior and must not start Compose.
Every integration Make target should depend on the `.env` bootstrap helper before calling Docker Compose.

Add the first live tests under `tests/integration/pyspark`. The first scenario should reuse the v1 orders model from
`res/testing/model/v1/orders` and generated output from `res/testing/model/v1/structure_generated` because those
fixtures already represent the happy path. The test should:

1. create or receive a live `SparkSession`;
2. build small `orders`, `customers`, and `products` DataFrames;
3. run the source transform online through `StructureSession`;
4. run the generated PySpark class for the same inputs;
5. compare row contents, column order, and schema shape where Spark exposes it reliably;
6. assert no forbidden fallback appears in the generated/runtime path, such as Python UDFs, row maps, `collect` inside
   Structure code, or `toPandas`.

The test may call Spark actions such as `collect()` at the assertion boundary. Structure implementation code must not
call those actions implicitly.

Store integration input data under `res/testing/data/v1/orders`. Prefer CSV fixtures over Python row factory methods
for tabular inputs so test data is reviewable independently from test control flow. If a future data shape no longer
fits CSV cleanly, add a sibling JSON fixture with the same versioned directory convention.

Add a small infrastructure smoke test that runs before the expensive parity test. It should assert the Spark version,
PySpark version, Python version, and selected backend lane, then print or record them in pytest output. This makes
compatibility evidence auditable when CI logs are reviewed later.

Document the workflow in `infra/compose/README.md` and update [Testing.md](../Testing.md) with a short "Integration Tests"
section. If implementation reveals common local failures, add remedies to [Troubleshooting.md](../Troubleshooting.md). Examples
include Docker not running, backend service typo, image pull failure, Java/Spark startup failure, and port collision.

## Milestones

Milestone 1 adds the Compose skeleton and environment contract. At the end of this milestone, `infra/compose` exists,
`.env_example` documents local settings, `.gitignore` protects local `.env` files, `scripts/ensure_compose_env.py`
copies `.env_example` to `.env` only when `.env` is missing, and `docker compose --env-file infra/compose/.env -f

behavior changes yet.

Milestone 2 adds pytest opt-in semantics. At the end of this milestone, `pyproject.toml` registers the `integration`
marker, pytest accepts `--run-integration` and `--integration-backend`, and a placeholder marked test under
`tests/integration` is skipped by default but selected when the flag is present. Run `poetry run pytest` and confirm it
does not require Docker or PySpark.

Milestone 3 adds the Makefile integration command path. At the end of this milestone, `make integration` bootstraps
`.env`, starts the all-version stack, and runs both PySpark lanes. `make integration BACKEND=pyspark35` and
`make integration BACKEND=pyspark40` still start the same all-version stack, then select one backend's tests for
debugging. `make build INTEGRATION=1` runs the default build followed by the matrix. Plain `make build` remains
unchanged.

Milestone 4 adds live infrastructure smoke tests. At the end of this milestone, each lane can start Spark and run a
small pytest test that reports the backend lane, Python version, PySpark version, and Spark runtime version. This proves
the local infrastructure is usable before Structure behavior enters the picture.

Milestone 5 adds the first end-to-end generated execution test. At the end of this milestone, the v1 orders scenario
runs through `StructureSession(execution_mode="generated")` against each backend lane and the observed output matches
the expected rows and schema. This is the first live evidence supporting the generated PySpark side of the PySpark
3.5.x and 4.0.x compatibility claim.

Milestone 5b adds true online/generated parity after the live online PySpark executor exists. At the end of this
milestone, the same v1 orders scenario runs online and generated execution against each backend lane, and the observed
output is identical.

Milestone 6 broadens coverage to backend-sensitive behavior. Add tests for nullability/type coercion, serial joins,
hooks, validation placement, and generated schema modules as those features become stable. Each new public runtime
claim should have at least one integration test on each supported backend version before the claim is marked complete
in documentation.

Milestone 7 wires documentation and troubleshooting. At the end of this milestone, [Testing.md](../Testing.md),
`infra/compose/README.md`, and [Troubleshooting.md](../Troubleshooting.md) explain how to run, debug, and interpret integration
tests. If the compatibility policy changes, update [Compatibility.md](../../Compatibility.md) in the same change.

## Validation and Acceptance

Acceptance is met when a human can verify all of the following from the repository root:

- `poetry run pytest` does not start Docker, does not import PySpark through Structure internals, and skips integration
  tests unless `--run-integration` is present.
- `make build` keeps the current default behavior and does not run integration tests.
- `make build INTEGRATION=1` runs the ordinary build and then the integration matrix.
- `make integration` creates `infra/compose/.env` from `.env_example` if needed, starts every currently claimed
  backend version, and runs the full integration matrix.
- `make integration BACKEND=pyspark35` starts the all-version stack and runs only the PySpark 3.5 test selection.
- `make integration BACKEND=pyspark40` starts the all-version stack and runs only the PySpark 4.0 test selection.
- Each lane records Python, PySpark, and Spark versions in test output.
- The v1 orders generated-execution scenario produces the expected output on PySpark 3.5.x and PySpark 4.0.x.
- After the live online PySpark executor exists, the v1 orders scenario produces identical online and generated outputs
  on PySpark 3.5.x and PySpark 4.0.x.
- Integration containers stop cleanly after tests pass or fail.
- [Testing.md](../Testing.md) and `infra/compose/README.md` explain the commands clearly enough for a new contributor.

The first implementation should not be declared complete until at least one lane has been run locally and the remaining
lane is either passing or has a recorded blocker with exact failure output.

## Idempotence and Recovery

The workflow must be safe to rerun. Compose services should use project-scoped names from `.env` so repeated runs
replace the same local infrastructure instead of accumulating orphaned containers. Test output should go into a
dedicated ignored directory such as `.pytest-workspace-tmp/integration`.

The `.env` bootstrap must also be safe to rerun. If `infra/compose/.env` already exists, leave it untouched. If it does
not exist, copy `infra/compose/.env_example` to `infra/compose/.env` and print a short message telling the developer
which file was created. If `.env_example` is missing, fail early with a direct error.

If Compose validation fails, run:

    docker compose --env-file infra/compose/.env -f infra/compose/docker-compose.yaml config

and fix the YAML or environment values before running tests. If containers remain after a failed run, stop only this
project's Compose stack:

    docker compose --env-file infra/compose/.env -f infra/compose/docker-compose.yaml down --remove-orphans

If one backend lane fails because an image tag vanished or a Spark patch version changed, update
`infra/compose/.env_example`, record the exact old and new tags in this plan, and rerun the smoke test before rerunning
the full parity test.

If a live Spark semantic difference is found between PySpark 3.5.x and 4.0.x, do not paper over it in tests. Record it
in `Surprises & Discoveries`, decide whether Structure should normalize behavior or report a backend capability
diagnostic, and update the compatibility documentation if the claim must narrow.

## Artifacts and Notes

The initial Compose topology should look like this:

    default all-version stack:
      spark35-master
      spark35-worker
      spark40-master
      spark40-worker
      structure-integration-pyspark35
      structure-integration-pyspark40

Spark 3.5 and Spark 4.0 services must expose different host ports for local inspection. Test runners should execute
pytest from inside version-matched containers so the installed PySpark package matches the Spark server lane. The host
should need Docker and Docker Compose, not Java or PySpark.

The pytest marker contract should be visible in test files:

    pytestmark = pytest.mark.integration

The Makefile contract should be explicit:

    make build              # ordinary build, no integration
    make build INTEGRATION=1 # ordinary build plus live backend matrix
    make integration         # live backend matrix only

The first parity test should favor tiny data. Integration tests prove live compatibility; they are not stress tests.
Performance and scale tests can be added later as separate benchmark or stress lanes.

## Interfaces and Dependencies

Use Docker Compose as the only infrastructure orchestrator for this plan. Do not require developers to install Java,
Spark, or PySpark on the host for integration tests.

Use pytest markers and command-line options for test selection:

    @pytest.mark.integration
    --run-integration
    --integration-backend=pyspark35|pyspark40|all

Use environment variables for runtime details visible to tests:

    STRUCTURE_INTEGRATION_BACKEND
    STRUCTURE_SPARK_MASTER
    STRUCTURE_SPARK35_MASTER
    STRUCTURE_SPARK40_MASTER
    STRUCTURE_EXPECTED_PYSPARK
    STRUCTURE_EXPECTED_SPARK

Do not add PySpark to the default Poetry dependency group. If host-side helper dependencies are needed, keep them in
the dev group only when they are Spark-free. PySpark belongs in the integration containers.

The initial implementation should avoid adding a new Python orchestration library. Makefile plus Docker Compose plus
pytest is enough. If orchestration becomes complex later, propose a separate plan for a small Python runner or pytest
plugin rather than hiding complexity inside shell snippets.
