## Purpose / Big Picture

After this work, Structure users can express common analytical joins without hiding business logic inside hooks. They
can filter by existence, intentionally multiply rows, choose one lookup row by a deterministic policy, and model
time-aware lookups while preserving compiler diagnostics, static traceability, generated-code reviewability, and
online/generated parity.

The observable result is a set of small Spark parity fixtures. A developer can run tests showing that `exists(...)`,
`not_exists(...)`, `join_many(...)`, deduped `join_one(...)`, `temporal_one(...)`, and backward `as_of_one(...)` produce
the same rows online and in generated PySpark.

## Surprises & Discoveries

- Observation: The v1 `JoinSemantics.md` already deliberately excludes semi, anti, temporal, and row-multiplying
  joins.
  Evidence: `JoinSemantics.md` says `join_one(...)` is narrow and defers `join_many(...)`; C27 expands the deferred
  set into a staged feature family.

- Observation: Backend capability checks already have a `join` group.
  Evidence: `docs/specifications/BackendCapabilities.md` lists `join.join_one` and an unsupported `join.join_many`
  example, so analytical joins can use the same extension point.

## Decision Log

- Decision: Keep v1 `join_one(...)` unchanged.
  Rationale: A lookup join that silently dedupes right rows would break cardinality transparency and generated-code
  reviewability.
  Date/Author: 2026-06-21 / Codex

- Decision: Start v2 analytical joins with existence predicates and `join_many(...)`.
  Rationale: They cover frequent analytical use cases and have simpler cardinality than temporal or as-of joins.
  Date/Author: 2026-06-21 / Codex

- Decision: Model existence joins as predicates, not joined scopes.
  Rationale: Semi and anti joins do not expose right-side fields. Predicate syntax keeps the source shape honest.
  Date/Author: 2026-06-21 / Codex

- Decision: Require deterministic policies for deduped lookups, temporal joins, and as-of joins.
  Rationale: Structure should never hide nondeterministic row choice behind convenient API names.
  Date/Author: 2026-06-21 / Codex

## Context and Orientation

Structure is a Python library for typed Spark transforms. Developers write schema-returning transform methods. The
compiler symbolically executes those methods into backend-neutral `TransformPlan` IR. A PySpark target layer lowers
checked IR into recipes consumed by both online execution and generated PySpark source.

Current v1 join support is `join_one(...)`, specified in `docs/specifications/JoinSemantics.md`. It enriches the
current row with fields from one right input and promises that each current row matches zero or one right row.

C27 exists because common analytical pipelines need more join shapes:

- Semi joins keep rows that have a match.
- Anti joins keep rows that do not have a match.
- Row-multiplying joins intentionally produce one output row per match.
- Dedupe lookups choose one right row by a deterministic policy.
- Temporal joins match a right row whose validity window contains an event time.
- As-of joins choose the nearest right-side time at or before a current-row time.

## Plan of Work

Start with capability gates. Add unsupported capability decisions for every analytical join form. Then promote
`join.exists` and `join.not_exists` for the ordinary PySpark target when the implementation exists. Capability checks
must run without importing PySpark.

Implement existence joins first. Add methods on symbolic input scopes named `exists(...)` and `not_exists(...)`. They
accept `on` and optional `hint`, validate the same equi-join condition model as `join_one(...)`, and return symbolic
boolean predicates. Record them as join operations with row-filtering cardinality. Lower them to shared PySpark recipes
and add online/generated parity tests where duplicate right rows do not change output.

Implement `join_many(...)` next. It should return a joined scope and record row-multiplying cardinality. Support
`Join.INNER` and `Join.LEFT` first. Add tests proving right duplicates multiply current rows and left joins preserve
unmatched current rows.

Then add deterministic lookup dedupe. Extend `join_one(...)` with a `dedupe` policy object. Implement
`JoinDedupe.latest_by(...)` and `JoinDedupe.earliest_by(...)` with `TiePolicy.ERROR`. Lower the right-side reduction
through a shared recipe before the join recipe. Add diagnostics for unproven ties and generated code that makes the
selection rule reviewable.

Add temporal validity joins after dedupe support is stable. Implement `temporal_one(...)` with `on`, `at`,
`valid_from`, `valid_to`, `how`, and `overlaps`. Use closed-open interval semantics: `valid_from <= at < valid_to`.
Treat null `valid_to` as open-ended. Add optional overlap runtime checks only when explicitly enabled.

Add backward as-of joins last. Implement `as_of_one(...)` for `AsOf.BACKWARD`, optional `tolerance`, and `Join.LEFT` or
`Join.INNER`. Defer forward and nearest direction until backward joins are stable.

For every slice, update static traceability, `structure explain`, diagnostics, generated-code snapshots, online execution,
generated execution, and streaming compatibility classification.

## Validation and Acceptance

The documentation phase is accepted when:

- `docs/specifications/AnalyticalJoinCoverage.md` exists;
- `docs/dev/design/AnalyticalJoinCoverage.md` exists;
- D06212601 exists;
- this ExecPlan exists;
- C27 in `docs/dev/design/Challenges.md` points to those documents;
- project-management docs schedule the work under v2 analytical pipeline features.

The implementation phase is accepted when tests prove:

- `exists(...)` keeps only current rows with matching right rows;
- `not_exists(...)` keeps only current rows without matching right rows;
- right duplicates do not affect existence output;
- `join_many(...)` multiplies rows intentionally;
- left `join_many(...)` preserves unmatched rows with null right fields;
- deduped `join_one(...)` selects the latest or earliest right row by an explicit ordering policy;
- unproven dedupe ties produce a diagnostic;
- `temporal_one(...)` matches closed-open validity windows;
- null `valid_to` means open-ended current record;
- optional overlap checks catch overlapping right windows;
- backward `as_of_one(...)` selects the latest right row at or before the left time;
- as-of tolerance rejects distant matches;
- every supported form has backend capability checks, IR snapshots, generated-code snapshots, online/generated parity,
  diagnostics, static traceability, and streaming compatibility classification.

## Idempotence and Recovery

Implement the work in additive slices. Do not modify `join_one(...)` behavior while adding existence joins or
`join_many(...)`. If dedupe changes existing `join_one(...)` generated snapshots when `dedupe=None`, stop and isolate
the new policy path.

If PySpark is unavailable locally, run no-Spark compiler, capability, and IR tests. Record that Spark parity tests were
not run. Do not weaken the no-Spark compiler contract to make join work easier.
