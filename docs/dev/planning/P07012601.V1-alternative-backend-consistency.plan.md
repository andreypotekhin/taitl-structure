# P07012601 V1 Alternative Backend Consistency

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can reconcile the current v1
implementation with the newly advertised alternative-backend documentation without needing prior conversation history.

## Purpose / Big Picture

Structure currently supports PySpark as the only v1 execution backend, while the documentation now advertises near-term
alternative-backend syntax for v2 and beyond. The goal of this plan is to make v1 honest and safe: advertised future
configuration keys, CLI flags, and hook target metadata should not crash immediately when used in a PySpark project,
but v1 must still fail fast for actual non-PySpark execution. After this change, a user can keep PySpark as the active
backend, add reserved compatibility settings such as `compat_targets = ["polars"]`, mark a hook with
`target_backend="pyspark"`, run `structure check`, and see clear output that active PySpark checks ran while
non-PySpark compatibility is pending.

This plan does not implement Polars, DuckDB, Spark SQL, Ibis, or other target lowering. It preserves the advertised
features in the docs because v2 work is expected soon, but it prevents the v1 docs from describing options that the
resolver, CLI, or decorators reject as unknown.

## Surprises & Discoveries

- Observation: `docs/specifications/ConfigSchema.md` currently advertises `hook_target_default`, `target_profile`, and
  `compat_targets`, but `ResolveStructureConfig._keys` only accepts the v1 keys through `target_pyspark`.
  Evidence: `src/structure/app/configuration/commands/ResolveStructureConfig.py` lists only `target_backend` and
  `target_pyspark` in this area.

- Observation: The CLI documentation advertises compatibility flags, but Click only exposes `--target-backend` and
  `--target-pyspark`.
  Evidence: `src/structure/app/cli/api/cli.py` defines `_config_options` without `--target-profile` or
  `--compat-targets`.

- Observation: Hook target scoping is documented as an option, but `before(...)`, `after(...)`, `_hook(...)`, and
  `HookPlan` do not accept or store it.
  Evidence: `src/structure/app/dsl/model/transforms/transform_api.py` has no `target_backend` parameter on hook
  decorators, and `src/structure/app/compiler/ir/model/HookPlan.py` has only existing hook fields.

- Observation: `project(...)` is a real public export and is documented in QuickRef, but the DSL spec public import
  block omits it.
  Evidence: `src/structure/__init__.py` and `src/structure/app/dsl/api/__init__.py` export `project`; `docs/QuickRef.md`
  teaches it; `docs/specifications/DSL.md` does not list it.

- Observation: Backend-family terms currently mix implementation family and future semantic family.
  Evidence: v1 code and `docs/specifications/BackendCapabilities.md` use `ordinary_pyspark`; the alternative-backend
  docs use names such as `pyspark_dataframe` and `distributed_relational`.

## Decision Log

- Decision: Do not remove advertised alternative-backend features from the docs.
  Rationale: v2 is expected soon, and removing the syntax would churn the public story. The safer v1 fix is to mark the
  syntax as reserved/pending where behavior is not implemented and make the current code accept those knobs without
  pretending to execute non-PySpark targets.
  Date/Author: 2026-07-01 / Codex

- Decision: Keep v1 executable backend support limited to `target_backend = "pyspark"`.
  Rationale: This plan resolves contract drift. It does not implement a new backend adapter, SQL renderer, or
  compatibility matrix.
  Date/Author: 2026-07-01 / Codex

- Decision: Add reserved config and CLI support for future keys instead of treating them as unknown.
  Rationale: The docs already advertise the keys. Accepting, validating, storing, and reporting them avoids a bad user
  experience while preserving fail-fast behavior for actual unsupported execution.
  Date/Author: 2026-07-01 / Codex

- Decision: Treat `target_backend` on hooks as active v1 metadata for PySpark hooks.
  Rationale: A user following the documented hook example should not get an unexpected keyword error. V1 should accept
  the metadata, carry it through IR, and reject hook target declarations that do not include the active PySpark target.
  Date/Author: 2026-07-01 / Codex

- Decision: Distinguish current backend implementation family from future semantic family.
  Rationale: Existing code uses `ordinary_pyspark` and tests likely depend on it. Future docs can introduce
  `semantic_family` vocabulary without forcing a v1 code migration.
  Date/Author: 2026-07-01 / Codex

## Context and Orientation

The repository root is `C:\Taitl\Code\taitl\taitl-structure`. Structure uses a small public Python DSL exported from
the `structure` package. A "backend" is the concrete execution target selected by configuration. In v1 the only
executable backend is ordinary PySpark, represented by `target_backend = "pyspark"` and `target_pyspark = ">=3.5,<4.1"`.

The configuration pipeline starts in `src/structure/app/configuration/commands/ResolveStructureConfig.py`. It merges
defaults from `StructureConfigDefaults`, user files, and CLI overrides, validates values through
`StructureConfigValidator`, and builds a frozen `StructureConfig` through `StructureConfigBuilder`.

The CLI is implemented with Click in `src/structure/app/cli/api/cli.py`. The helper `_config_options` declares options
shared by `structure check`, `structure compile`, `structure explain`, and `structure clean`.

Hooks are escape hatches for arbitrary DataFrame code. Their decorators live in
`src/structure/app/dsl/model/transforms/transform_api.py`. Hook metadata is collected into
`src/structure/app/compiler/ir/model/HookPlan.py` and later mapped into PySpark target recipes.

Backend capabilities are static metadata under `src/structure/app/target/capabilities`. The current backend id has
`family = "ordinary_pyspark"`. Alternative-backend docs should call this the current implementation family and reserve
semantic-family names such as `pyspark_dataframe`, `sql_relation`, and `local_lazy_dataframe` for compatibility
reporting.

## Plan of Work

First, update the documentation to state the v1/v2 boundary precisely while preserving advertised syntax. In
`docs/specifications/ConfigSchema.md`, keep `target_profile`, `compat_targets`, and `hook_target_default`, but label
them as v1-recognized reserved fields. State that only PySpark active execution is supported in v1; non-PySpark
compatibility targets are recorded and reported as pending. In `docs/specifications/CLI.md`, keep the future flags but
describe v1 output as an honest pending compatibility summary rather than a full compatibility matrix. In
`docs/specifications/HookSemantics.md`, state that `target_backend` metadata is accepted in v1 and must include
`pyspark` for hooks that run under the current backend. In `docs/specifications/DSL.md`, add `project` to the public
import block. In `docs/specifications/BackendCapabilities.md` and `docs/specifications/AlternativeBackends.md`, define
the split between implementation family and semantic family.

Second, add reserved configuration fields. In `StructureConfigDefaults`, add:

    target_profile = None
    compat_targets = []
    hook_target_default = ["pyspark"]

In `ResolveStructureConfig._keys`, add those three keys. In `StructureConfig`, add `target_profile: str | None`,
`compat_targets: tuple[str, ...]`, and `hook_target_default: tuple[str, ...] | str`. In `StructureConfigBuilder`,
normalize `compat_targets` and list-form `hook_target_default` to tuples. In `StructureConfigValidator`, validate that
`target_profile` is `None` or a string, `compat_targets` is a list of strings, and `hook_target_default` is either a
list of strings or the string `"explicit"`. Continue to reject active `target_backend` values other than `pyspark`.

Third, add CLI support that accepts but does not overclaim compatibility. Add `click.option("--target-profile")` and
`click.option("--compat-targets")` to `_config_options`. Parse `--compat-targets` as a comma-separated string and pass
the resulting list through overrides. For `structure check` and `structure explain`, append a compact pending-status
line when `config.compat_targets` is non-empty, such as:

    compatibility targets: polars, duckdb
    compatibility status: non-PySpark target checks are reserved for v2+; active PySpark checks passed

Do not add this line to `compile` or `clean` unless a future spec explicitly asks for it. If `compat_targets` includes
`pyspark`, report it as active and supported by the ordinary PySpark capability checks.

Fourth, add hook target metadata. Extend `before(...)`, `after(...)`, and `_hook(...)` with
`target_backend: str | Iterable[str] | None = None`. Normalize accepted values in a small helper:

    None means inherited default and stores ("pyspark",) with target_defaulted=True in v1.
    "configured" means the active configured backend and stores ("pyspark",) in v1.
    "all" means the hook author claims portability and stores ("all",).
    A string backend id stores that one id.
    A non-empty iterable of strings stores those ids.

Store `target_backend` and `target_defaulted` in `_structure_hook`. Extend `HookPlan` with
`target_backend: tuple[str, ...] = ("pyspark",)` and `target_defaulted: bool = True`. Update
`CompilerHookCollector` and the PySpark hook recipe mapper/model to carry these fields. Add a v1 compileability check
where hook metadata is already validated: if a hook target set is neither `("all",)` nor contains `"pyspark"`, fail
before generation or online execution with a hook diagnostic. If adding a new diagnostic code is too large for this
slice, use the existing hook error path but include the hook name, requested target backend, and suggested fix:
`target_backend="pyspark"`.

Fifth, add tests. Add or extend configuration tests under `tests/specifications/config-schema` so TOML and CLI
overrides with the reserved keys resolve successfully. Add CLI tests under `tests/specifications/cli` that prove
`structure check --compat-targets polars,duckdb` is accepted and prints the pending-status line. Add hook tests under
`tests/user_stories/16_hooks` or `tests/specifications/symbolic-execution` proving
`@after(..., target_backend="pyspark")` compiles and `target_backend="polars"` fails in v1. Add a DSL spec test or
public API snapshot update proving `project` is listed as a public DSL primitive. Add a backend-capabilities
documentation or test assertion only if the current tests inspect family names; do not rename `ordinary_pyspark` in v1
code.

## Validation and Acceptance

Acceptance is met when a minimal project configuration containing the reserved future keys resolves without
`CONF-E0101`:

    [tool.structure]
    target_backend = "pyspark"
    target_pyspark = ">=3.5,<4.1"
    target_profile = ">=3.5,<4.1"
    compat_targets = ["polars", "duckdb"]
    hook_target_default = ["pyspark"]

Acceptance is met when `structure check --compat-targets polars,duckdb` exits successfully for a valid PySpark v1
project and prints an honest pending-status line rather than pretending Polars or DuckDB checks ran.

Acceptance is met when this documented hook shape is accepted:

    @after(normalize, lane=orders, target_backend="pyspark")
    def clean(self, *, orders, spark, ctx):
        return orders

Acceptance is met when a hook declared only for `target_backend="polars"` fails in v1 before runtime with a clear
message explaining that active v1 execution is PySpark only.

Acceptance is met when `docs/specifications/DSL.md` lists `project` in the public import block and the public API
snapshot remains intentional.

Acceptance is met when docs consistently distinguish:

- implementation family: current code value such as `ordinary_pyspark`;
- semantic family: future reporting vocabulary such as `pyspark_dataframe`, `sql_relation`, and
  `local_lazy_dataframe`.

## Idempotence and Recovery

All edits are additive or clarifying. Re-running tests is safe. If a new config key causes unexpected resolver
behavior, first inspect `ResolveStructureConfig._keys` and `StructureConfigValidator`; unknown-key errors usually mean
the key was added to docs or defaults but not to the resolver key set. If hook tests fail with an unexpected keyword
error, inspect the public `before(...)` and `after(...)` signatures first, then `_hook(...)`, then
`CompilerHookCollector`.

Do not use `git reset --hard` to recover. If a generated artifact or formatter changes an unrelated file, inspect the
diff and revert only the unrelated file with an explicit, non-destructive checkout after confirming it was not part of
this plan.

## Interfaces and Dependencies

No new third-party dependency is required. The reserved config keys are ordinary TOML values. The CLI remains Click.
Hook target metadata is plain Python metadata attached by decorators and carried through existing dataclasses.

The final public hook decorator signature must include:

    target_backend: str | Iterable[str] | None = None

The final `StructureConfig` must include:

    target_profile: str | None
    compat_targets: tuple[str, ...]
    hook_target_default: tuple[str, ...] | str

The final CLI must accept:

    --target-profile RANGE
    --compat-targets TARGETS

where `TARGETS` is a comma-separated list such as `polars,duckdb`.
