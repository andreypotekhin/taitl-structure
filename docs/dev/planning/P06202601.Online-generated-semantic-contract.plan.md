# Implement the Online and Generated Execution Semantic Contract

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement the shared
execution semantic contract without prior conversation history.

## Purpose / Big Picture

Structure has two v1 execution forms. Online execution runs transforms directly through `StructureSession`. Generated
execution imports PySpark source emitted by `structure compile`. Users must be able to trust that both forms produce
the same DataFrame behavior for the same source transform.

After this work, online execution and generated code share one PySpark semantic target plan. A contributor can prove the
contract by running a parity test that executes the same model transform online and through the generated class, then
compares column order, schema shape where reliable, and row contents.

## Context and Orientation

Structure source transforms compile to backend-neutral intermediate representation, abbreviated IR. The key IR object
is `TransformPlan`. It records transform inputs, source-ordered steps, filters, joins, projections, hooks, validation
points, expression trees, and provenance anchors without live Spark objects.

Two v1 consumers use that IR:

- `OnlinePySparkRunner`, which runs a transform at runtime from source metadata and live input DataFrames;
- `PySparkCodeGenerator`, which renders deterministic PySpark source under `generated/structure_generated`.

The risk is that these consumers could become two separate lowerers. If one path casts a literal differently, places a
validation call earlier, chooses another join alias, or projects fields in another order, users lose trust in both
modes.

The shared contract is:

    TransformPlan + PySparkCapabilities -> PySparkExecutionPlan

`PySparkExecutionPlan` is an internal target plan made of immutable recipes. It contains semantic decisions for PySpark
execution, but it contains no generated source text and no live PySpark objects.

## Plan of Work

First, add internal recipe records in the future PySpark target layer. Use the code structure in `docs/dev/Code.md`:
business logic belongs in app logic packages, and general helpers belong in libraries. The exact package may shift as
the executable package skeleton matures, but the responsibility split must stay stable.

Define immutable records equivalent to:

    PySparkExecutionPlan
    PySparkStepRecipe
    PySparkExpressionRecipe
    PySparkJoinRecipe
    PySparkValidationRecipe
    PySparkHookRecipe
    PySparkCapabilities

For the first slice, implement only the subset needed for projection-only transforms:

    input validation recipes
    projection assignment recipes
    output validation recipes
    final return recipe

Then add `LowerPySparkExecutionPlan`, an action that accepts a checked `TransformPlan` and `PySparkCapabilities`.
It should produce deterministic recipes and should fail if an operation lacks a recipe.

Wire `OnlinePySparkRunner` to interpret projection recipes. It should bind declared input DataFrames, apply input
validation recipes, apply projection recipes with live PySpark Column operations, apply output validation recipes, and
return the resulting DataFrame.

Wire `PySparkCodeGenerator` to render the same projection recipes. It should keep owning imports, formatting, file
headers, comments, generated paths, and write-if-changed behavior. It must not choose projection order, aliases,
literal casts, or validation placement outside the shared recipes.

Add the first parity fixture. Use a small projection-only transform with deterministic rows. Run it online through
`StructureSession`, compile generated code, import the generated class, run it with the same input DataFrame, and assert
that output columns and rows match.

After the projection slice works, expand one operation family at a time:

- filters and multiple filters;
- expression helpers and literals;
- schema base overlays;
- joins and repeated join aliases;
- hooks and `pass_inputs=True`;
- validation modes, `schema_mode`, and `project_output`.

Each operation family must add recipe records, online interpretation, generated rendering, guardrail checks, and at
least one parity test before it is considered supported.

    capabilities

    output field name
    expression recipe
    output type
    output nullability

## Validation and Acceptance

The documentation phase is accepted when:

- `docs/specifications/ExecutionSemanticContract.md` exists;
- `docs/dev/design/ExecutionSemanticContract.md` exists;
- `docs/dev/design/decisions/D06202601.Online-generated-semantic-contract.md` exists;
- C24 in `docs/dev/design/Challenges.md` points to those documents;
- online, generator, IR, testing, roadmap, implementation, and sprint docs reference the shared contract.

The implementation phase is accepted when tests prove:

- a projection-only transform lowers to a shared PySpark target plan;
- online execution consumes that target plan;
- generated emission renders that target plan;
- online and generated projection outputs have the same columns and row values;
- validation placement comes from recipes, not separate runner logic;
- unsupported recipe kinds fail before execution or rendering;
- compiled recipes contain no prohibited UDF, RDD, collection, or row-wise behavior;
- compiler commands remain Spark-free.

Expected parity test shape:

    online_result = NormalizeOrders(orders=orders_df).run(session)
    generated_result = NormalizeOrdersGenerated(spark=spark).run(orders=orders_df)

    assert online_result.columns == generated_result.columns
    assert online_result.collect() == generated_result.collect()

The equality helper may sort rows only when the transform semantics do not promise row order.

## Idempotence and Recovery

The documentation updates are additive and safe to repeat. If wording conflicts appear, use `rg` to find old language
that says online and generated execution merely "should" agree, then update it to reference the shared semantic
contract.

Implementation should be additive. Keep existing generated-code behavior working while introducing shared recipes.
If online and generated behavior diverge, add a failing parity test that names the divergence before changing either
path. Fix the shared recipe or its consumers, not only one path.

If PySpark is unavailable locally, run recipe construction and no-Spark compiler tests, then record that runtime parity
tests were not run and why. Do not weaken the no-Spark compiler boundary to make runtime tests easier.

## Artifacts and Notes

Example conceptual target plan:

    PySparkExecutionPlan(
        transform="orders.transforms.order.NormalizeOrders",
        capabilities=pyspark_35_to_40,
        steps=[
            PySparkStepRecipe(
                step="normalize",
                operations=[
                    ProjectRecipe(
                        assignments=[
                            ("id", field("orders", "id")),
                            ("customer_id", field("orders", "customer_id")),
                            ("total", cast(field("orders", "total"), "decimal(12,2)")),
                        ]
                    )
                ],
                validations=[
                    PySparkValidationRecipe(
                        target="current",
                        schema="OrderNormalized",
                        mode="strict",
                        reason="final",
                    )
                ],
            )
        ],
    )

Generated source may render that plan as `.select(...)`. Online execution may interpret it as live Column objects. The
recipe is the shared semantic authority.
