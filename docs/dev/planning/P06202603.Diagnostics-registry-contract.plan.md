This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement the diagnostics
registry without prior conversation history.

## Purpose / Big Picture

Structure should fail in a way developers can act on immediately. After this work, every important diagnostic comes
from a registry with a stable code, severity, title, suggested fix, and documentation link. A user can run
`structure check` on broken source, see a code such as `DSL-E0401`, follow the docs link, and fix the problem.

The feature is working when tests prove that representative configuration, compiler, generated-output, CLI, and
runtime failures emit registered diagnostics, and that every emitted public code has a stable documentation anchor.

## Decision Log

- Decision: Use component-scoped prefixes such as `CONF`, `DSL`, `GEN`, `STREAM`, `ONLINE`, and `CLI`.
  Rationale: Component prefixes make the source of a diagnostic obvious while keeping stable machine-readable codes.
  Date/Author: 2026-06-20 / Codex

- Decision: Allocate numbers within each component prefix.
  Rationale: Local numbering keeps each component free to grow without coordinating global numeric ranges.
  Date/Author: 2026-06-20 / Codex

- Decision: Keep `docs/Diagnostics.md` as the stable public index.
  Rationale: Public tools need one durable link target even when deeper specs are reorganized.
  Date/Author: 2026-06-20 / Codex

- Decision: Test high-signal structured fields rather than full rendered terminal blocks.
  Rationale: Renderers should be free to improve layout without breaking semantic diagnostic tests.
  Date/Author: 2026-06-20 / Codex

## Context and Orientation

Structure is a Python library for schema-enforced Spark transformations. It has several phases that can produce
expected user-facing errors:

- configuration loading and validation;
- source-root discovery and import-safe source inspection;
- schema declaration and schema model validation;
- symbolic execution of transform methods;
- intermediate representation, abbreviated IR, validation;
- join and hook validation;
- backend capability checks for PySpark;
- generated-code freshness checks;
- runtime transform invocation and input binding;
- CLI command behavior.

A diagnostic is a structured error, warning, info message, or unexpected internal failure. It is not just terminal
text. A renderer turns it into terminal output, an exception message, a CI annotation, or a future IDE marker.

## Plan of Work

First, inspect the current executable package skeleton:

    git status --short
    rg -n "Diagnostic|CompileError|Configuration|RenderDiagnostic|click|StructureSession" src tests docs

Choose an internal package location that matches the existing structure. If the package skeleton follows
`src/structure/app` and `src/structure/lib`, prefer an internal diagnostics package such as:

    src/structure/lib/diagnostics/

Do not export registry internals from `structure.__init__`.

Add immutable or effectively immutable diagnostic records:

    Diagnostic
    DiagnosticCode
    DiagnosticSeverity
    DiagnosticEntry
    DiagnosticRegistry
    DiagnosticContext

The implementation may use dataclasses, enums, named tuples, or project-equivalent value classes. The important
contract is that tests can inspect code, severity, title, problem, suggested fix, docs link, and context fields without
parsing rendered terminal text.

Add registry validation. It must reject duplicate codes, malformed codes, unknown severities, unknown statuses, unknown
or wrong component prefixes, missing docs links for published codes, and deprecated codes without replacements.

Seed the registry with a small vertical slice:

    CONF-E0101 unknown configuration key
    CONF-E0102 invalid configuration value
    DSL-E0401 unsupported symbolic expression
    GEN-E0901 generated output is stale
    ONLINE-E1001 unknown transform input
    CLI-X1101 unexpected internal failure

The exact numbers may change during implementation if adjacent codes already exist, but they must use the component
prefixes defined by `docs/specifications/Diagnostics.md`.

Wire one representative source from each main surface:

- configuration validation uses `CONF-E0101` and `CONF-E0102`;
- unsupported expression checking uses `DSL-E0401`;
- `compile --fail-on-diff` uses `GEN-E0901`;
- transform invocation or runtime input binding uses `ONLINE-E1001`;
- CLI unexpected exceptions use `CLI-X1101`.

Add `RenderDiagnostic` or update the existing renderer. The renderer should accept a `Diagnostic` value and produce
human-readable text. It must preserve the code, severity, title, problem, suggested fix, and docs link.

Update `docs/Diagnostics.md` with public anchors for the initial active codes. Each anchor should explain common
causes and the expected fix in concise user-facing language.

## Validation and Acceptance

The implementation is accepted when tests prove:

- registry validation rejects duplicate codes;
- registry validation rejects malformed codes;
- registry validation rejects unknown component prefixes;
- registry validation rejects missing docs links for active, deprecated, and retired codes;
- registry validation rejects deprecated codes without `replaced_by`;
- configuration unknown-key and invalid-value errors emit registered codes and docs links;
- unsupported symbolic expression errors emit a registered code and direct DSL fix;
- stale generated output emits a registered generated-output code;
- unknown runtime transform input emits a registered runtime code;
- unexpected CLI failures render as an internal diagnostic without a user-facing stack trace by default;
- CLI rendering preserves code, severity, title, problem, suggested fix, and docs link;
- `docs/Diagnostics.md` has anchors for every emitted active code in the initial slice;
- `structure check` and registry tests run without PySpark, Java, Spark startup, a `SparkSession`, or a Spark cluster.

Expected diagnostic shape:

    CompileError DSL-E0401: Unsupported symbolic expression

    Source:
      src/orders/transforms/order.py:42

    Problem:
      Python string methods cannot be compiled to Spark Column expressions.

    Use:
      customer_id=lower(trim(order.customer_id))

    See docs/Diagnostics.md#dsl-e0401

## Idempotence and Recovery

The registry is additive. Adding a new diagnostic should add one registry entry, one docs anchor, and one focused test.
Running registry validation multiple times must not modify source files.

If an implementation discovers that an initial code number is already used, choose the next free number in the same
range and update this plan, the decision log, docs anchors, and tests in the same change.

If docs-anchor validation fails, prefer adding or restoring the anchor over changing emitted diagnostics. Published
links are part of the contract.

If a diagnostic is emitted from two phases for subtly different meanings, split it into two codes. Shared messages are
fine; shared codes require shared meaning.

## Interfaces and Dependencies

        def get(self, code): ...

    class Diagnostic:
        code
        severity
        title
        problem
        use
        docs
        context
        source

    class RenderDiagnostic:
