# Ordered Subtransform Operations and Bare Joins

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [Plans.md](../auto/Plans.md). It is self-contained so a contributor can implement ordered
subtransform operation capture without relying on prior conversation history.

## Purpose / Big Picture

After this change, Structure preserves the order in which a user writes compiled relational operations inside a
subtransform. A `where(...)` written before `join_one(...)` runs before the join, and a `where(...)` written after
`join_one(...)` runs after the join. Users can also call `join_one(product, on=...)` without assigning its return value;
later `product.name` reads are treated as reads from the joined scope, matching the compact style already available for
`where(...)`.

The observable result is a transform method such as:

    def add_product(self, order: Order, product: Product) -> Enriched:
        where(order.status.is_not_null())
        join_one(product, on=product.id == order.product_id)
        where(product.name.is_not_null())
        return Enriched(id=order.id, product_name=product.name)

Generated PySpark and online execution must apply the first filter, then the join, then the second filter, then the
projection.

## Surprises & Discoveries

- Observation: Current docs already describe filter/join source-order semantics, but implementation previously stored
  filters and joins separately and rendered all joins before all filters.
  Evidence: `RenderPySparkStep` and `RunOnlinePySparkTransform` iterated `step.joins` before `step.filters`.

- Observation: Bare join assignment can be implemented without Python local-variable rebinding.
  Evidence: relation parameters are mutable symbolic `InputScope` objects during compilation; `join_one(...)` can update
  that proxy so later field access delegates to a nullable joined `RowScope`.

## Decision Log

- Decision: Store ordered operation records alongside compatibility `filters` and `joins` fields.
  Rationale: This preserves existing callers and tests while making generated and online execution consume the ordered

  Date/Author: 2026-06-29 / Codex.

- Decision: Make `project(TargetSchema)` use the active driving row only inside compiled symbolic execution.

## Context and Orientation

Structure is a Python-to-PySpark compiler. Users write typed transform methods with symbolic schema rows. The compiler
executes those methods with symbolic objects, captures filters and joins, builds projection assignments from the
returned schema value, then lowers the plan to online PySpark execution or generated PySpark source.

The old capture model stored filters on `CompileContext.filters` and joins on `CompileContext.joins`. That was enough
for simple steps but lost relative order between filters and joins. The new model adds an ordered operation stream while
keeping `filters` and `joins` as compatibility views.

Key files:

- `src/structure/app/compiler/symbolic_execution/model/CompileContext.py` owns active symbolic capture state.
- `src/structure/app/dsl/model/transforms/transform_api.py` captures `where(...)` and `project(...)`.
- `src/structure/app/dsl/model/expr/InputScope.py` captures `join_one(...)` and relation field access.
- `src/structure/app/compiler/frontend/commands/CompileTransform.py` validates and builds `StepPlan`.
- `src/structure/app/target/pyspark/commands/RenderPySparkStep.py` renders generated PySpark step bodies.
- `src/structure/app/runtime/execution/online/commands/RunOnlinePySparkTransform.py` executes lowered recipes online.

## Plan of Work

Add immutable operation records for filter and join operations. Capture each `where(...)` and `join_one(...)` call into
that stream at call time. Keep `filters` and `joins` populated so existing traceability, explain, compatibility checks,
and tests can migrate gradually.

Change `join_one(relation, on=...)` so it records the join and mutates the relation proxy to read from the joined scope
for the rest of the subtransform. This makes `join_one(product, on=...)` equivalent to `product = join_one(product,
on=...)` for later symbolic field reads. Validate relation reads by walking the ordered operation stream, so filters
before a join cannot read the relation, while filters and projections after the join can.

Lower ordered operations to PySpark recipe operations. Generated rendering and online execution must use the operation
stream when present. For backward compatibility, recipes without operations should keep the previous behavior of joins,
then filters. Adjacent filters may be rendered as one `.where(...)` joined with `&` when no join separates them.

Implement source-less `project(TargetSchema)` only inside an active compiled subtransform. The active context supplies
the driving row as the source. Existing `project(source, target)` remains the explicit form and the recommended fix when
implicit projection is not available.

## Validation and Acceptance

The feature is complete when all of these behaviors are observable:

- `where(...)` before `join_one(...)` renders and runs before the join.
- `where(...)` after `join_one(...)` renders and runs after the join.
- bare `join_one(product, on=...)` allows later `product.name` reads.
- left bare joins make later right-side fields nullable.
- relation reads before their join still fail with `JOIN-E0601`.
- `project(TargetSchema)` works inside a compiled subtransform when the driving row is available.
- explicit `product = join_one(product, on=...)` remains valid.
- existing `filters` and `joins` consumers continue to work during migration.
- generated PySpark and online execution preserve the same order.
- `make build` passes.

## Idempotence and Recovery

The change is additive. If a downstream consumer does not understand ordered operations yet, it can continue reading
`filters` and `joins`. If ordered rendering breaks an existing generated-code assertion, inspect whether the source
method actually relied on filter-before-join order. Update the renderer or test expectation only after confirming the
source order.

If bare joins create surprising relation nullability, inspect `InputScope.__getattr__`. Before a join it must return
the original relation scope. After a left join it must delegate to a nullable joined `RowScope`.

## Artifacts and Notes

Expected operation order for:

    where(order.status.is_not_null())
    join_one(product, on=product.id == order.product_id)
    where(product.name.is_not_null())

is:

    filter(order.status.is_not_null)
    join(product)
    filter(product.name.is_not_null)

Generated PySpark should follow that order, with projection after the relational prefix.
