# Freestyle Transforms

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `docs/dev/auto/Plans.md`. It adds freestyle transforms: an explicit Structure transform mode for users
who want ordinary PySpark code behind Structure's input binding, schema validation, and session ergonomics without
using symbolic compilation or generated PySpark for the transform body.

## Purpose / Big Picture

Structure's compiled DSL is intentionally strict. It gives users Spark-plan-visible expressions, generated-code
reviewability, compiler traceability, and Spark-free checks. That strictness is valuable, but it can slow adoption when
users need PySpark features the compiler does not yet understand or when a team is migrating existing PySpark code.

After this change, a developer can write:

    @transform(type=TransformType.FREESTYLE, output=OrderEnriched)
    class EnrichOrders(Transform):
        orders = input(OrderRaw)
        customers = input(Customer)
        products = input(Product)

        def freestyle(self, *, orders, customers, products, spark, ctx):
            from pyspark.sql import functions as F

            return (
                orders
                .join(customers, customers.id == orders.customer_id, "left")
                .join(products, products.id == orders.product_id, "left")
                .where(F.col("products.id").isNotNull())
                .select(
                    orders.id.alias("id"),
                    customers.name.alias("customer_name"),
                    products.name.alias("product_name"),
                )
            )

The transform still uses `StructureSession`:

    result = EnrichOrders(
        orders=orders_df,
        customers=customers_df,
        products=products_df,
    ).run(session)

Structure validates declared inputs before the freestyle method runs and validates the declared output schema after it
returns. The body itself is opaque ordinary PySpark. Freestyle transforms do not generate transform classes and do not
provide column-level compiler traceability.

## Surprises & Discoveries

- Observation: The current `StructureSession.run(...)` always compiles a transform to IR and lowers it to PySpark
  recipes before checking the session's `execution_mode`.
  Evidence: `src/structure/app/runtime/logic/model/StructureSession.py` calls
  `lower_pyspark_plan(compile_transform(type(invocation)))` at the start of `run(...)`.

- Observation: The current configuration setting named `execution_mode` already means the execution target for compiled
  transforms: `online` or `generated`.
  Evidence: `ResolveStructureConfig._enums["execution_mode"]` permits only `("online", "generated")`, and
  `docs/Configuration.md` describes `online` as IR interpretation and `generated` as delegation to generated classes.

- Observation: Freestyle transforms should not weaken the no-Spark compiler boundary for ordinary `structure check`.
  Evidence: `docs/specifications/OnlineExecution.md`, `docs/specifications/PySparkCodeGeneration.md`, and
  `docs/specifications/ExecutionSemanticContract.md` all require compiler commands to remain Spark-free.

## Decision Log

- Decision: Add freestyle transforms as a per-transform mode selected by `@transform(type=TransformType.FREESTYLE, output=...)`.
  Rationale: This keeps strict compiled transforms as the default while giving users a deliberate escape hatch for
  ordinary PySpark. A per-transform mode avoids making one transform mean radically different things under different
  project configuration.
  Date/Author: 2026-06-22 / Codex.

- Decision: Keep project/session `execution_mode = "online" | "generated"` unchanged for compiled transforms.
  Rationale: That setting already has a clear meaning: whether checked IR is interpreted online or delegated to
  generated PySpark. Reusing it for opaque freestyle transforms would blur two different axes.
  Date/Author: 2026-06-22 / Codex.

- Decision: Use a required `output=SchemaClass` decorator option for freestyle transforms.
  Rationale: A freestyle method returns a PySpark DataFrame, so a method return annotation such as `-> OrderEnriched`
  would be misleading. The decorator output option gives Structure a declared final schema for validation and caller
  schema access.
  Date/Author: 2026-06-22 / Codex.

- Decision: Require the freestyle method name `freestyle`.
  Rationale: `run` is already reserved for `Transform.run(session)`. A fixed method name keeps discovery simple and
  makes diagnostics direct: freestyle transforms must define exactly one `freestyle(...)` method.
  Date/Author: 2026-06-22 / Codex.

- Decision: The freestyle method receives keyword-only DataFrame inputs matching declared `input(...)` names plus
  keyword-only `spark` and `ctx`.
  Rationale: Direct named inputs are easier to read and type in ordinary PySpark code than an `inputs` namespace, while
  keyword-only parameters let Structure validate signatures precisely.
  Date/Author: 2026-06-22 / Codex.

- Decision: Freestyle transforms are online-only and incompatible with generated transform execution.
  Rationale: There is no symbolic IR for the body, so generating an equivalent class would require source capture or
  opaque function shipping. If a session is configured with `execution_mode="generated"` and receives a freestyle
  transform, it must fail with a diagnostic that suggests online execution or a compiled transform.
  Date/Author: 2026-06-22 / Codex.

- Decision: `structure compile` should skip freestyle transform class generation but keep schema generation and report
  freestyle transforms as skipped opaque artifacts.
  Rationale: Mixed projects should still be able to generate artifacts for compiled transforms. Skipping with an
  explicit summary is more useful than failing the whole compile because one transform deliberately has no generated
  body.
  Date/Author: 2026-06-22 / Codex.

## Outcomes & Retrospective

This plan is ready for implementation. The intended result is a controlled pressure valve: users can opt into ordinary
PySpark transform bodies while retaining Structure's input/output schema boundary checks and invocation ergonomics.
The feature deliberately does not promise generated code, compiler-visible expressions, column-level traceability, or
streaming compatibility proof for freestyle transform bodies.

## Context and Orientation

Structure currently has one main transform kind: a compiled transform. A compiled transform is a `@transform` class with
public schema-returning subtransform methods. The compiler symbolically executes those methods with row proxies, builds
backend-neutral `TransformPlan` IR, lowers that IR to PySpark execution recipes, and then either interprets those
recipes online or renders generated PySpark source.

The current session entrypoint is `StructureSession.run(...)`. It compiles every transform class through
`compile_transform(...)` and `lower_pyspark_plan(...)` before dispatching to online or generated execution. Freestyle
transforms need a new branch before symbolic compilation.

The current `@transform` decorator accepts only `validate_intermediate` and `streaming_compatible`. Freestyle transforms
need additional decorator metadata: `type=TransformType.FREESTYLE` and `output=OutputSchema`. The default mode remains
compiled, so existing transform classes keep their behavior.

A freestyle transform is opaque to the compiler. Its body is ordinary Python and PySpark. Structure does not inspect its
DataFrame operations, does not generate equivalent PySpark code, and does not infer column dependencies from it.
Structure still owns metadata that can be known without executing the body: declared inputs, declared output schema,
source class identity, freestyle method signature, and input/output validation policy.

## Plan of Work

First, add the public enum `TransformType` to the DSL exports. It should contain at least `TransformType.COMPILED` and
`TransformType.FREESTYLE`. The default transform decorator behavior is equivalent to `type=TransformType.COMPILED`.
Document clearly that this enum is the transform authoring mode. It is distinct from the existing project/session
configuration string `execution_mode`, which selects `online` versus `generated` execution for compiled transforms.

Extend `@transform` to accept `type` and `output`. For compiled transforms, `output` is invalid because the output
schema comes from the final subtransform or Flow Method step. For freestyle transforms, `output` is required and must be
a `Structure` subclass. Unknown options still fail. Invalid combinations must produce structured diagnostics where
possible and clear import-time `TypeError` messages where decorator validation still happens at import time.

Add freestyle transform metadata inspection. A freestyle transform must declare exactly one method named `freestyle`.
It must have only keyword-only parameters after `self`. It must include one parameter for every declared input name,
plus `spark` and `ctx`. Extra parameters are invalid in the first release. Missing parameters are invalid. The method
body is not executed by `structure check`.

Add a freestyle transform runner. `StructureSession.run(...)` should detect `type=TransformType.FREESTYLE` before calling
`compile_transform(...)`. It should validate that all declared inputs are bound, validate input DataFrame schemas using
the same schema validation logic used by online execution, call `invocation.freestyle(...)` with the bound inputs,
`spark=session.spark`, and `ctx=session.ctx`, then validate the returned DataFrame against the declared output schema.
The invocation's `schemas.output` should be materialized from the declared output schema just as online compiled
execution exposes final schemas.

Keep generated mode honest. If a freestyle transform is run with `StructureSession(execution_mode="generated")`, fail
with a diagnostic such as `FREESTYLE-E1301 Freestyle transform cannot use generated execution`. The diagnostic should tell
the user to use `execution_mode="online"` or rewrite the transform as a compiled transform if generated artifacts are
required.

Update CLI behavior. `structure check` should validate freestyle transform metadata and method signatures without
executing the freestyle method and without importing PySpark beyond normal user module import. `structure compile` should
generate schemas and compiled transform artifacts as usual, skip freestyle transform classes, and include a summary line
that names skipped freestyle transforms. `structure compile --fail-on-diff` must not expect generated transform files for
freestyle transforms. `structure explain` should show freestyle transforms as opaque with declared inputs and output.

Update traceability and diagnostics. Compiler traceability for freestyle transforms should record the transform,
declared inputs, declared output, and an opaque freestyle body boundary. It must not claim column-level dependencies.
Streaming compatibility for freestyle transforms is unknown unless a later plan adds an explicit trusted option such as
`streaming_safe=True`.

Update docs. Add freestyle transform sections to the public usage/configuration docs and specification docs. The docs
must say that freestyle transforms are useful for migration, prototyping, and unsupported PySpark operations, but that
compiled transforms remain the default for generated code, explainability, and static checking. Examples should import
PySpark inside the `freestyle(...)` method to preserve Spark-free compiler commands when PySpark is not installed.

## Validation and Acceptance

The feature is complete when tests prove:

- `TransformType` is importable from `structure`.
- Existing `@transform` classes still default to compiled mode.
- `@transform(type=TransformType.FREESTYLE, output=Schema)` records freestyle transform metadata.
- Freestyle transforms require one `freestyle(...)` method.
- Freestyle method signatures must contain keyword-only declared input names, `spark`, and `ctx`, with no extras.
- Freestyle transforms require `output=StructureSubclass`.
- Compiled transforms reject `output=...` unless a future plan explicitly allows it.
- `StructureSession(execution_mode="online")` can run a freestyle transform against Spark DataFrames.
- Freestyle execution validates declared inputs before calling the freestyle method.
- Freestyle execution validates the declared output after the freestyle method returns.
- Freestyle execution exposes `transform.schemas.output`.
- Freestyle transforms fail clearly under `execution_mode="generated"`.
- `structure check` validates freestyle metadata without executing freestyle methods.
- `structure compile` skips freestyle transform class generation and still handles schemas and compiled transforms.
- `structure explain` shows freestyle transforms as opaque input-to-output boundaries.
- Traceability for freestyle transforms is coarse and does not claim column-level dependencies.
- Compiler commands remain Spark-free when freestyle transform modules avoid top-level PySpark imports.
- Existing online/generated parity tests for compiled transforms continue to pass.

## Idempotence and Recovery

This feature is additive. Existing compiled transforms should not change behavior. If a freestyle implementation breaks
ordinary compiled transforms, restore `StructureSession.run(...)` to the old compile-first path for non-freestyle
transforms before continuing.

Do not remove generated-code support. Freestyle transforms are an escape hatch, not a replacement for compiled
transforms. If compile behavior for mixed projects becomes unclear, prefer the conservative behavior: generate compiled
artifacts, skip freestyle transform classes, and report the skip explicitly.

Freestyle transform methods may contain arbitrary user PySpark and side effects. Structure must not call them during
`check`, `compile`, `explain`, source discovery, or traceability generation. They run only through
`StructureSession.run(...)`.
