## Purpose / Big Picture

Structure is approaching v2 work. V2 is meant to add mainstream analytical pipeline features such as analytical joins,
aggregations, windows, higher-order functions, optimization directives, richer explain output, generated docs, test
tooling, and incremental compile. Before coding starts, the project needs one explicit readiness review that reconciles
the current v1 state, recent behavior changes, v2 planning documents, v2 fixtures, and the newly advertised
alternative-backend boundary.

After this review, a contributor can read the final readiness report in this plan and know whether v2 coding may start,
which v1 gates still block it, which documents or fixtures must be reconciled first, and which follow-up ExecPlans must
be created before feature implementation. The observable result is not new runtime behavior. The observable result is a
written review section in this file with a clear v2 start recommendation and a list of concrete next plan documents.

This plan creates the review process. It does not implement v2 analytical features, alternative backends, or v1
alternative-backend consistency. The v1 bridge is owned by
`docs/dev/planning/P07012601.V1-alternative-backend-consistency.plan.md` and should be referenced rather than
duplicated.

## Context and Orientation

The repository root is `C:\Taitl\Code\taitl\taitl-structure`. Structure is a Python library that compiles a small
schema-first DSL into backend-neutral intermediate representation, called IR. IR is a plain Python model of a transform
before it is lowered to online PySpark execution or generated PySpark files. In v1, PySpark is the only executable
backend.

V2 is tracked as M7 in `docs/dev/project-management/Milestones.md`. M7 is split into:

- M7A: v2 scope and analytical IR foundations;
- M7B: analytical join coverage;
- M7C: aggregations, windows, and higher-order functions;
- M7D: optimization, explain, docs, test tooling, and incremental compile.

The v2 sprint sequence lives under `docs/dev/project-management/sprints`:

- `Sprint06_V2ScopeAndAnalyticalIR.md`;
- `Sprint07_AnalyticalJoinCoverage.md`;
- `Sprint08_AggregationsWindowsHigherOrderFunctions.md`;
- `Sprint09_OptimizationExplainDocsTooling.md`.

Recent v1 work matters for v2 because it changed the base compiler model. `project(...)` is now a public projection
primitive. Ordered operation capture records filters and joins in source order. Inferred `join_one(...)` lets the
compiler select the joined relation from an unambiguous `on` expression. Behavior-rigidity work adds public API
snapshots, golden generated-output checks, concept tests, metamorphic tests, and compatibility consistency checks.
Alternative-backend documentation adds a future target-adapter boundary that v2 planning must either incorporate or
explicitly defer.

### Recent V1 Foundation Effects

`project(...)`, ordered operation capture, inferred `join_one(...)`, hook target metadata, and the plain Python
expression extension make v2 easier to express but also raise the bar for consistency. V2 operations should reuse
ordered operation sequencing and the shared expression pipeline instead of adding parallel metadata paths. Every new
feature should declare cardinality and capability requirements at capture time. Behavior-rigidity work should be
treated as acceptance infrastructure for v2, especially public API snapshots, golden examples, compatibility matrix
checks, and generated-output snapshots.

### Alternative Backends

Alternative Backend docs should be treated as foundation/adoption scaffolding, not analytical feature implementation.
P07012601 now closes v1 reserved syntax and honest pending output. P07012603 should reserve compatibility-report
vocabulary only where it supports unsupported capability reporting; a later plan should own real compatibility reports
if they do not fit cleanly in the foundation gate. Actual Polars, DuckDB, Spark SQL, or other backend lowering remains
deferred.
