# Flow Method Transform Pipeline

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [Plans.md](../../auto/Plans.md). It adds an alternate transform authoring style called a Flow Method. A Flow
Method is one ordinary Python method that describes a linear transform pipeline and marks typed intermediate schema
boundaries with `step(...).produce(...)`. It must compile to the same backend-neutral `TransformPlan` and `StepPlan`
model already used by method-per-subtransform transforms.

## Purpose / Big Picture

The current transform style is explicit and dependable, but long enrichment chains can become visually busy because
each small schema boundary requires a separate method. After this change, a developer can write a compact linear
pipeline in one method while preserving Structure's strict schema boundaries, optimizer-visible expressions, hooks,
traceability, online execution, and generated PySpark behavior.

The user-visible behavior is demonstrated by compiling a transform like this:

    @transform
    class EnrichOrders(Transform):
        orders = input(OrderRaw)
        customers = input(Customer)
        products = input(Product)

        @pipeline
        def enrich(self, order: OrderRaw) -> OrderWithProduct:
            customer = self.customers.join_one(
                on=(self.customers.tenant.tenant_id == order.tenant.tenant_id)
                & (self.customers.id == order.customer_id),
                how=Join.LEFT,
            )

            order = step("add_customer").produce(
                OrderWithCustomer.base(order)(
                    customer_name=customer.name,
                    customer_tier=customer.tier,
                    customer_region=customer.region,
                )
            )

            product = self.products.join_one(
                on=(self.products.tenant.tenant_id == order.tenant.tenant_id)
                & (self.products.id == order.product_id),
                how=Join.LEFT,
            )

            where(product.id.is_not_null())

            return step("add_product").produce(
                OrderWithProduct.base(order)(
                    product_name=product.name,
                    product_category=product.category,
                )
            )

`compile_transform(EnrichOrders)` should produce two normal ordered steps named `add_customer` and `add_product`.
Online execution and generated execution should behave as if those steps were written as two public schema-returning
subtransform methods.

## Surprises & Discoveries

- Observation: The repository currently has no fluent source transform API. The published DSL is intentionally centered
  on public schema-returning subtransform methods.
  Evidence: Searches for `pipeline`, `pipe(`, `stage`, and fluent transform builders found only general prose and
  implementation internals, not a public alternate authoring surface.

- Observation: The existing method-per-subtransform model carries three important contracts: the step name, schema flow,
  and hook anchor.
  Evidence: [DSL.md](../../../specifications/DSL.md), [SymbolicExecution.md](../../../specifications/SymbolicExecution.md), and `CompileTransform.py` all
  collect public schema-returning methods in source order and attach hook metadata to concrete step names.

## Decision Log

- Decision: Add Flow Method as an alternate authoring style, not a replacement for public schema-returning methods.
  Rationale: Existing source, docs, tests, generated code, and user mental models already rely on method-per-step
  transforms. Flow Method should reduce ceremony for long linear pipelines while preserving backward compatibility.
  Date/Author: 2026-06-22 / Codex.

- Decision: Use `@pipeline` to mark exactly one Flow Method on a transform class.
  Rationale: An explicit marker avoids accidental compilation of ordinary public helper methods and lets diagnostics
  explain that a transform must use either ordinary subtransform methods or one Flow Method, not both.
  Date/Author: 2026-06-22 / Codex.

- Decision: Use `step("name").produce(schema_instance)` as the only Flow Method schema-boundary primitive.
  Rationale: `step("name")` makes the named boundary visible, while `.produce(...)` clearly marks the schema expression
  that closes the step. Requiring an explicit name preserves diagnostics, generated-code comments, traceability, and
  hook anchors without turning the whole transform into a builder chain.
  Date/Author: 2026-06-22 / Codex.

- Decision: Extend `before(...)` and `after(...)` to accept string step names for Flow Method steps.
  Rationale: Flow Method steps do not exist as Python methods, so method-object hook targets are unavailable. String
  targets keep the Flow Method syntax lean. The compiler must validate that every string target resolves to exactly one
  `step(...)` name in the same transform.
  Date/Author: 2026-06-22 / Codex.

- Decision: Preserve the existing backend-neutral IR shape.
  Rationale: Online execution, generated PySpark, traceability, streaming checks, and diagnostics already consume
  ordered `StepPlan` records. Flow Method should be a frontend syntax feature that produces the same IR rather than a
  second execution model.
  Date/Author: 2026-06-22 / Codex.

## Outcomes & Retrospective

This plan is ready for implementation. The work has not yet been coded. The intended outcome is a compact Flow Method
syntax that compiles to ordinary `StepPlan` records and remains compatible with existing generated and online execution

## Context and Orientation

Structure is an IR-first toolkit for schema-driven PySpark pipelines. Users declare `@transform` classes with named
`input(...)` declarations. The current canonical transform body is a sequence of public methods. Each public method
with one row parameter and a `Structure` return annotation is a compiled subtransform. The compiler symbolically
executes each method with row proxies, records calls to `where(...)` and `join_one(...)`, captures the returned schema
construction as the projection, and creates one `StepPlan`.

The main compiler entrypoint is `src/structure/app/dsl/logic/actions/CompileTransform.py`. The public DSL helpers live
in `src/structure/app/dsl/logic/model/transforms/transform_api.py` and are re-exported through `structure`. The IR
records are under `src/structure/app/dsl/logic/model/plans`. Existing online and generated PySpark behavior consumes
the resulting `TransformPlan`, so this feature must feed that same plan shape.

A Flow Method is a decorated transform method that is symbolically executed once. Inside it, every call to
`step("name").produce(schema_instance)` closes the current step, records the operations since the previous step
boundary, and returns a symbolic row proxy for the step output schema. Later code in the same Flow Method uses that
returned proxy as the current row. A bare `step("name")` is only a pending boundary and is invalid unless `.produce(...)`
is called.

For the first Flow Method step, the input schema is the Flow Method row parameter annotation. For later steps, the
input schema is the output schema of the previous `step(...).produce(...)`. The Flow Method return annotation must
match the output schema of the final produced step.

## Plan of Work

Add public DSL symbols `pipeline` and `step` and export them from `structure` and `structure.app.dsl.api`. The
`@pipeline` decorator should attach metadata to an instance method and return the original callable. `step(name)` should
return a pending step boundary object that exposes `.produce(row)`. Calling either `step(...)` or `.produce(...)` should
be valid only during symbolic execution of a Flow Method. Outside a Flow Method, they must fail with a structured
diagnostic or a clear runtime error matching the current style of `where(...)`.

Extend transform discovery in `CompileTransform.py`. If a transform declares one `@pipeline` method, compile it through
the new Flow Method path. If it declares ordinary public schema-returning subtransform methods, keep the current path.
If it declares both, fail with a DSL structure diagnostic telling the developer to choose one authoring style. More
than one `@pipeline` method must also fail, because v1 Flow Method syntax describes one linear pipeline.

Implement Flow Method symbolic execution by reusing the existing `CompileContext` concept. The implementation should
create an active flow context before invoking the decorated method. Calls to `where(...)` and `join_one(...)` continue
to append to the active context. Calls to `step(...)` validate the supplied name and create a pending step boundary.
Calls to `.produce(...)` validate the schema instance, convert the operations captured since the prior boundary into a
`StepPlan`, clear boundary-local operations, and return a new `RowScope` for the output schema. The final method result
must be the value returned by the final `.produce(...)`, and its schema must match the method return annotation.

Preserve source-order semantics. In a Flow Method, filters and joins before the first `.produce(...)` belong to the
first step. Filters and joins after `order = step(...).produce(...)` and before the next `.produce(...)` belong to the
next step. Hooks with string targets are attached to the matching step after all steps have been collected.

Add diagnostics for Flow Method misuse. Required failures include duplicate step names, non-string step names, empty
step names, `step(...)` or `.produce(...)` outside a Flow Method, a bare pending `step(...)` that never calls
`.produce(...)`, Flow Method with no produced steps, method return annotation not matching the last step output schema,
a Flow Method returning a bare schema construction instead of the final `.produce(...)` result, string hook target that
does not match a step, and mixed Flow Method plus ordinary subtransform methods.

Update docs and examples after tests prove the behavior. [DSL.md](../../../specifications/DSL.md),
[SymbolicExecution.md](../../../specifications/SymbolicExecution.md), [Concepts.md](../../Concepts.md), and one user-facing usage document should describe
Flow Method as an alternate layout. Do not rewrite the canonical examples wholesale. Present Flow Method as best suited
for long linear enrichment chains.

## Validation and Acceptance

Acceptance is behavioral. The feature is complete when a Flow Method transform compiles into the same ordered
`StepPlan` shape as the equivalent method-per-subtransform transform, and existing online/generated tests still pass.

The test suite must prove:

- `pipeline` and `step` are importable from `structure`.
- A transform can use one `@pipeline` method with multiple `step(...).produce(...)` boundaries.
- Each `.produce(...)` creates one ordered `StepPlan` with the expected input schema, output schema, filters, joins,
  projection, ordinal, and hooks.
- The row proxy returned by `.produce(...)` is usable by later joins, filters, and projections.
- `@after("step_name")` and `@before("step_name")` attach to Flow Method steps and preserve hook options.
- Existing public schema-returning subtransform methods continue to compile unchanged.
- Mixed Flow Method and ordinary subtransform declarations fail clearly.
- Duplicate or missing Flow Method step targets fail clearly.
- Online/generated parity tests continue to pass for existing method-per-subtransform fixtures.

The final command evidence should show `pytest` passing. If `make build` is run, record its success as well.

## Idempotence and Recovery

The implementation is additive. Existing transform classes should not need migration. Re-running tests is safe. If the
Flow Method compiler path starts affecting ordinary subtransform compilation, revert the discovery branching first so
ordinary transforms continue through the existing `_steps(...)` path, then reintroduce Flow Method handling behind the
explicit `@pipeline` marker.

Avoid generated-file churn until the frontend tests pass. If documentation examples are updated before implementation
is stable, keep them in a clearly marked Flow Method section so they can be revised without disrupting canonical v1
examples.

## Artifacts and Notes

Preferred source shape:

    @pipeline
    def enrich(self, order: OrderRaw) -> OrderWithProduct:
        customer = self.customers.join_one(...)
        order = step("add_customer").produce(OrderWithCustomer.base(order)(...))

        product = self.products.join_one(...)
        where(product.id.is_not_null())
        return step("add_product").produce(OrderWithProduct.base(order)(...))

Equivalent conceptual IR:

    TransformPlan EnrichOrders
      StepPlan add_customer
        input_schema: OrderRaw
        output_schema: OrderWithCustomer
        operations: join_one customers, project OrderWithCustomer
      StepPlan add_product
        input_schema: OrderWithCustomer
        output_schema: OrderWithProduct
        operations: join_one products, filter product.id is not null, project OrderWithProduct

## Interfaces and Dependencies

New public DSL:

- `pipeline(function) -> function`
- `step(name: str) -> PendingStep`
- `PendingStep.produce(row: Structure) -> RowScope`
- `before(target, ...)` and `after(target, ...)` continue accepting method targets and additionally accept string step
  names when the transform uses a Flow Method.

Do not add dependencies. Do not import PySpark in compiler phases. Do not change the runtime invocation API. Do not
create a new IR type for Flow Methods unless implementation proves it is necessary; the intended design is to lower
Flow Method source into ordinary `StepPlan` records.
