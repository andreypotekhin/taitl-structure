# Add the Backend Capability Interface

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can design, document, implement,
and validate the backend capability interface without needing prior conversation history.

## Purpose / Big Picture

Structure compiles user-authored schema transforms into backend-neutral IR. That IR is consumed by online PySpark
execution and generated PySpark code. Before the project adds more expressions, joins, streaming behavior, Spark
Connect, or alternate backends, it needs one static capability boundary that says what the configured backend can
lower.

After this work, a contributor can ask:

    capabilities = resolve_backend_capabilities(
        target_backend="pyspark",
        target_pyspark=">=3.5,<4.1",
    )

    capabilities.require(CapabilityRequirement(group="join", name="join_one"))

Supported v1 features succeed. Unsupported features such as `join_many` fail with the backend diagnostic path. This
works without importing PySpark.

## Decision Log

- Decision: Use `BACKEND-E2401` and `BACKEND-E2402` for C23 diagnostics.
  Rationale: The current diagnostics contract uses component prefixes such as `BACKEND`, not the older generic
  `STRUCT` prefix used in early examples.
  Date/Author: 2026-06-20 / Codex

- Decision: Keep backend capabilities internal for now.
  Rationale: The immediate need is compiler and emitter architecture, not a public reporting API. Public users benefit
  through diagnostics and future explain output.
  Date/Author: 2026-06-20 / Codex

- Decision: Make `supports(...)` non-throwing and `require(...)` diagnostic-producing.
  Rationale: Checks and reports often need to inspect support without failing immediately, while compiler validity
  paths need one strict failure mechanism.
  Date/Author: 2026-06-20 / Codex

- Decision: Model target behavior as static metadata.
  Rationale: `structure check` and `structure compile` must stay Spark-free, so capability checks cannot probe an
  installed PySpark runtime.
  Date/Author: 2026-06-20 / Codex

## Context and Orientation

Structure source transforms compile to IR, short for intermediate representation. IR describes transform meaning
without naming concrete PySpark calls. Online execution and generated PySpark both consume IR. The backend capability
interface is the static target knowledge used between checked IR and target lowering.

Relevant files:

- `docs/dev/design/Challenges.md` contains C23.
- `docs/specifications/BackendCapabilities.md` is the capability behavior contract.
- `docs/dev/design/BackendCapabilities.md` explains the design.
- `src/structure/app/target/capabilities/api/capabilities.py` exposes the internal entrypoint.
- `src/structure/app/target/capabilities/logic/model/capabilities.py` defines the data model.
- `src/structure/app/target/capabilities/logic/rules/PySparkCapabilityRules.py` defines the v1 PySpark profile.
- `tests/specifications/backend-capabilities/test_backend_capabilities.py` proves behavior.

Terms:

- Backend: the execution implementation family selected by configuration, currently ordinary PySpark.
- Target range: the configured version range, currently `target_pyspark = ">=3.5,<4.1"` by default.
- Capability requirement: a question such as "does this backend support `join.join_one`?"
- Capability decision: the answer, including diagnostics when unsupported.

## Plan of Work

Add the internal target package under `src/structure/app/target`. Keep the implementation pure Python and Spark-free.
Use frozen data classes for value objects so decisions are safe to pass between compiler phases.

Add `docs/specifications/BackendCapabilities.md` and `docs/dev/design/BackendCapabilities.md`. Update existing docs so
backend capability behavior has one owner.

Add focused tests under `tests/specifications/backend-capabilities`. The tests should insert `src` into `sys.path` if the
package is not installed in editable mode, then import the internal capability API. They should assert both supported
and unsupported decisions and prove PySpark is not imported.

## Validation and Acceptance

Acceptance is met when:

- resolving default PySpark capabilities succeeds;
- resolving an unknown backend raises a diagnostic with `BACKEND-E2401`;
- requiring unsupported `join.join_many` raises a diagnostic with `BACKEND-E2402`;
- requiring supported `join.join_one` succeeds;
- generated import metadata is deterministic;
- focused tests pass without PySpark installed or imported.

## Idempotence and Recovery

The capability implementation is additive. Re-running tests is safe. If future work renames capability requirements,
update the spec, tests, and PySpark profile together so unsupported features still fail through `BACKEND-E2402`.

## Artifacts and Notes

The initial supported capability set is intentionally small. Adding a new DSL operation requires adding a capability
name, documenting whether it is supported for the configured backend, and adding tests before lowering code depends on
it.

## Interfaces and Dependencies

Use only the Python standard library for this slice. Do not add a PySpark dependency. Do not add a version-specifier
dependency until the configuration layer needs richer range analysis. The initial profile accepts the documented v1
target ranges exactly.
