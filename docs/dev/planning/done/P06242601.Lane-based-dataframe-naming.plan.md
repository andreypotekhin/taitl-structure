# Lane-Based DataFrame Naming And Hook ABI

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `docs/dev/auto/Plans.md`.

## Purpose / Big Picture

Generated PySpark currently uses `df` as the current DataFrame variable and hook keyword. That name is a leftover from
the single-input, single-output era. After this change, generated code and hooks use meaningful lane names: the lane
starts as the selected `input(...)` field name, such as `orders`, and keeps that name until a subtransform explicitly
changes lanes with `@transform(output=...)`.

The visible result is generated code with `orders = ...` or `published = ...` instead of `df = ...`, plus hook
signatures that match explicit decorator selectors:

    @before(normalize, lane=orders)
    def prepare(self, *, orders, spark, ctx): ...

    @after(publish, lane=published)
    def polish(self, *, published, spark, ctx): ...

No backward compatibility shim for old `df` hooks is required because Structure has not been released.

## Surprises & Discoveries

- Observation: The current compiler already tracks `lane` and `frame` separately for step results.
  Evidence: `StepResultPlan` and `PySparkStepResultRecipe` both contain `lane` and `frame`, and multi-result rendering
  already uses output names such as `accepted` and `audited`.

- Observation: Current docs and tests still make `df` the hook ABI.
  Evidence: `docs/specifications/HookSemantics.md` requires `def hook(self, *, df, spark, ctx)`, and tests assert
  generated calls such as `self._impl.audit(df=audited, ...)`.

- Observation: Direct v2 fixture regeneration is blocked by an existing v2 package import issue.
  Evidence: Importing `testing.model.v2.orders.transforms.order` imports `analytics.py`, which imports aggregation
  helpers such as `count` from `structure`; those helpers are not exported in this workspace. The v2 hook-affected
  generated order fixture was updated directly to match the lane naming contract.

- Observation: After the initial lane change, an undecorated later method could still fall back to an original input
  when its parameter schema did not match the current lane.
  Evidence: The full test suite failed `test_v1_schema_flow_mismatch_reports_transform_structure_diagnostic`; the fix
  makes original-input restarts require explicit `@transform(input=...)` once a current lane exists.

## Decision Log

- Decision: Replace `df` with explicit lane-name hook parameters.
  Rationale: The hook author should not deduce an implicit name, and generated code should not hide lane identity
  behind a generic variable.
  Date/Author: 2026-06-24 / Codex and project owner.

- Decision: Require `@before(target, lane=...)` and `@after(target, lane=...)`.
  Rationale: Explicit selectors make hook signatures self-documenting and remove ambiguity for single-result and
  multi-result subtransforms alike.
  Date/Author: 2026-06-24 / Codex and project owner.

- Decision: A lane starts from the selected class-level input name and changes only through explicit
  `@transform(output=...)`.
  Rationale: This preserves the old "current lane continues through steps" behavior of `df`, but gives it a meaningful
  name.
  Date/Author: 2026-06-24 / Codex and project owner.

- Decision: Require at least one class-level `input(...)` declaration.
  Rationale: Lane naming depends on declared inputs, and early failure is clearer than inventing implicit source names.
  Date/Author: 2026-06-24 / Codex and project owner.

## Context and Orientation

The relevant implementation is split across the DSL, compiler IR, PySpark target, online runtime, docs, and testing
model.

The DSL hook decorators live in `src/structure/app/dsl/model/transforms/transform_api.py`. The compiler builds lanes
and hooks in `src/structure/app/compiler/frontend/commands/CompileTransform.py`. Generated PySpark step rendering lives
in `src/structure/app/target/pyspark/commands/RenderPySparkStep.py`. Online hook invocation lives in
`src/structure/app/runtime/execution/online/logic/PySparkHookInvoker.py`.

A "lane" is the named current DataFrame stream. Today the compiler often calls that lane `df`; after this work, the
default lane is the selected input declaration name, such as `orders`. A "lane change" is an explicit subtransform
output binding, such as `@transform(output=published)`, after which the current lane name is `published`.

## Plan of Work

First, update the public hook API. Change `before(...)` and `after(...)` to require `lane=`.
The selector should accept a declared lane source: `input(...)` declarations for input/current lanes and `output(...)`
declarations for explicit output lanes. Remove the old `df=` decorator argument. Store the selector name in hook
metadata as the hook lane name.

Next, update compilation. Add a compile-time check that a transform declares at least one input. Replace the implicit
`"df"` lane with the resolved driving input lane. For a normal single-lane pipeline, the first step sourced from
`orders = input(OrderRaw)` produces lane `orders`; later unbound steps continue to read and write lane `orders`. If a
method declares `@transform(output=published)`, it produces lane `published`; downstream explicit
`@transform(input=published)` reads it.

Validate hooks during compilation. A `before` hook must select the lane consumed by its target step. An `after` hook
must select a lane produced by its target step. The hook method must declare a keyword-only parameter with exactly that
selected lane name, plus `spark` and `ctx`; `inputs` is required only when `pass_inputs=True` and invalid otherwise.
Extra runtime parameters remain invalid.

Then update PySpark lowering and rendering. Generated local variables should use lane names, not `df`, for lane
intermediates and hook calls. Join helper variables must avoid collisions, for example by using `<input>_joined` or
another deterministic suffix. The renderer should still use stable aliases derived from schemas for Spark aliasing;
only Python local DataFrame names and hook keywords change.

Update online execution in parallel with generated execution. `PySparkHookInvoker.apply(...)` should receive the
selected lane name from the hook recipe and call hooks with `{lane_name: current_frame, "spark": ..., "ctx": ...}`.
Generated and online execution must invoke the same source hook with the same keyword arguments.

Finally, update documentation and examples. Replace `df` in hook docs, generated-code docs, README examples, usage
docs, whitepaper examples, and testing model transforms. Update `docs/specifications/UserStories.md` so completed hook
stories describe explicit lane selectors and matching hook signatures rather than `df`.

## Validation and Acceptance

Add or update focused tests for these scenarios:

- A transform with no `input(...)` declaration fails with a diagnostic that tells the developer to declare at least one
  input.
- `@before(step)` without `lane=` fails.
- `@after(step)` without `lane=` fails.
- A hook selector that does not match the target boundary fails.
- A hook signature missing the selected lane keyword fails and names the expected keyword.
- A same-lane pipeline generated from `orders = input(...)` uses `orders` as the DataFrame local across subtransforms.
- A lane-changing pipeline uses the explicit output name after `@transform(output=...)`.
- Online and generated execution pass identical hook keyword names.

Run focused tests first:

    pytest tests/specifications/multiple-schema-parameters tests/specifications/multiple-outputs tests/specifications/pyspark-code-generation

Then run the broad project test command used in this repository. If PySpark integration tests are unavailable in the
local environment, record that in this plan's `Outcomes & Retrospective` and run all Spark-free tests that are feasible.

Acceptance is met when no generated model transform hook call contains `df=`, public docs no longer teach `df` hook
signatures, and updated tests pass.

## Idempotence and Recovery

All changes are source edits and test updates. Re-running tests is safe. Regenerating model output is safe if the
generator writes deterministic files. If a formatting or generation command modifies unexpected tracked files, inspect
the diff and keep only changes tied to lane naming and hook ABI.

Do not preserve old `df` compatibility paths. If an old test depends on `df`, update the test to the new lane-name
behavior or replace it with a negative test proving `df` is rejected.
