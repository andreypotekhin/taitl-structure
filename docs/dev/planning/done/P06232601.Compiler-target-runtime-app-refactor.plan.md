## Purpose / Big Picture

Structure is organized as apps: focused parts of the system that own one business concern. The current source layout
keeps some compiler concerns under the DSL app, names PySpark target code as `backend`, keeps streaming compatibility
as its own narrow top-level app, and stores runtime execution helpers in broad `runtime.logic.actions`. This makes the
architecture harder to read because source authoring, compilation, target lowering/rendering, and runtime invocation
are different concerns.

After this refactor, a contributor can see the intended architecture directly in the package tree. DSL code owns public
authoring helpers. Compiler code owns turning DSL source into backend-neutral intermediate representation. Target code
owns PySpark-specific recipes and rendering. Runtime code owns sessions, schemas, and online/generated execution.

## Decision Log

- Decision: Use `structure.app.compiler.frontend` for `CompileTransform`.
  Rationale: `frontend` is the conventional name for the compiler phase that accepts source-level DSL classes and
  produces IR. The project accepts the temporary frontend/target naming imbalance because `backend` will become
  `target`.
  Date/Author: 2026-06-22 / Codex

- Decision: Keep the `CompileContext` class name while moving it under `structure.app.compiler.symbolic_execution`.
  Rationale: The name is already clear and preferred by the user. The package path supplies the additional context that
  it is symbolic-execution capture state.
  Date/Author: 2026-06-22 / Codex

- Decision: Rename `structure.app.backend` to `structure.app.target`.
  Rationale: The package holds PySpark target capabilities, lowering, recipes, rendering, and generated file handling.
  `target` avoids implying a balanced compiler frontend/backend pair and matches the architecture wording about target
  boundaries.
  Date/Author: 2026-06-22 / Codex

- Decision: Move current streaming compatibility into compiler compileability.
  Rationale: The current checker classifies an already-lowered plan during compile/check/explain flows. Future
  top-level streaming orchestration is larger and should not be conflated with this narrow compileability pass.
  Date/Author: 2026-06-22 / Codex

- Decision: Split runtime into `execution/online`, `execution/generated`, `session`, and `schemas`.
  Rationale: Online execution, generated execution, session orchestration, and schema materialization are distinct
  runtime responsibilities and should be independently discoverable.
  Date/Author: 2026-06-22 / Codex

## Context and Orientation

The DSL app is the user authoring surface. Its public API lives at `src/structure/app/dsl/api/__init__.py` and exports
schema declarations, transform decorators, expression helpers, and `compile_transform`. Keeping the public re-export is
important because tests and user code import `compile_transform` from `structure.app.dsl.api`.

The compiler app currently contains diagnostics and traceability only. This refactor adds compiler-owned subapps:
`frontend` for source-to-IR orchestration, `ir` for backend-neutral plan model classes, `symbolic_execution` for the
active compile context used by `where(...)` and `join_one(...)`, and `compileability/streaming_compatibility` for the
current streaming compatibility checker.

The target app replaces the old backend app. Target code is specific to concrete execution targets. In this change,
`capabilities` and `pyspark` move from `src/structure/app/backend` to `src/structure/app/target` and all internal
imports update accordingly.

The runtime app owns user sessions and runtime execution. `StructureSession` remains the programmatic entry point, but
its implementation moves under `runtime/session`. Online and generated runners move under `runtime/execution/online`
and `runtime/execution/generated`. Schema materialization helper code moves under `runtime/schemas`.

## Plan of Work

First, add the new compiler packages and move existing files into them. Move `CompileTransform.py` into
`src/structure/app/compiler/frontend/logic/actions/`. Move all files currently in
`src/structure/app/dsl/logic/model/plans/` into `src/structure/app/compiler/ir/logic/model/`. Move
`CompileContext.py` into `src/structure/app/compiler/symbolic_execution/logic/model/`. Move the current streaming
compatibility API, action, and model files into `src/structure/app/compiler/compileability/streaming_compatibility/`.
Update import paths and preserve public re-exports where they are already used.

Second, rename `src/structure/app/backend` to `src/structure/app/target`. Update all imports from
`structure.app.backend` to `structure.app.target`. Keep class names such as `BackendCapabilities` unchanged in this
pass because they describe backend capability concepts inside the target layer and renaming them would enlarge the
change without improving the package boundary.

Third, split runtime files into leaf subapps. Move `RunOnlinePySparkTransform.py` to
`src/structure/app/runtime/execution/online/logic/actions/`, `RunGeneratedPySparkTransform.py` to
`src/structure/app/runtime/execution/generated/logic/actions/`, `BuildTransformSchemas.py` to
`src/structure/app/runtime/schemas/logic/actions/`, and session/error/result model files to
`src/structure/app/runtime/session/logic/model/`. Update `src/structure/app/runtime/api/__init__.py` to export
`StructureSession` from the new path.

Fourth, update tests that import old target or streaming paths. The public `structure.app.dsl.api.compile_transform`
path must continue to pass, but direct internal paths in tests should use the new packages.
