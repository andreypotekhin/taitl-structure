# Design and Implement Join Semantics Definitions

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and

This document follows [Plans.md](../../auto/Plans.md). It is self-contained so a contributor can implement the join semantics
design without needing prior conversation history.

## Purpose / Big Picture

Structure lets developers write PySpark enrichment pipelines as typed Python schema transforms. Joins are the first
feature where row count, null behavior, aliasing, and optimizer hints all meet. After this v1 work, a developer can
write `join_one(...)` when lookup cardinality is intended and get clear compiler feedback when the join condition, key
null behavior, aliasing, or uniqueness assumptions are unsafe. `join_many(...)` (v2) covers the companion case where
row multiplication is intended; it uses the same cardinality-aware design but lands later because it changes
validation, traceability, and output-row expectations.

The observable outcome is a compiler and generator that can compile symbolic lookup joins to explicit PySpark joins,
with tests proving the v1 cases described in [JoinSemantics.md](../../../specifications/JoinSemantics.md).

## Decision Log

- Decision: Store the resolved C9 contract as
  [JoinSemantics.md](../../../specifications/JoinSemantics.md).
  Rationale: This matches the repository's current specification convention better than adding a standalone
  `JoinSemantics.md` beside older design notes.
  Date/Author: 2026-06-17 / Codex

- Decision: Require `how` explicitly for v1 join calls.
  Rationale: Join type changes row retention. Requiring the enum in source avoids a hidden default that readers must
  remember.
  Date/Author: 2026-06-17 / Codex

- Decision: Support `Join.LEFT` and `Join.INNER` in the v1 compiled DSL, and reject right, full, cross, semi, and anti
  joins until the row-centric schema model has explicit semantics for them.
  Rationale: Left and inner joins preserve the model where each output row is built from the current row plus optional
  joined scopes. The deferred shapes need separate projection and nullability rules.
  Date/Author: 2026-06-17 / Codex

- Decision: Treat `join_one(...)` as a cardinality promise, not as a deduplication request.
  Rationale: Silently choosing one duplicate right row would hide a data-quality defect and make generated behavior
  harder to trust.
  Date/Author: 2026-06-17 / Codex

- Decision: Represent case-insensitive or normalized joins as explicit expression helpers in the key condition.
  Rationale: Key normalization is business logic. Keeping it in the expression makes generated PySpark and traceability
  reviewable.
  Date/Author: 2026-06-17 / Codex

- Decision: Make null-safe equality an explicit operator through `null_safe_eq(...)`.
  Rationale: Spark normal equality and null-safe equality answer different business questions. Nullable fields alone
  are not enough evidence to choose null matching.
  Date/Author: 2026-06-17 / Codex

- Decision: Mark `join_many(...)` as v2 while preserving its design contract in the join specification.
  Rationale: Intentional row multiplication is a first-class join semantic, not an afterthought. It changes validation,
  traceability, generated-code review, and streaming compatibility expectations enough to deserve its own stage. v1 should
  prove lookup joins first.
  Date/Author: 2026-06-18 / Codex

## Context and Orientation

[Challenges.md](../../design/Challenges.md) C9 says `join_one(...)` needs precise semantics before implementation. It calls out
composite keys, null-safe equality, case-normalized keys, duplicate right-side rows, right-side column projection,
alias naming, field name collisions, join order, broadcast hints, and `join_many(...)` row multiplication.

The public examples in `Readme.md` show transforms declaring named inputs:

    orders = input(OrderRaw)
    customers = input(Customer)
    products = input(Product)

Inside a subtransform, a developer calls a join from an input scope:

    customer = self.customers.join_one(
        on=self.customers.id == order.customer_id,
        how=Join.LEFT,
        hint=JoinHint.BROADCAST,
    )

The joined scope is then used in the output schema constructor:

    return OrderWithCustomer(
        id=order.id,
        customer_name=customer.name,
    )

The relevant design files are:

- [DSL.md](../../design/DSL.md), which names the public imports and says joins are part of the user-facing DSL.
- [SymbolicExecutionEngine.md](../../design/SymbolicExecutionEngine.md), which says symbolic execution records joins from `join_one(...)`.
- [IntermediateRepresentation.md](../../design/IntermediateRepresentation.md), which lists `Join` as a core v1 operation.
- [CompileabilityChecker.md](../../design/CompileabilityChecker.md), which lists boolean join conditions and `join_one(...)` uniqueness
  warnings as compileability checks.
- [PySparkCodeGenerator.md](../../design/PySparkCodeGenerator.md), which requires generated joins to be explicit and UDF-free.
- [NullabilityAndTypeCoercion.md](../../../specifications/NullabilityAndTypeCoercion.md), which owns type compatibility and nullability
  rules for expressions used inside join keys and projections.

Terms used in this plan:

- Current row scope: the symbolic schema object passed to the current subtransform method, such as `order`.
- Joined input scope: a named input declared on the transform class, such as `self.customers`.
- Joined scope: the symbolic object returned by `join_one(...)`, such as `customer`. v2 extends the same idea to
  `join_many(...)`.
- Key pair: one equality comparison between a joined-side expression and a current-side expression.
- Composite key: two or more key pairs combined with logical AND.
- Cardinality: how many output rows can be produced for one current row.
- Lookup join: a join where each current row should match zero or one right-side row.
- Row-multiplying join: a join where one current row may intentionally produce many output rows.
- Null-safe equality: a Spark comparison where null equals null, expressed in Structure as `null_safe_eq(...)`.

## Plan of Work

Start by implementing public API pieces. Add `Join` and `JoinHint` enum values needed by v1, and add `join_one(...)`
to input scope proxies. The method should require keyword-only `on` and `how` arguments. The public method should not
import Spark or touch real DataFrames. It should create symbolic join expressions only when symbolic execution is
active. Leave `join_many(...)` out of the v1 public compiled DSL unless it is explicitly guarded as unsupported with a
diagnostic that points to v2.

Next, teach symbolic execution to record joins. When a subtransform calls `self.customers.join_one(...)`, symbolic
execution should create a joined scope object and append a join operation to the current step plan. The joined scope
must expose fields from the right schema. Later expressions may refer to those fields in output projection and
post-join filters.

Then add join IR. The IR must preserve the joined input name, join occurrence, join method kind, join type, optional
hint, ordered key pairs, equality kind for each pair, right-side fields referenced later, source text when available,
and whether right-side uniqueness is proven. Keep the IR immutable or treat it as immutable after construction, matching
the repository's existing IR direction.

After that, implement condition analysis. The compiler should flatten `&` into ordered key pairs, accept normal
equality and `null_safe_eq(...)`, and reject OR, inequalities, string SQL conditions, same-side comparisons, and
non-boolean conditions. Type compatibility for key expressions should reuse the nullability and type coercion rules.

Then implement cardinality checks. For `join_one(...)`, prove uniqueness when the right key exactly matches a primary
key or future unique-key metadata. If no proof exists, emit a warning by default. Do not deduplicate right rows.
`join_many(...)` (v2) means row multiplication is intentional. If one current row matches three right rows, the
downstream step sees three rows. V1 records the design but implements lookup joins first.

Next, implement PySpark generation. Generate joins in source order with stable aliases. Use qualified column references
for both sides, lower normal equality to `==`, lower `null_safe_eq(...)` to Spark null-safe equality, and apply
`JoinHint.BROADCAST` to the right side only. Project only right-side fields needed by join keys, filters, projection,
traceability, or diagnostics.

Finally, add focused tests and diagnostics. Tests should compile small transforms and, where Spark execution tests are
available, run generated code against small DataFrames to prove cardinality and null behavior. Diagnostics should name
the transform, subtransform, joined input, condition, normalized key pairs, problem, suggested DSL fix, and
[JoinSemantics.md](../../../specifications/JoinSemantics.md).

## Validation and Acceptance

The feature is accepted when the following behaviors are observable through tests:

- A single-field `join_one(...)` with a right primary key compiles without a uniqueness warning.
- A composite `join_one(...)` preserves source key order and generates an AND condition.
- Normal equality on nullable keys does not match nulls.
- `null_safe_eq(...)` on nullable keys matches nulls by lowering to Spark null-safe equality.
- Case-normalized keys lower expression helpers on both sides of the join.
- `join_one(...)` without a uniqueness proof emits a warning and does not deduplicate right rows.
- `Join.LEFT` makes joined fields nullable for output compatibility checks.
- `where(joined.id.is_not_null())` after a left join can require a match.
- Joining the same input twice produces deterministic aliases.
- Right-side projection includes referenced fields and excludes unused right columns.
- Unsupported join types and unsupported conditions fail with diagnostics that include the shortest useful DSL fix.

A representative warning should read like:

    CompileWarning JOIN-W0601: join_one(...) uniqueness is not proven

    Joined input:
      customers

    Join key:
      customers.id == order.customer_id

    Why this matters:
      If customers has duplicate id values, this join can multiply rows.

    Use:
      mark Customer.id as primary_key=True, declare a unique key, or defer this transform until v2 join_many(...)
      if multiplication is intended.

    See [JoinSemantics.md](../../../specifications/JoinSemantics.md)

## Idempotence and Recovery

This work should be additive. Re-running symbolic compilation or tests must not mutate source files except for explicit
generated-code update commands. Avoid hidden Spark startup during import or compileability checks.

If a generated-code snapshot changes unexpectedly, compare the alias order and key pair order first. They must follow
source order and deterministic alias suffixes. If a runtime Spark test fails only because a backend lacks a specific
hint syntax, keep semantic tests and gate the hint syntax through backend capability checks.

Do not paper over duplicate right rows in `join_one(...)` by adding `dropDuplicates(...)`, window ranking, or `first`.
That would change the semantics. Use a uniqueness warning, a runtime uniqueness check, or v2 `join_many(...)`.

## Interfaces and Dependencies

The v1 public imports are:

    from structure import Join, JoinHint

Required enum values:

- `Join.LEFT`
- `Join.INNER`
- `JoinHint.BROADCAST`

Deferred join types may exist in an enum for future compatibility, but the v1 compileability checker must reject them
inside compiled subtransforms.

Required public method on an input symbolic scope:

    join_one(*, on, how, hint=None)

`join_many(*, on, how, hint=None)` is v2 work. If the symbol exists early for forward compatibility, v1 checks must
reject it inside compiled subtransforms with a diagnostic that points to the v2 roadmap.

Required expression support:

- `==` for normal equality;
- `null_safe_eq(...)` for null-safe equality;
- `&` for composite key conjunction;
- compileable helpers such as `lower(...)` and `trim(...)` inside key expressions.

Required compiler components:

- symbolic execution records join operations and returns joined scopes;
- join IR stores ordered key pairs and join metadata;
- compileability checks reject unsupported conditions and warn on unproven `join_one(...)` uniqueness;
- PySpark generation emits explicit joins, aliases, right-side projection, and broadcast hints;
- tests prove cardinality, null semantics, aliasing, projection, diagnostics, and generated code behavior.
