## Purpose / Big Picture

The first executable-slice coding milestone turns Structure from documentation plus a backend-capability slice into an importable
library surface that can understand one real transform. After this change, the first executable slice model fixture under
`res/testing/model/v1/orders` can import `structure`, declare schemas and a transform, and compile that transform into a
deterministic in-memory plan containing inputs, filters, projection fields, and expression helper calls. This is the
smallest useful vertical slice before online PySpark execution and generated PySpark source emission.

The observable result is testable without Spark: importing the first executable slice model does not import PySpark, and compiling
`orders.transforms.order.NormalizeOrders` yields one input named `orders`, one step named `normalize`, two filters, and
three projected output fields in `OrderNormalized` schema order.

## Context and Orientation

The installable package lives under `src/structure`. Project docs describe apps under `src/structure/app` and shared
libraries under `src/structure/lib`. Existing implemented code is concentrated in `src/structure/app/backend`, where a
PySpark capability profile can accept or reject backend feature requirements without importing PySpark.

The first executable slice fixture lives under `res/testing/model/v1/orders`. Its schema module declares `OrderRaw` and `OrderNormalized`.
Its transform module declares `NormalizeOrders`, with one input named `orders`, one expression helper named `clean_id`,
and one public schema-returning method named `normalize`.

For this plan, a "schema" means a Python class inheriting from `Structure` whose class attributes are `field(...)`
declarations. A "symbolic expression" means an immutable Python object representing a Spark-plan-visible operation such
as field access, `lower(...)`, `trim(...)`, decimal cast, or `coalesce(...)`; it is not a PySpark `Column`. A "transform
plan" means the in-memory representation of what a transform does: inputs, ordered steps, filters, and projections.

## Plan of Work

First, add the public DSL surface in small internal modules and re-export it from `src/structure/__init__.py`. The first executable slice
surface must include `Structure`, `field`, `String`, `Decimal`, `Transform`, `transform`, `input`, `expr_fn`, `where`,
`lower`, `trim`, `to_decimal`, and `coalesce`. Importing these symbols must not import PySpark.

Second, implement schema metadata. `Structure` should collect `field(...)` declarations during subclass creation and
store field names in declaration order. Calling a schema class with keyword arguments during symbolic execution should
create a projection record, not a runtime data object. first executable slice needs `String()` and `Decimal(precision, scale)` only, but the
code should leave a clear path for the remaining scalar and collection types.

Third, implement transform metadata. `input(schema)` stores a declarative input object whose name is assigned when the
transform class is created. `@transform` marks a class and records ordered inputs. `Transform.__init__(**inputs)` stores
deferred runtime inputs and rejects unknown names. `Transform.run(session)` delegates to `session.run(self)` later.

Fourth, implement symbolic first executable slice expressions and plan construction. A symbolic row proxy exposes schema fields as
expressions. `where(...)` records filters in an active symbolic context. Expression helpers build call expressions for
`lower`, `trim`, `to_decimal`, and `coalesce`. Calling an `@expr_fn` helper through `self.clean_id(...)` during symbolic
execution should call the wrapped function and return the resulting symbolic expression.

Fifth, add a compiler-facing API such as `compile_transform(transform_class)` under the compiler or symbolic app. For
the first executable slice fixture it should return a plan with one input, one step, two filters, and one projection. The exact module can
move as the compiler app grows, but tests should exercise a stable internal API until the CLI exists.

Sixth, add tests under `tests/user_stories/v1_first_contract` or the closest existing spec package. Tests should insert
`res` on `sys.path`, import the fixture, assert no `pyspark` module was imported, compile
`NormalizeOrders`, and inspect the resulting plan.

## Validation and Acceptance

The first milestone is accepted when a test proves these behaviors:

- `res/testing/model/v1/orders/schemas/order.py` imports successfully.
- `res/testing/model/v1/orders/transforms/order.py` imports successfully.
- Neither import adds a module named `pyspark` to `sys.modules`.
- `NormalizeOrders(orders=object())` stores a deferred input and performs no Spark work.
- Compiling `NormalizeOrders` produces one transform plan named `NormalizeOrders`.
- The plan has one input named `orders` with schema `OrderRaw`.
- The plan has one step named `normalize` from `OrderRaw` to `OrderNormalized`.
- The step has two filters in source order: `order.id.is_not_null()` and `order.customer_id.is_not_null()`.
- The projection has fields `id`, `customer_id`, and `total` in `OrderNormalized` declaration order.
- The `total` projection contains a `coalesce` expression whose first argument is a decimal cast to precision 12 and
  scale 2, and whose second argument is literal `0`.

This milestone does not require PySpark, online DataFrame execution, generated files, joins, hooks, configuration
loading, or CLI commands. Those are later v1 milestones.

## Idempotence and Recovery

The implementation is additive. Re-running tests is safe. Importing the first executable slice fixture repeatedly should produce the same
metadata. If symbolic context leaks between tests, clear it in the plan builder or use a context manager around each
subtransform compilation.

If a new module name turns out awkward, move it with ordinary source edits and update imports in tests. Avoid changing
the first executable slice model fixture unless the public syntax itself is intentionally revised, because that fixture is the contract for
this milestone.

## Interfaces and Dependencies

The first implementation should expose these public imports from `structure`:

    Structure, field, String, Decimal, Transform, transform, input, expr_fn, where,
    lower, trim, to_decimal, coalesce

The first compiler-facing internal API should expose:

    compile_transform(transform_class) -> TransformPlan

`TransformPlan`, `InputPlan`, `StepPlan`, field metadata, and expression nodes should be ordinary Python dataclasses or
equivalent small immutable values. They must not depend on PySpark.
