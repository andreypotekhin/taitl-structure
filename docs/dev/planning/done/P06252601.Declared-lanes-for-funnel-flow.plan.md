# Declared Lanes For Funnel Flow

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [Plans.md](../../auto/Plans.md).

## Purpose / Big Picture

Structure transforms currently use original `input(...)` and final `output(...)` declarations as both public contract
and internal lane selectors. That overworks output declarations when a developer wants descriptive intermediate streams
such as `orders_normalized` or `orders_with_product`. After this change, developers can declare intermediate lanes with
`lane(Schema)`, then use `input(s)` only to start a funnel and `lane(s)` to continue mid- or end-of-funnel work.

The visible result is source code like:

    @transform
    class EnrichOrders(Transform):
        orders_raw = input(OrderRaw)
        products = input(Product)
        orders = lane(OrderNormalized)
        orders_with_product = lane(OrderWithProduct)
        orders_enriched = output(OrderEnriched)

        @transform(input=orders_raw, output=orders)
        def normalize(self, order: OrderRaw) -> OrderNormalized: ...

        @transform(lanes=[orders, products], output=orders_with_product)
        def add_product(self, order: OrderNormalized, product: Product) -> OrderWithProduct: ...

        @transform(lane=orders_with_product, output=orders_enriched)
        def publish(self, order: OrderWithProduct) -> OrderEnriched: ...

## Surprises & Discoveries

- Observation: Existing compiler and PySpark recipes already store lane names separately from frame names.
  Evidence: `StepPlan`, `StepResultPlan`, `PySparkStepRecipe`, and `PySparkStepResultRecipe` all carry lane/frame
  fields, so declared lanes can reuse the existing naming model.

- Observation: Hook decorator normalization must treat `None` as absent.
  Evidence: The first focused test run failed plural hook syntax because the parser saw `lane=None` before
  `lanes=[accepted, audited]`; `_declarations(...)` now ignores `None` singular/plural values.

- Observation: Existing hook usage reasonably selects input lanes for first-funnel hooks.
  Evidence: Tests and docs use `@after(normalize, lane=orders)` where `orders` is an `input(...)`; hook `lane(s)` now
  accepts input declarations while subtransform `lane(s)` remains limited to declared lanes and final outputs.

## Decision Log

- Decision: Add declared `lane(Schema)` instead of string lane names.
  Rationale: Declaration objects preserve IDE navigation, avoid mixed string/symbol APIs, and let intermediate lanes
  declare their intended schema contract.
  Date/Author: 2026-06-25 / Codex and project owner.

- Decision: Keep `input(s)` for original transform inputs and use `lane(s)` for mid- and end-of-funnel sources.
  Rationale: The decorator names now describe whether the method starts a funnel or continues an existing stream.
  Date/Author: 2026-06-25 / Codex and project owner.

## Context and Orientation

The DSL declaration layer lives in `src/structure/app/dsl/model/transforms/`. `InputDeclaration` and
`OutputDeclaration` are class-level descriptors collected by `Transform.__init_subclass__`. The public API is exported
through `src/structure/app/dsl/api/__init__.py` and `src/structure/__init__.py`.

The compiler entry point is `src/structure/app/compiler/frontend/commands/CompileTransform.py`. It collects transform
inputs, discovers schema-returning methods in source order, resolves the source lane for each method, symbolically
executes method bodies, and emits compiler IR. Generated and online PySpark both lower from that IR, so compiler lane
resolution is the central behavior.

Hooks are collected by `CompilerHookCollector`, represented by `HookPlan`, lowered to `PySparkHookRecipe`, rendered by
`RenderPySparkStep`, and executed online by `RunOnlinePySparkTransform` through `PySparkHookInvoker`.

## Plan of Work

First add `LaneDeclaration`, `lane(schema)`, and transform class lane collection. A lane declaration behaves like an
output declaration for descriptor access, but it is neither a constructor input nor a returned result.

Next update decorator parsing. Method-level `input` and `inputs` should accept only original input declarations.
Method-level `lane` and `lanes` should accept declared lanes and declared final outputs that are already available as
pipeline lanes. Method-level `output` and `outputs` should accept declared lanes or declared final outputs. Strings
should be rejected everywhere in this lane API.

Then update compiler resolution. Beginning-of-funnel methods use `input(s)` or unambiguous parameter inference and may
write to a declared lane. Mid-funnel methods use `lane(s)` and may omit `output` only when a single result updates the
first selected lane. Final outputs remain class-level `output(Schema)` declarations.

Then expand hooks. A hook can select one or more sources using `input(s)` or `lane(s)` and can replace one or more
target lanes using `output(s)`. Existing `lane=x` hook syntax remains shorthand for selecting and replacing `x`.

Finally update docs and tests around the new funnel vocabulary, then run focused tests followed by the full test suite.
