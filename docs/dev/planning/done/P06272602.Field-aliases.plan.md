# Add Field Aliases

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [Plans.md](../../auto/Plans.md). It is self-contained so a contributor can implement Structure field
aliases without needing prior conversation history.

## Purpose / Big Picture

Structure currently uses one field name for two different jobs: the Python schema attribute and the Spark DataFrame
column name. That prevents users from modeling input or output DataFrames whose columns are not valid Python
identifiers, such as `promo-code`, `customer id`, or `order.total`.

After this change, users can keep Python code IDE-friendly while preserving the real Spark column name:

    class OrderRaw(Structure):
        promotion_code = field(String(), nullable=True, alias="promo-code")

    def normalize(self, order: OrderRaw) -> OrderNormalized:
        return OrderNormalized(promotion_code=trim(order.promotion_code))

The observable behavior is that Structure code uses `promotion_code`, while input validation, generated Spark schemas,
generated PySpark projections, online execution, and final output DataFrames use the Spark column `promo-code`.

## Decision Log

- Decision: Add `alias: str | None = None` to `field(...)`.
  Rationale: The requested syntax is explicit, compact, source-compatible, and keeps Python attribute names stable.
  Date/Author: 2026-06-27 / Codex

- Decision: Keep `FieldDefinition.name` as the Python field name and add a separate Spark-facing name, exposed as
  `FieldDefinition.alias` plus a convenience property such as `column`.
  Rationale: Most Structure internals, constructor keywords, output assignment matching, and IDE access should continue
  to use Python identifiers. Only Spark-facing boundaries should use the alias.
  Date/Author: 2026-06-27 / Codex

- Decision: If `alias` is omitted, the Spark column name is the Python field name.
  Rationale: Existing schemas, tests, and generated code must continue to behave exactly as they do now.
  Date/Author: 2026-06-27 / Codex

- Decision: Validate aliases as non-empty strings and reject duplicate Spark column names within one effective schema.
  Rationale: Empty names and duplicate output columns are not useful contracts. Failing during schema declaration keeps
  mistakes close to the source.
  Date/Author: 2026-06-27 / Codex

- Decision: Allow aliases that are not Python identifiers, including hyphens, spaces, dots, and Python keywords.
  Rationale: The main user value is supporting real external DataFrame names that Python cannot express as attribute
  names.
  Date/Author: 2026-06-27 / Codex

- Decision: Do not add Spark identifier quoting, escaping, or sanitizing as part of aliases.
  Rationale: This feature addresses Python identifier limitations by letting users provide a separate Spark column name.
  Spark column validity and any backend-specific escaping remain the user's responsibility.
  Date/Author: 2026-06-27 / Codex

- Decision: Make aliases schema-local, except through ordinary schema inheritance.
  Rationale: Each schema class owns the physical DataFrame shape it validates and emits. An input alias such as
  `OrderRaw.promotion_code -> promo-code` should be consumed when projecting into a separately declared
  `OrderNormalized` field unless `OrderNormalized` also declares the alias. If a schema inherits the field definition,
  it inherits the alias because it inherits the field contract.
  Date/Author: 2026-06-27 / Codex

- Decision: Do not add a second public access mechanism such as `row["promo-code"]` in this slice.
  Rationale: `alias=` solves the stated problem while preserving Structure's IDE-friendly Python API. Bracket access is
  a separate design with different ergonomics and diagnostics.
  Date/Author: 2026-06-27 / Codex

## Context and Orientation

Structure is a Python package rooted at `src/structure`. Users define schemas by subclassing `Structure` and assigning
`field(...)` declarations to Python class attributes. These attributes are the names users type in transform methods:

    class OrderRaw(Structure):
        promotion_code = field(String(), nullable=True)

    order.promotion_code

Before this plan, the Python attribute name `promotion_code` also became the Spark column name. This is visible in
generated schema constants, online materialized schemas, validation, compiled projections, and expression rendering.

For this plan, use these terms consistently:

- Python field name: the schema class attribute, such as `promotion_code`. It must be a valid Python identifier because
  Python syntax requires it.
- Spark column name: the real DataFrame field name, such as `promo-code`. This is `alias` when supplied, otherwise the
  Python field name.
- Effective schema: the ordered fields on a schema after inherited fields and local fields are combined.
- Projection: a Spark `select(...)` that emits columns in the output schema order.

Validate aliases in `FieldDeclaration` or `field(...)`. An alias must be `None` or a non-empty string. Reject non-string

Do not reject hyphens, spaces, dots, or Python keywords. Do reject duplicate Spark column names after inheritance is

but the expression data it creates must include the Spark column path:

        "scope": self._structure_scope_name,
        "field": field.column,
        "path": (field.column,),

Update `Expression.__getattr__` for nested structs so it looks up nested fields by Python name but appends the nested
field's Spark column name to `path`. Keep the current Spark pass-through behavior for column strings. Do not add a
quoting or sanitizing layer for aliases. Generated code rendering should still use `repr(...)` when embedding column
names and aliases in source text so generated Python remains syntactically valid when a name contains quotes:

    f"F.col({column_path!r})"
    f".alias({assignment.field.column!r})"
    f"T.StructField({field.column!r}, ...)"

Update these Spark-facing locations to use `field.column`:

- `RenderPySparkExpression._field(...)`
- `PySparkExpressionEvaluator.evaluate(...)`
- projection aliases in `RenderPySparkStep`
- projection aliases in `RunOnlinePySparkTransform`
- `RenderPySparkSchema.field(...)`
- `MaterializePySparkSchema.field(...)`
- `PySparkFrameValidator.validate(...)` and `project(...)`
- generated runtime `project_schema(...)` in `RenderPySparkRuntimeModule`

Validation must continue to read `StructType` fields by Spark column name because live DataFrames know only Spark

    OrderRaw.promotion_code expected Spark column 'promo-code'

Update schema construction and base overlay only if tests show a gap. The constructor keywords should remain Python

    Spark "promo-code" -> promotion_code = field(String(), nullable=True, alias="promo-code")

The generated Python name must be deterministic, valid, non-keyword, and unique within the generated schema class.

## Validation and Acceptance

The implementation is accepted when all of the following are true:

- Existing field declarations without aliases behave exactly as before.
- `field(String(), alias="promo-code")` is accepted.
- `field(String(), alias="")` and `field(String(), alias=123)` fail clearly.
- A schema cannot have two effective fields with the same Spark column name after aliases are resolved.
- Python schema access uses Python field names: `order.promotion_code`.
- Output construction uses Python keyword names: `OrderRaw(promotion_code=...)`.
- Input validation expects the Spark column `promo-code`.
- Generated and materialized Spark schemas contain `StructField("promo-code", ...)`.
- Generated PySpark projections emit `.alias("promo-code")` or equivalent valid Python source.
- Generated PySpark field references pass aliases through to Spark without Structure-specific quoting or sanitizing.
- Online execution and generated execution produce the same output column names.
- Hook validation and `project_output=True` project by Spark column names.
- Aliases are schema-local: `OrderRaw.promotion_code = field(..., alias="promo-code")` reads `promo-code`, while a
  separately declared `OrderNormalized.promotion_code = field(...)` emits `promotion_code`.
- Inherited aliases remain inherited field contract: `class OrderStillRaw(OrderRaw): pass` keeps `promo-code`.
- Nested struct aliases work when accessing `row.address.postal_code`, using the nested schema's declared alias.
- StructureTools schema generation emits `alias=...` for non-identifier Spark fields and Python keywords.
- Public docs show the `promotion_code = field(..., alias="promo-code")` pattern.
- The new user story is marked complete with `+`.
- `make build` passes, or any unrelated failure is recorded here with exact reproduction details.

## Idempotence and Recovery

The implementation is source-compatible for schemas that omit `alias`. If alias work breaks existing tests, first
search for direct uses of `.name` in Spark-facing paths and decide whether that use should be Python-facing or
Spark-facing. Python-facing code keeps `.name`; Spark-facing code uses the chosen alias-aware property.

If a user needs Spark-specific escaping for an unusual alias, document that the alias string is passed through to Spark
and that the user must choose a Spark-compatible physical column name or normalize the DataFrame before Structure sees
it. Do not add automatic quoting in this feature without a new design decision.

If schema generation creates duplicate Python names after sanitizing Spark names, append `_2`, `_3`, and so on to the
Python field name while keeping the original Spark alias. Do not silently drop or merge fields.

If fixture updates become too broad, first prove the feature with small tests outside `res/testing/model`, then update
fixtures in a separate commit or milestone. Keep this plan updated with the reason.

## Interfaces and Dependencies

Public API:

    def field(
        type: StructureType,
        *,
        nullable: bool = True,
        primary_key: bool = False,
        alias: str | None = None,
        metadata: Mapping[str, object] | None = None,
        description: str | None = None,
    ) -> FieldDeclaration:

Internal field model:

    @dataclass(frozen=True)
    class FieldDefinition:
        name: str
        type: StructureType
        nullable: bool = True
        primary_key: bool = False
        alias: str | None = None
        metadata: Mapping[str, object] = dataclass_field(default_factory=dict)
        description: str | None = None

        @property
        def column(self) -> str:
            return self.alias or self.name
