# Design and Implement Nullability and Type Coercion Rules

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement the nullability
and type coercion design without needing prior conversation history.

## Purpose / Big Picture

Structure lets developers write schema-to-schema PySpark transformations in Python. The compiler cannot safely check
those transformations until every expression has a known Structure type and nullability. After this work, a developer
can tell whether an output assignment is accepted, rejected, narrowed by a filter, or needs an explicit conversion
helper before generated PySpark is written.

The observable outcome is a compiler that rejects unsafe assignments such as a nullable expression flowing into a
non-nullable output field, while accepting Spark-compatible assignments such as integer-to-long widening under the
default Spark SQL ANSI settings.

## Decision Log

- Decision: Use Spark-native dotted configuration keys directly under `[tool.structure]`.
  Rationale: Users already know Spark settings as keys such as `spark.sql.ansi.enabled`; preserving that spelling
  avoids a Structure-only alias that must be translated mentally and documented separately.
  Date/Author: 2026-06-17 / Codex

- Decision: Default to `spark.sql.ansi.enabled = true` and `spark.sql.storeAssignmentPolicy = "ANSI"`.
  Rationale: These are the modern Spark defaults and they fail earlier for unsafe conversions.
  Date/Author: 2026-06-17 / Codex

- Decision: Do not require exact type matches for every output assignment.
  Rationale: Exact-only assignment would reject useful and unsurprising Spark-compatible cases such as integer-to-long
  assignment, compatible decimal widening, and typed literals.
  Date/Author: 2026-06-17 / Codex

- Decision: Require explicit helpers for semantic parsing conversions such as string-to-decimal and string-to-timestamp.
  Rationale: Parsing carries data-quality meaning and runtime failure/null behavior. The source should show that choice.
  Date/Author: 2026-06-17 / Codex

- Decision: Let Structure source use ordinary Python literals in expression contexts.
  Rationale: The source DSL should feel like Python. Generated PySpark can still emit `F.lit(...)`.
  Date/Author: 2026-06-17 / Codex

- Decision: Include both `Float()` and `Double()` schema types in v1, and infer Python `float` literals as `Double()`.
  Rationale: Spark exposes separate 32-bit and 64-bit floating types, while Python `float` values are double precision.
  Date/Author: 2026-06-18 / Codex

## Context and Orientation

`docs/dev/design/Challenges.md` lists C3, "Nullability and Type Coercion Rules Are Missing". The challenge asks whether
nullable inputs can feed non-nullable outputs, how casts are represented, how numeric widening and decimal precision
work, what happens on invalid casts, and how defaults with `coalesce(...)` are represented.

The related schema syntax contract is `docs/specifications/SchemaDeclarationSyntax.md`. That document
defines field declarations such as `field(String(), nullable=False)` and `field(Decimal(12, 2), nullable=True)`. This
ExecPlan builds on that contract by specifying which expressions may be assigned to those fields.

The compiler design docs are:

- `docs/dev/design/SymbolicExecutionEngine.md`, which describes turning source expressions into compiler IR.
- `docs/dev/design/IntermediateRepresentation.md`, which describes expression nodes such as `FieldRef`, `Literal`,
  `CallExpr`, and `CastExpr`.
- `docs/dev/design/CompileabilityChecker.md`, which describes compile-time checks such as type compatibility and
  nullability compatibility.

Terms used in this plan:

- Structure type: a Structure schema type object such as `String()`, `Long()`, `Decimal(12, 2)`, or `Timestamp()`.
- Nullability: whether an expression can produce null. This is static compiler metadata, not a scan of actual data.
- Assignment compatibility: whether an expression can be projected into an output schema field.
- Narrowing: a compiler refinement that treats an expression as less nullable after a guard such as
  `where(order.id.is_not_null())`.
- Semantic parsing conversion: a conversion from text or another broad representation into a business type, such as
  string-to-decimal or string-to-timestamp.

## Plan of Work

First, implement configuration support for Spark SQL assumptions. The parser that reads `[tool.structure]` must accept
the dotted keys `spark.sql.ansi.enabled` and `spark.sql.storeAssignmentPolicy`. Defaults are `true` and `"ANSI"`.
The compiler should validate that `storeAssignmentPolicy` is one of `ANSI`, `LEGACY`, or `STRICT`, while the v1 design
only promises detailed assignment rules for the default ANSI policy.

Next, extend expression IR so every expression carries a Structure type and a nullability value. Field references
inherit their type and nullability from schema fields. Literals infer their type from Python values. Calls infer result
metadata from helper rules. For example, `to_decimal(order.total, precision=12, scale=2)` has type `Decimal(12, 2)`,
and `coalesce(expr, 0)` is non-null when any argument is statically non-null.

Then, implement nullability narrowing. A filter `where(order.id.is_not_null())` makes the simple field reference
`order.id` non-null for later projection in the same subtransform. Do not apply broad symbolic theorem proving in v1.
Only simple field references and direct aliases are required.

After that, implement assignment compatibility in the compileability checker. A nullable expression cannot feed a
non-nullable field unless it has been narrowed or made non-null. A type mismatch is accepted only when the configured
Spark SQL assignment policy permits it and the conversion is not a semantic parsing conversion. String-to-decimal,
string-to-date, and string-to-timestamp must use explicit helpers.

Finally, add diagnostics and tests. Diagnostics must explain the failing field, source expression, expected field
contract, and the shortest fix. Tests should cover accepted widening, rejected parsing conversions, nullable-to-
non-nullable assignment, filter narrowing, and literal lowering.

## Concrete Steps

Run all commands from the repository root:

    C:\Taitl\Code\taitl\taitl-structure

Start by adding or updating the configuration model. The end state should accept:

    [tool.structure]
    spark.sql.ansi.enabled = true
    spark.sql.storeAssignmentPolicy = "ANSI"

Then add expression metadata to the IR classes. If the implementation names differ from the design docs, choose the
local names but preserve these concepts:

- `type`: the Structure type of the expression.
- `nullable`: whether the expression may produce null.
- `source`: enough diagnostic context to name the original source expression.

Implement literal inference for these Python values:

- `None` becomes a nullable untyped null literal that can be promoted when another expression supplies a type.
- `str` becomes `String()`.
- `bool` becomes `Boolean()`.
- `int` becomes `Integer()` unless the value is outside 32-bit range, in which case use `Long()`.
- `float` becomes `Double()` by default because Python `float` is double precision.
- `datetime.date` becomes `Date()`.
- `datetime.datetime` becomes `Timestamp()`.

Add helper result rules:

- `to_decimal(value, precision=P, scale=S)` returns `Decimal(P, S)`.
- `coalesce(a, b, ...)` returns the least common compatible Structure type and is nullable only if all arguments are
  nullable.
- `lower`, `upper`, and `trim` return `String()` and preserve the input nullability.
- `is_null` and `is_not_null` return non-nullable `Boolean()`.

Add compileability checks for schema construction. When compiling:

    return OrderNormalized(total=coalesce(to_decimal(order.total, precision=12, scale=2), 0))

the compiler should accept the expression for `total = field(Decimal(12, 2), nullable=False)` because `coalesce(...)`
is non-null and the literal `0` can be typed as `Decimal(12, 2)` through the helper's least-common-type rule.

## Validation and Acceptance

Run:

    poetry run pytest

If the repository still has no tests, record the result and add tests as part of the implementation. At completion,
the following scenarios must be covered by tests:

- A nullable field assigned directly to a non-nullable output field is rejected.
- The same field is accepted after `where(field.is_not_null())`.
- `coalesce(nullable_decimal, 0)` is accepted for a non-nullable decimal output.
- Integer-to-long assignment is accepted under ANSI assignment rules.
- String-to-decimal assignment without `to_decimal(...)` is rejected with a helper suggestion.
- `to_decimal(string_value, precision=12, scale=2)` is accepted for a decimal output.
- Source Python literals generate PySpark `F.lit(...)` where needed.
- A non-default Spark SQL assignment policy is parsed and reflected in diagnostics.

Expected diagnostic shape:

    CompileError SCHEMA-E0301: Nullable expression assigned to non-nullable field

    Output field:
      OrderNormalized.total: Decimal(12, 2), nullable=False

    Source expression:
      to_decimal(order.total, precision=12, scale=2)

    Problem:
      The expression may produce null, but the output field is non-nullable.

    Use:
      total=coalesce(to_decimal(order.total, precision=12, scale=2), 0)

    See docs/specifications/NullabilityAndTypeCoercion.md

strict, add a failing fixture first, update the rule, and record the decision in this ExecPlan.

## Interfaces and Dependencies

The public source DSL should support Python literals in expression contexts. Users should write:

    total=coalesce(to_decimal(order.total, precision=12, scale=2), 0)

Generated PySpark may write:

    F.coalesce(F.col("total").cast("decimal(12,2)"), F.lit(0))

The runtime library must not create or reconfigure a Spark session. Structure uses Spark SQL configuration keys as
declared assumptions for compile-time checking and optional generated runtime assertions.
