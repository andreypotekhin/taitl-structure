# v1 Symbolic Compile Slice

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [Plans.md](../../auto/Plans.md). The first v1 coding slice is intentionally narrow: make the v1 model
fixture importable and compileable into a symbolic plan without requiring PySpark, Java, a SparkSession, or a Spark
cluster. A symbolic plan is the in-memory description produced by `compile_transform(...)`; it records inputs, ordered
subtransforms, filters, projections, joins, and hooks before online execution or generated-code emission exists.

## Purpose / Big Picture

Structure v1 needs to move from the v1 first executable checkpoint single-step projection contract to realistic enrichment pipelines: nested
schemas, multiple ordered subtransforms, lookup joins, expression helpers, hooks, and output schema composition. After
this slice, a developer can import `testing.model.v1.orders.transforms.order.EnrichOrders` and run
`compile_transform(EnrichOrders)` in a plain Python environment with no PySpark installed. The resulting plan should
show the five v1 subtransforms in source order, four declared inputs, three lookup joins, hook metadata around the
right steps, filters, and output schema `OrderPublished`.

This does not yet execute Spark code or emit generated files. It creates the compiler-facing contract that later v1
runtime, code generation, validation, traceability, and diagnostics work can consume.

## Surprises & Discoveries

- Observation: The v1 transform fixture imports `pyspark.sql.functions as F` at module import time only for hook bodies.
  Evidence: `importlib.util.find_spec("pyspark")` returned `None` in this workspace, while v1 compile must remain
  Spark-free. The fixture needs lazy PySpark imports inside hook methods or another Spark-free hook expression.

- Observation: `Structure.base(...)` needs two forms. Enrichment steps call `OrderWithCustomer.base(order)(...)` because
  the right-side join fields are supplied as overrides, while the publish step calls `OrderPublished.base(order, flags)`
  because the sources already provide every output field.
  Evidence: The first v1 compile probe failed with `TypeError: Subtransform returned function, not OrderPublished`
  until `base(...)` returned a schema instance when all fields were available.

## Decision Log

- Decision: Start v1 implementation with symbolic compilation rather than online execution or generated-code emission.
  Rationale: The existing codebase already has a v1 first-contract compiler plan and target capability checks. Symbolic compilation
  is the smallest independently verifiable step that unlocks subsequent v1 runtime, generator, validation, and traceability
  work while preserving the no-Spark compiler contract.
  Date/Author: 2026-06-21 / Codex.

- Decision: Keep the v1 symbolic slice additive and fixture-backed.
  Rationale: The repository already contains v1 model source and expected generated artifacts. Making the fixture
  compileable anchors implementation in the intended user-facing DSL instead of an artificial toy case.
  Date/Author: 2026-06-21 / Codex.

- Decision: Move the v1 fixture's PySpark imports inside hook bodies instead of adding PySpark as a project dependency.
  Rationale: v1 compiler commands must run without PySpark, Java, Spark startup, or a Spark cluster. Hooks are opaque
  runtime escape hatches, so their Spark imports should occur only if a hook actually executes.
  Date/Author: 2026-06-21 / Codex.

- Decision: Preserve the v1 first-contract plan fields and add v1 metadata to `StepPlan` as `joins`, `before_hooks`, and `after_hooks`.
  Rationale: Existing tests and future code can continue to use `filters` and `projection` unchanged, while v1-specific
  lowering can consume the richer metadata explicitly.
  Date/Author: 2026-06-21 / Codex.

## Context and Orientation

The current public package entrypoint is `src/structure/__init__.py`, which re-exports the DSL names from
`src/structure/app/dsl/api/__init__.py`. Schema declarations are implemented by `Structure`, `field(...)`, and type
objects under `src/structure/app/dsl/logic/model/schemas` and `src/structure/app/dsl/logic/model/types`.

Transforms are declared with `@transform`, `input(...)`, and `@expr_fn` in
`src/structure/app/dsl/logic/model/transforms/transform_api.py`. The compiler action
`src/structure/app/dsl/logic/actions/CompileTransform.py` creates a `TransformPlan` with `InputPlan` and `StepPlan`
objects under `src/structure/app/dsl/logic/model/plans`.

The v1 first-contract fixture in `res/testing/model/v1/orders` proves the initial executable contract: one input, one subtransform,
field expressions, filters, expression helpers, and projection. The v1 fixture in `res/testing/model/v1/orders` is the
next source of truth for this slice. It adds nested schemas, arrays, maps, booleans, dates, timestamps, integers, longs,
multiple inputs, hooks, `join_one(...)`, null-safe equality, binary expression operators, and schema composition through
`OutputSchema.base(row)(...)`.

A hook is a user-written method attached before or after a subtransform. Structure treats hook bodies as opaque Spark
escape hatches: the compiler records where they run and their options, but it does not inspect their internal PySpark
code.

A lookup join is a join that adds fields from another declared input without intentionally multiplying rows. In this
slice, `join_one(...)` should produce a symbolic joined row scope whose fields can be projected, and the join metadata
should be recorded on the current step.

## Plan of Work

First, add the missing public DSL names needed by the v1 fixture. This includes type constructors `Boolean()`,
`Integer()`, `Long()`, `Date()`, `Timestamp()`, `Array(...)`, `Map(...)`, and `Struct(...)`; enums or enum-like values
`Join.LEFT`, `JoinHint.BROADCAST`, and `SchemaMode.ALLOW_EXTRA_COLUMNS`; hook decorators `before(...)` and `after(...)`;
and expression operators for `&`, `|`, `~`, equality, subtraction, greater-than, null-safe equality, and nested field
access.

Second, extend compile-time context so `where(...)` continues to collect filters and `join_one(...)` can collect joins
for the active subtransform. `InputDeclaration` should behave as a descriptor: on a transform instance used during
compilation, `self.customers` should return a symbolic input scope, not the declaration object. This keeps join source
syntax natural.

Third, extend the plan model with small focused dataclasses for joins and hooks. `StepPlan` should remain readable and
keep existing v1 first-contract fields stable, while adding `joins`, `before_hooks`, and `after_hooks`. Existing tests should keep
passing without changes except where they benefit from richer metadata.

Fourth, make the v1 fixture import without PySpark by moving `pyspark.sql.functions` imports into hook methods. This is
fixture code, not production compiler code, and it protects the no-Spark compile contract.

Finally, add tests under `tests/specifications/symbolic-execution/`. The tests should assert no PySpark import occurs, the v1
plan has the expected source-order steps and inputs, joins are attached to the right steps, hooks preserve options, and
representative projection expressions such as `net_total = total - discount` and `is_large = total > 1000` are captured.

## Validation and Acceptance

Acceptance is behavioral. In a workspace where `pyspark` is not installed, this Python snippet should complete without
importing Spark:

    import sys
    from structure.app.dsl.api import compile_transform
    from testing.model.v1.orders.transforms.order import EnrichOrders

    before = {name for name in sys.modules if name.startswith("pyspark")}
    plan = compile_transform(EnrichOrders)
    after = {name for name in sys.modules if name.startswith("pyspark")}

    assert after == before
    assert plan.name == "EnrichOrders"
    assert [step.name for step in plan.steps] == [
        "normalize",
        "add_customer",
        "add_product",
        "add_promotion",
        "publish",
    ]

The full `pytest` suite must pass. The new test is expected to fail before implementation because v1 imports names
that the public DSL does not export yet and because the fixture imports PySpark eagerly.

## Idempotence and Recovery

restore the earlier v1 first-contract assertion shape first, then reintroduce v1 metadata additively so `StepPlan.filters` and
`StepPlan.projection` remain backwards-compatible.

## Interfaces and Dependencies

This slice depends only on Python standard library features already used by the project. Do not add PySpark as a
compiler dependency. Do not add a new runtime dependency for symbolic compilation. The public DSL names must be
available from both `structure` and `structure.app.dsl.api` because the fixtures import from `structure`.

The intended new interfaces are:

- `before(target, *, lane, pass_inputs=False, schema_mode=SchemaMode.STRICT, project_output=False, streaming_safe=False)`
- `after(target, *, lane, pass_inputs=False, schema_mode=SchemaMode.STRICT, project_output=False, streaming_safe=False)`
- `InputDeclaration.join_one(*, on, how=Join.LEFT, hint=None)` when accessed through a symbolic input scope.
- `Structure.base(*rows_or_structs)` returning a callable that creates an output schema instance from copied base
  fields plus overrides.
