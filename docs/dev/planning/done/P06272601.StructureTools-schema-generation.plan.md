# Add StructureTools Schema Generation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can implement the
`StructureTools` schema generation API and CLI without needing prior conversation history.

## Purpose / Big Picture

Structure users often start with existing data: a parquet dataset, a Delta table, a Spark table, a live DataFrame, or
a hand-written PySpark `StructType`. Today they must translate that Spark schema into Structure schema classes by hand.
After this work, users can ask Structure to generate a canonical Structure schema class source string from that
existing Spark shape.

The observable result is a small aid tool:

    from structure import StructureSession, StructureTools

    session = StructureSession(spark=spark)
    code = StructureTools.schemas.generate(
        from_path="data/orders.parquet",
        format="parquet",
        session=session,
        to="OrderRaw",
    )
    print(code)

The same behavior is available from the CLI when the shell can start or access PySpark:

    structure tools schemas generate --from-path data/orders.parquet --format parquet --to OrderRaw

The generated source is a starting point that users can paste into their source package and then refine with
`primary_key=True`, descriptions, inheritance, or semantic constraints that Spark schema metadata cannot infer.

## Decision Log

- Decision: Expose one public method, `StructureTools.schemas.generate(...)`, instead of separate `from_path(...)` and
  `from_table(...)` methods.
  Rationale: A single entrypoint keeps the end-user surface compact while named parameters make the source explicit.
  Date/Author: 2026-06-27 / Codex

- Decision: `generate(...)` accepts exactly one source among `schema`, `from_path`, and `from_table`.
  Rationale: Mutual exclusivity avoids guessing which source wins and keeps errors direct.
  Date/Author: 2026-06-27 / Codex

- Decision: Path and table sources accept either `spark=` or `session=StructureSession(...)`, but not both.
  Rationale: Many users already have a `StructureSession`; accepting it avoids unnecessary plumbing while preserving a
  clear Spark source. Supplying both is ambiguous and should fail early.
  Date/Author: 2026-06-27 / Codex

- Decision: Direct PySpark schema generation through `schema=` does not require Spark or `StructureSession`.
  Rationale: A `StructType` already contains all shape metadata needed to generate Structure source.
  Date/Author: 2026-06-27 / Codex

- Decision: Generate nested Spark structs as named Structure classes.
  Rationale: The current Structure DSL represents nested structs with `Struct(SchemaClass)`, not inline anonymous
  structs. Named classes preserve IDE navigation and match existing schema syntax.
  Date/Author: 2026-06-27 / Codex

- Decision: Add CLI under `structure tools schemas generate`.
  Rationale: `tools` gives future end-user aids a home without crowding compiler commands such as `check` and
  `compile`; `schemas` groups schema-oriented tools.
  Date/Author: 2026-06-27 / Codex

- Decision: CLI output prints generated source to stdout and does not write files in this first slice.
  Rationale: Returning or printing source is safe, reviewable, and does not require overwrite or package-layout policy.
  Date/Author: 2026-06-27 / Codex

- Decision: The CLI documentation must explicitly mention Spark-available runtime requirements.
  Rationale: The CLI process must import PySpark and obtain Spark metadata for path/table sources, unlike ordinary
  compiler commands that remain Spark-free.
  Date/Author: 2026-06-27 / Codex

## Context and Orientation

Structure is a Python package rooted at `src/structure`. The public package root is `src/structure/__init__.py`.
End-user schema declarations use classes that inherit `Structure` and fields declared with `field(...)`:

    from structure import Structure, field, String, Decimal

    class OrderRaw(Structure):
        id = field(String(), nullable=False)
        total = field(Decimal(12, 2), nullable=True)

The current Structure schema type constructors are `String()`, `Integer()`, `Long()`, `Float()`, `Double()`,
`Decimal(precision, scale)`, `Boolean()`, `Date()`, `Timestamp()`, `Array(type, contains_null=True)`,
`Struct(SchemaClass)`, and `Map(key_type, value_type, value_contains_null=True)`.

PySpark schemas use `pyspark.sql.types.StructType`, `StructField`, and data type objects such as `StringType()`,
`DecimalType(12, 2)`, `ArrayType(...)`, and `MapType(...)`. This feature maps those PySpark schema objects back into
Structure source code. This is the reverse of the existing PySpark target code that renders Structure schemas into
Spark `StructType` constants.

## Plan of Work

First, create the tools app API. Add `StructureTools` as an endpoint class with a `schemas` sub-endpoint. The endpoint
should instantiate a command class for `generate(...)`. Export `StructureTools` from `src/structure/__init__.py`. This
import must not import PySpark.

The public API should behave as:

    StructureTools.schemas.generate(schema=df.schema, to="OrderRaw")
    StructureTools.schemas.generate(schema=some_struct_type, to="OrderRaw")
    StructureTools.schemas.generate(schema=df, to="OrderRaw")
    StructureTools.schemas.generate(from_path="data/orders.parquet", format="parquet", spark=spark, to="OrderRaw")
    StructureTools.schemas.generate(from_path="data/orders.parquet", format="parquet", session=session, to="OrderRaw")
    StructureTools.schemas.generate(from_table="catalog.db.orders", spark=spark, to="OrderRaw")
    StructureTools.schemas.generate(from_table="catalog.db.orders", session=session, to="OrderRaw")

The method signature should use named parameters only after `self` or on the command `__call__`:

    def __call__(
        self,
        *,
        schema=None,
        from_path: str | None = None,
        from_table: str | None = None,
        format: str | None = None,
        spark=None,
        session=None,
        options: Mapping[str, str] | None = None,
        to: str,
    ) -> str:

Implement request validation before touching Spark. Validation rules are:

- `to` is required and must be a valid Python class name.
- Exactly one source among `schema`, `from_path`, and `from_table` must be supplied.
- `schema=` mode must not require `spark` or `session`.
- `from_path=` mode requires `format`.
- `from_path=` and `from_table=` modes require exactly one of `spark` or `session`.
- If `session` is supplied, it must expose a non-empty `.spark`.
- If both `spark` and `session` are supplied, fail.
- `options` is allowed only with `from_path=`.
- Supported path formats in the first slice are `parquet` and `delta`.

Use a Structure-specific runtime or tool error with an actionable message. If the existing diagnostic registry has a
natural place for this, use it. Otherwise add a focused `StructureToolError` model under the tools app and render it
through the CLI boundary. Error text should tell the user what to pass next, for example:

    Live source schema generation needs Spark metadata access. Pass spark=... or session=StructureSession(...).

Next, implement source extraction. For `schema=`, accept a PySpark `StructType` object directly. Also accept a
DataFrame-like object by checking for `.schema`, then using that value. Do not call Spark actions such as `count`,
`collect`, `take`, `toPandas`, or `.rdd`.

For `from_path=`, resolve Spark from `spark` or `session.spark`, create `reader = spark.read`, apply `options` if
present, then load only enough to inspect schema:

    df = spark.read.options(**options).format(format).load(path)
    schema = df.schema

For `from_table=`, use:

    schema = spark.table(table).schema

Do not start or stop Spark inside StructureTools. The caller owns Spark lifecycle.

Then implement the PySpark-to-Structure mapping. Map these PySpark type classes:

    StringType      -> String()
    IntegerType     -> Integer()
    LongType        -> Long()
    FloatType       -> Float()
    DoubleType      -> Double()
    BooleanType     -> Boolean()
    DateType        -> Date()
    TimestampType   -> Timestamp()
    DecimalType     -> Decimal(precision, scale)
    ArrayType       -> Array(element_type, contains_null=<containsNull>)
    MapType         -> Map(key_type, value_type, value_contains_null=<valueContainsNull>)
    StructType      -> Struct(NestedClass)

The mapper should prefer duck typing over importing PySpark at module import time. Importing `pyspark.sql.types` inside
the called mapping method is acceptable. A fake types test should be possible without PySpark by using class names and
expected attributes if practical. If exact PySpark imports are simpler for runtime mapping, keep them inside the method
called by `generate(...)`, never at public package import time.

Generate source classes rather than Structure runtime class objects. The output should be deterministic and formatted
as plain Python source. Always include one import line:

    from structure import (
        Array,
        Boolean,
        Date,
        Decimal,
        Double,
        Float,
        Integer,
        Long,
        Map,
        String,
        Struct,
        Structure,
        Timestamp,
        field,
    )

If the line length becomes uncomfortable, render it as a parenthesized multiline import. Follow existing docs' 120
character preference.

For nested structs, create named classes before the class that references them. Names should be deterministic and based
on the top-level class plus field path. For example, `OrderRaw.shipping.address` becomes `OrderRawShippingAddress`.
Handle duplicate generated nested class names by appending the next path component if available or a numeric suffix as
a last resort. Record this in tests so snapshot output is stable.

Field names in generated Structure classes are Python attribute names. If a Spark field name is not a valid Python
identifier or is a Python keyword, fail with an actionable message instead of inventing aliases. Structure v1 does not
have field aliases, so silently renaming fields would create incorrect schemas.

Generate `nullable=False` explicitly only when the Spark field is non-nullable; generate `nullable=True` explicitly for
all fields as well if that is more consistent with current examples. Prefer explicit `nullable=...` on every generated
field because the generator's job is to preserve Spark metadata visibly.

Next, add the CLI. In `src/structure/app/cli/api/cli.py`, add a `tools` command group, then a nested `schemas` group,
then a `generate` command:

    structure tools schemas generate --from-path data/orders.parquet --format parquet --to OrderRaw
    structure tools schemas generate --from-table catalog.db.orders --to OrderRaw

The first CLI slice supports live path and table sources. Direct `StructType` object generation is API-only because a
live Python object has no natural shell representation. Do not add `--from-json-schema` or `--from-pyspark-schema-file`
until a separate design defines those input formats.

The CLI command options should be:

    --from-path PATH
    --from-table TABLE
    --format parquet|delta
    --option KEY=VALUE, repeatable
    --to CLASS_NAME

For the CLI Spark runtime, use the simplest explicit contract: the command starts a local `SparkSession` through
PySpark when invoked. Keep the import inside the CLI command path. Do not import PySpark for `structure --help`,
`structure check`, or `structure compile`. Build the session with:

    SparkSession.builder.getOrCreate()

Do not stop the session. The process exit owns cleanup. For Delta, the user's environment must already have Delta
support configured. If PySpark is unavailable or Spark cannot start, fail with a message that tells the user to run the
Python API inside their existing Spark environment and pass `session=StructureSession(spark=spark)`.

Add documentation. In `Readme.md` and `docs/QuickRef.md`, add a short section for "Schema Generation Tools". Show:

- generating from `df.schema`;
- generating from a hand-written PySpark `StructType`;
- generating from a path with `session=StructureSession(...)`;
- CLI command examples;
- the Spark-available CLI runtime requirement;
- a Delta note saying the shell must include the user's normal Delta-capable Spark setup.

Do not overpromise. Say that generated Structure classes preserve Spark schema shape but do not infer primary keys,
descriptions, inheritance, or data-quality constraints.

## Validation and Acceptance

The implementation is accepted when all of the following are true:

- `from structure import StructureTools` works.
- Importing `structure` and invoking `structure --help` do not import PySpark.
- `StructureTools.schemas.generate(schema=struct_type, to="OrderRaw")` returns canonical Structure source.
- `StructureTools.schemas.generate(schema=df, to="OrderRaw")` uses `df.schema` and performs no Spark action.
- `StructureTools.schemas.generate(from_path=..., format="parquet", spark=spark, to="OrderRaw")` returns schema source.
- `StructureTools.schemas.generate(from_path=..., format="parquet", session=session, to="OrderRaw")` uses
  `session.spark`.
- `StructureTools.schemas.generate(from_table=..., session=session, to="OrderRaw")` uses `session.spark.table`.
- Supplying both `spark` and `session` fails clearly.
- Supplying more than one source among `schema`, `from_path`, and `from_table` fails clearly.
- Path mode without `format` fails clearly.
- Path/table mode without Spark or session fails clearly.
- Unsupported Spark types fail with an error naming the unsupported type and field path.
- Invalid generated class names and invalid field names fail instead of being silently renamed.
- Generated nested structs become named Structure classes and are referenced through `Struct(NestedClass)`.
- Arrays and maps preserve element and value nullability.
- Decimal precision and scale are preserved.
- CLI `structure tools schemas generate --from-path ... --format parquet --to OrderRaw` prints source to stdout.
- CLI `structure tools schemas generate --from-table ... --to OrderRaw` prints source to stdout.
- CLI argument errors are readable and do not produce Python tracebacks.
- Docs explain Spark-available CLI runtime requirements and show the `StructureSession` API fallback.
- `make build` passes, or any unrelated failure is recorded with exact reproduction details.

## Idempotence and Recovery

The first implementation is additive. Re-running tests and CLI commands is safe because the tool prints source and does
not write files.

If PySpark import safety regresses, move PySpark imports from module scope into the called function or command path and
add a test around `sys.modules`. If schema generation from fake types becomes too brittle, keep fake tests for request
validation and import safety, and isolate real PySpark type mapping behind integration tests. Do not add PySpark as a
hard package dependency.

If CLI Spark startup is unreliable in the local environment, keep CLI argument and import-safety tests unit-level and
put live path/table behavior behind the existing integration marker. Document the requirement rather than trying to
hide Spark startup policy inside Structure.

If field-name aliasing becomes necessary for a real dataset, stop and create a separate design. Structure v1 schema
fields use Python attribute names as field names, so this plan must fail on non-identifier Spark field names instead of
generating incorrect aliases.
