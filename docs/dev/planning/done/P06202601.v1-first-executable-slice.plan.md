## Purpose / Big Picture

Structure's v1 roadmap is intentionally ambitious, but it is too large to be the first proof point. The first executable slice contract is
the smallest useful end-to-end behavior: a developer writes one typed transform, Structure runs it online through
`StructureSession`, Structure can also emit generated PySpark for the same behavior, and tests prove both paths return
the same rows.

After the first executable slice, the team can validate real user ergonomics before adding joins, hooks, compiler traceability, streaming reports,
doctor checks, and build integration. The first executable slice is an internal dev/test planning label, not public documentation language.

## Context and Orientation

Structure is a Python library for typed Spark transforms. Users write source files under ordinary Python packages.
The compiler discovers classes decorated with `@transform`, reads schema classes derived from `Structure`, builds an
intermediate representation of schema-to-schema methods, runs transforms online through `StructureSession`, and can
also emit generated PySpark code under `generated/structure_generated`.

The first executable slice fixture lives under `res/testing/model/v1`. Source fixture files are under `res/testing/model/v1/orders`.
Expected generated files are under `res/testing/model/v1/structure_generated`. The v1 fixture remains under
`res/testing/model/v1` and intentionally contains future breadth such as joins and hooks.

Terms used in this plan:

- Online execution means running the compiled transform through `StructureSession` without importing generated files.
- Generated execution means running emitted PySpark classes from `structure_generated`.
- Parity means online execution and generated execution produce the same DataFrame rows and schema for the same input.
- Input validation means the runtime checks the incoming DataFrame schema before applying transform logic.

## Plan of Work

First, implement only the public API needed by `res/testing/model/v1/orders/transforms/order.py`: `Structure`, `field`,
`String`, `Decimal`, `Transform`, `transform`, `input`, `expr_fn`, `lower`, `trim`, `coalesce`, `to_decimal`, `where`,
and `StructureSession`.

Next, build the minimal discovery and symbolic execution path. Discovery must find the `NormalizeOrders` transform,
its single `orders` input, and the `normalize` method. Symbolic execution must treat `order.id`, `order.customer_id`,
and `order.total` as field references; inline `clean_id(...)`; capture both `where(...)` calls; and lower the returned
`OrderNormalized(...)` constructor to projection fields.

Then, implement the shared PySpark execution recipe for the first executable slice operation set. Both online execution and generated
PySpark emission should use the same recipe model for filtering and projection so parity is not duplicated by hand.

Finally, add tests that exercise the first executable slice fixture in three ways: compile/check without Spark, online execution with a
small Spark DataFrame, and generated execution with the same data. The parity test should compare rows after sorting by
`id` and should assert the generated path validates input schema before transformation.

## Validation and Acceptance

The first executable slice is accepted when a human can observe these behaviors:

- `structure check --source-root res` reports success without importing PySpark or starting Spark.
- Online execution normalizes sample order rows, drops rows with missing `id` or `customer_id`, trims and lowercases
  identifiers, casts `total` to decimal, and fills missing totals with zero.
- Generated execution returns the same rows as online execution for the same input DataFrame.
- Generated code validates the input schema before applying filters and projection.
- The first executable slice fixture contains no joins, hooks, traceability artifacts, streaming metadata, or multi-input transform.

## Idempotence and Recovery

The plan is additive. Re-running static scans and tests is safe. Regenerating first executable slice output should either reproduce the
checked-in `res/testing/model/v1/structure_generated` fixture or fail with a clear diff. If implementation work expands beyond the
listed first executable slice scope, move the extra behavior to v1 planning before merging.

## Artifacts and Notes

The first executable slice source transform should have this shape:

    @transform
    class NormalizeOrders(Transform):
        orders = input(OrderRaw)
        normalized = output(OrderNormalized)

        @expr_fn
        def clean_id(value):
            return lower(trim(value))

        def normalize(self, order: OrderRaw) -> OrderNormalized:
            where(order.id.is_not_null())
            where(order.customer_id.is_not_null())

            return OrderNormalized(
                id=self.clean_id(order.id),
                customer_id=self.clean_id(order.customer_id),
                total=coalesce(to_decimal(order.total, precision=12, scale=2), 0),
            )

The generated PySpark transform should validate input, apply filters, project three fields, validate output if the
output helper exists by then, and return the DataFrame.

## Interfaces and Dependencies

The first executable slice depends on PySpark only for runtime execution tests and generated execution tests. Compiler/check behavior must stay
Spark-free. The public source API required for first executable slice is deliberately tiny and should not expose joins, hooks, traceability,
streaming compatibility flags, doctor checks, or build integration.
