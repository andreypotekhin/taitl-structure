# Multiple Transform Outputs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [Plans.md](../../auto/Plans.md). It adds named transform outputs so one Structure transform can return more
than one DataFrame while preserving explicit schema contracts, online execution, generated PySpark parity, and readable
DSL syntax.

## Purpose / Big Picture

Many real data transforms naturally produce more than one result. A quality gate may produce accepted rows and rejected
rows. A dimension enrichment may produce enriched orders and lookup misses. A CDC transform may produce upserts and
deletes. Today Structure treats the final step as the single transform output, so users must split these related results
into separate transform classes or hide one result inside hooks or ad hoc side effects.

After this change, a developer can declare named output lanes on a transform class. Transform methods execute in source
order, while each method reads from one lane and writes to one lane:

    @transform
    class RouteOrders(Transform):
        orders = input(OrderRaw)

        accepted = output(OrderAccepted)
        rejected = output(OrderRejected)

        def normalize(self, order: OrderRaw) -> OrderNormalized:
            return OrderNormalized.base(order)(
                id=order.id,
                customer_id=order.customer_id,
                amount=order.amount,
            )

        @transform(output=accepted)
        def accept(self, order: OrderNormalized) -> OrderAccepted:
            where(order.customer_id.is_not_null())

            return OrderAccepted.base(order)(
                status="accepted",
            )

        @transform(input=accepted, output=accepted)
        def keep_accepted(self, order: OrderAccepted) -> OrderAccepted:
            where(order.status == "accepted")

            return OrderAccepted.base(order)()

        @transform(output=rejected)
        def reject(self, order: OrderNormalized) -> OrderRejected:
            where(order.customer_id.is_null())

            return OrderRejected.base(order)(
                reason="missing customer",
            )

    result = RouteOrders(orders=orders_df).run(session)
    accepted_df = result.accepted
    rejected_df = result.rejected

The user-visible behavior is explicit: named outputs are declared in the transform body, their schemas are visible from
`transform.schemas.outputs`, and `run(session)` returns a read-only result object for every transform.

## Decision Log

- Decision: Represent multiple outputs with field declarations such as `accepted = output(OrderAccepted)`.
  Rationale: Class decorator parameters can declare one final schema but cannot give stable names to multiple outputs.
  Field declarations fit the existing `orders = input(OrderRaw)` mental model and keep the transform contract visible at
  class-body level.
  Date/Author: 2026-06-22 / Codex.

- Decision: Treat the transform output contract as required, but allow it to be satisfied either by class-level
  `@transform(to=Schema)` for a single unnamed/default output or by one or more field-declared `output(Schema)`
  ports.
  Rationale: The user wants output locking so accidental method reordering or extra terminal steps cannot silently
  change the public result. Field-declared outputs provide the same lock for multi-output transforms.
  Date/Author: 2026-06-22 / Codex.

- Decision: Replace terminal-only output methods with a source-order lane model.
  Rationale: Users need to split into multiple outputs at any point in the pipeline and then continue work on those
  branches. Lanes preserve source-order readability without introducing a DAG scheduler: undecorated methods are
  `df -> df`, `@transform(output=accepted)` is `df -> accepted`, and
  `@transform(input=accepted, output=accepted)` continues a branch.
  Date/Author: 2026-06-22 / Codex.

- Decision: Bind named outputs to ordinary schema-returning transform methods by using method-level
  `@transform(output=accepted)`.
  Rationale: The feature should stand on the canonical v1 transform model, where public methods produce typed schema
  boundaries. Reusing `@transform` avoids adding a second terminal-method decorator shape, while `output=` reads
  naturally as the method's destination result port. This keeps output names visible at class-body level, preserves
  method source locations for diagnostics and hooks, and avoids depending on the separate Flow Method `@pipeline` plan.
  Date/Author: 2026-06-22 / Codex.

- Decision: Method-level `input=` is explicit and always paired with `output=`.
  Rationale: A method that names its input lane is branch-aware and should also make its destination lane obvious.
  Omitting `input=` keeps the concise default of reading from canonical `df`; output-local filters then affect only the
  written lane and never mutate sibling lanes.
  Date/Author: 2026-06-22 / Codex.

- Decision: `run(session)` always returns a read-only `TransformResult`; single-output results expose `df`.
  Rationale: Returning a result object for both single-output and multi-output transforms keeps the runtime API uniform.
  The `df` name matches hook conventions and gives every single-output transform an obvious result slot. Named outputs
  remain available by attribute and mapping access: `result.accepted`, `result.rejected`, and `result["accepted"]`.
  Date/Author: 2026-06-22 / Codex.

- Decision: A transform with exactly one output does not require an explicit output lane binding.
  Rationale: If there is only one output contract, the final shared transform method can unambiguously produce it.
  When that output is field-declared, the result exposes both `result.df` and the field name, such as `result.out`.
  When there is more than one field-declared output, every public result lane must be written explicitly with
  `@transform(output=...)`.
  Date/Author: 2026-06-22 / Codex.

## Outcomes & Retrospective

This plan is implemented for the current v1 slice. It introduces named output lanes as a first-class part of the
transform contract and makes the runtime result shape uniform across single-output and multi-output transforms. The
final revision avoids arbitrary DAG scheduling while allowing branches throughout the source-order pipeline. Named
lanes can be produced, consumed, and updated at any point after they become available.

## Context and Orientation

Structure currently compiles a transform class into a linear `TransformPlan`. A `TransformPlan` has declared inputs,
ordered `StepPlan` entries, and an `output_schema` property that returns the output schema of the final step. PySpark
lowering turns that plan into `PySparkExecutionPlan`, which has input recipes, step recipes, and one final validation.
`StructureSession.run(...)` lowers the compiled plan, validates bound inputs, exposes schemas on the transform
invocation, and then runs either online PySpark interpretation or generated PySpark code.

This plan adds an output lane. An output lane is a named DataFrame stream inside a transform and, when field-declared,
a public transform result. It has a stable public name, such as `accepted`, and a declared Structure schema, such as
`OrderAccepted`. The compiler maintains a source-order lane table so a method may consume only a lane already available
earlier in the class body.

The current `input(Schema)` descriptor already reserves class-body names for transform inputs. This plan adds a sibling
descriptor `output(Schema)`. A schema-returning method writes an output lane with method-level
`@transform(output=...)`:

    accepted = output(OrderAccepted)

    @transform(output=accepted)
    def accept(self, order: OrderNormalized) -> OrderAccepted:
        where(order.customer_id.is_not_null())
        return OrderAccepted.base(order)(status="accepted")

The decorated method remains a normal compiled transform method: it has a row parameter, a Structure return annotation,
ordinary `where(...)` filters, joins if needed, and a schema construction return value. The method-level transform
metadata binds the method to lanes. `@transform(output=accepted)` reads from `df` and writes `accepted`.
`@transform(input=accepted, output=accepted)` reads and writes the accepted lane. Any method that specifies `input=`
must also specify `output=`.

## Plan of Work

First add the public DSL surface. Introduce `output(SchemaClass)` as a public function exported from `structure`. It
creates an `OutputDeclaration` descriptor, similar in spirit to `InputDeclaration`, with a name assigned from the class
attribute and a schema that must be a `Structure` subclass. Extend `Transform.__init_subclass__` to collect inherited
and local output declarations into `_structure_outputs`.

Extend `@transform` metadata so a transform has a required output contract. A transform is valid when it declares
exactly one class-level `to=Schema` or one or more field-declared outputs. If both forms are present in the first
release, fail with a direct diagnostic asking the author to choose one style. A compiled transform with no output
contract should fail during check/compile with a message explaining how to add `@transform(to=Schema)` or
`name = output(Schema)`. Keep class-level `to=` reserved for result schema contracts. Method-level lane binding uses
`input=OutputDeclaration` and `output=OutputDeclaration`; passing `output=` on a class decorator, or passing `to=` on a
method decorator, should fail clearly.

Add IR for named outputs. Create `OutputPlan` with fields `name`, `schema`, `source`, `source_scope`, `source_schema`,
and `ordinal`. Add `outputs: tuple[OutputPlan, ...]` to `TransformPlan`. Keep method-level output bindings in
source-order `StepPlan` entries and add each step's `source`, `input_lane`, and `output_lane`. Change
`TransformPlan.output_schema` into a compatibility property that returns the sole output schema when there is exactly
one output and raises a clear error when there are multiple outputs. New code should use `TransformPlan.outputs`.

Add canonical lane method binding. Extend `@transform` so it may decorate methods with
`@transform(output=output_declaration)` and `@transform(input=input_declaration, output=output_declaration)`. During
transform discovery, every schema-returning method remains a source-order step. The compiler maintains a lane table:
undecorated methods are `df -> df`, `@transform(output=accepted)` is `df -> accepted`, and
`@transform(input=accepted, output=accepted)` continues a branch. The row parameter annotation must match the current
schema of the input lane. A decorated method's return annotation must match the schema declared by its output lane.

For single-output transforms, synthesize or resolve one primary output from the final `df` lane when no method
explicitly writes an output. For `@transform(to=Schema)`, the primary output is unnamed and exposed as `df`. For one
field-declared output, such as `out = output(OrderOut)`, the final `df` lane is exposed as both `df` and `out`. This
keeps method-based transforms concise while moving runtime execution to the uniform `TransformResult` model. Transforms
with more than one field-declared output must write every output lane explicitly with `@transform(output=...)`.

Extend PySpark lowering. Replace the single `final_validation` field in `PySparkExecutionPlan` with
`outputs: tuple[PySparkOutputRecipe, ...]`, where each output recipe contains the output name, schema, source step name
or source ordinal, filters, projection, and strict validation recipe. For the synthesized single output, create one
recipe named `result`. Keep a compatibility property or adapter for old tests only during migration, then update tests
to assert the new output recipes.

Extend online execution. `RunOnlinePySparkTransform` should execute steps in source order, reading each step's source
frame and storing the produced frame under the step name. Each output DataFrame must be validated against its declared
schema. Every transform returns a read-only `TransformResult`. When there is exactly one output, `result.df` references
that DataFrame; when that output is field-declared, the declared name also references it.

Add `TransformResult`. It should preserve output names and reject mutation. It should support `df` for single-output
results, attribute access for valid Python output names, mapping access by string name, iteration over names, and an
`as_dict()` method. It should raise a helpful `AttributeError` for unknown names and should not silently expose
internal attributes as outputs. For multi-output results, `df` should be absent unless a future primary-output rule is
explicitly added.

Extend runtime schema exposure. Change `TransformSchemas` to include `outputs: Mapping[str, object]`. Keep `.output` as
a convenience property that returns the only output schema. If a transform has multiple outputs, `.output` must raise a
message such as `Transform has multiple outputs: accepted, rejected. Use schemas.outputs["accepted"] instead.` For an
unnamed class-level single output, expose the schema under `outputs["df"]`. For one field-declared output, expose it
under the field name and keep `.output` as the single-output convenience.

Extend generated PySpark. Generated transform classes should calculate shared steps once, then calculate and validate
each output DataFrame. Generated transforms should return the generated or runtime `TransformResult` object for both
single-output and multi-output transforms. Generated traceability should list named outputs and the source
step/projection for each.

Update CLI and specifications. `structure check` should validate output declarations and output bindings without
running Spark. `structure explain` should show declared outputs and lane-writing methods. `structure compile` should
generate schema constants for every output schema and generated transform code that returns a named result object.
Public docs should include accepted/rejected, lookup misses, CDC upsert/delete, single-output `result.df`, and
single-field-output `result.df` plus `result.out` examples.

## Validation and Acceptance

The feature is complete when tests prove:

- `output` is importable from `structure`.
- `output(Schema)` may be used as a transform class field.
- Transform classes collect output declarations in inheritance-aware declaration order.
- A transform must declare an output contract through either `@transform(to=Schema)` or field-declared outputs.
- A transform cannot mix class-level `to=Schema` and field-declared outputs in the first release.
- Canonical schema-returning methods can bind multiple named outputs with method-level `@transform(output=...)`.
- Method-level `@transform(output=...)` accepts only an `OutputDeclaration`.
- Class-level `@transform(to=...)` remains the only result-schema form and accepts only a schema class.
- A single-output transform does not require an explicit method `output=...` binding.
- A transform with more than one field-declared output requires explicit `output=...` lane writes.
- `@transform(output=accepted)` reads from `df` and writes `accepted`.
- `@transform(input=accepted, output=accepted)` continues the accepted branch.
- A method with `input=` but no `output=` is rejected.
- A method cannot consume a lane before it is available in source order.
- Row parameter schema mismatch with the input lane fails clearly.
- Return schema mismatch with the output lane fails clearly.
- Output-local `where(...)` filters affect only the written lane.
- Output projection validation uses the declared output schema.
- `TransformPlan.outputs` contains named output plans.
- Single-output transforms still expose `TransformPlan.output_schema` and return `TransformResult`.
- `TransformResult` supports `result.df` for single-output transforms.
- `TransformResult` supports field-declared aliases such as `result.out` for single-output field declarations.
- `TransformResult` supports `result.accepted`, `result["accepted"]`, and `result.as_dict()`.
- `transform.schemas.outputs` exposes every materialized output schema.
- `transform.schemas.output` works for single-output transforms and fails clearly for multi-output transforms.
- Online execution validates every output DataFrame.
- Generated execution returns the same output names and rows as online execution.
- `structure check`, `structure explain`, and `structure compile` understand named outputs.
- Existing tests for compiled single-output transforms continue to pass.

## Idempotence and Recovery

This change changes the runtime return shape for single-output transforms. Keep the compiler path working at every
milestone by synthesizing one primary output recipe for old-style transforms, then wrapping it in `TransformResult`
before extending to multi-output recipes. If multi-output lowering destabilizes online execution, restore the
single-output `TransformResult` adapter first, then reintroduce named output recipes behind tests.

Do not remove method-based compiled transforms or the Flow Method plan. Multiple outputs should work first with
canonical method-based transforms because that is the stable source shape. Flow Method output binding may be added later
as a convenience after both plans have landed.

If generated PySpark support becomes too large for one change, land online multi-output execution first behind a
diagnostic that generated execution for multi-output transforms is not yet implemented. Record that temporary limitation
in this plan and remove it before marking the plan complete.
