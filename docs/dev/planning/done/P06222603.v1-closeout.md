Some project-management documents still mark v1 milestones as in progress, and the live PySpark matrix is implemented
but not validated in this local PySpark-free workspace. The first executable slice is now part of v1, so repository
fixtures, tests, and documentation should tell one v1 story.

The visible result is that stale first-slice version references are gone, `make build` passes, and the v1 closeout
status in project-management docs names only real release validation follow-ups.

## Context and Orientation

The repository keeps test fixtures under `res/testing/model`. The v1 fixture under `res/testing/model/v1` contains the
current orders example and generated expected outputs. Tests under `tests/user_stories/` back user-story items from
`docs/specifications/UserStories.md`; tests under `tests/specifications/` back formal specification documents under
`docs/specifications/`.

The v1 first-contract test proves four behaviors: Spark-free fixture imports, a one-input normalization transform plan,
decimal `coalesce(to_decimal(...), 0)` projection lowering, and deferred invocation unknown-input validation. It proves
them against `testing.model.v1.orders.transforms.order.EnrichOrders` by inspecting the `normalize` step and constructing
`EnrichOrders(...)` invocations.

## Plan of Work

First, move the first-contract spec test to v1-first-slice terminology and point it at
`testing.model.v1.orders.transforms.order.EnrichOrders`. The test should inspect the first `normalize` step. It should
keep the same behavioral assertions where possible: Spark-free import, input plan shape, filters on required IDs,
projection of normalized fields, decimal coalesce expression lowering, and deferred invocation errors.

Second, remove the retired first-slice fixture tree and delete obsolete pyproject mypy overrides. Because the behavior
is covered through the v1 fixture, deleting the old tree should not remove unique coverage.

Third, replace stale version wording in docs with "first executable slice", "first executable slice", or simply "v1"
depending on context. Historical completed plans should be updated too because the closeout goal is repository-wide
terminology consistency, not only active code cleanup.

Fourth, update project-management docs. Milestones should distinguish "implemented and build-clean" from "needs live
PySpark matrix validation". Sprint 05 should remain open only for the external PySpark validation environment if local
PySpark is unavailable.

Finally, run focused tests for the rewritten v1-first-slice coverage and config/build-sensitive tests, then run the
full build with the known Black cache workaround:

    $env:BLACK_CACHE_DIR='NUL'; make build

coverage.
