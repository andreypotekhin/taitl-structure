# Add Multiple Schema Parameters and Results to Subtransforms

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [Plans.md](../../auto/Plans.md). It extends the canonical schema-returning subtransform model without
introducing a second transform language or bypassing the backend-neutral intermediate representation.

## Purpose / Big Picture

Structure subtransforms currently accept exactly one schema-annotated row parameter and return exactly one schema
instance. This forces a method that enriches one row stream from another declared input to reach through `self`, even
when both schemas are essential to understanding the method. It also leaves method-level input and output mapping
ambiguous when several declarations use the same schema.

After this change, a developer can make every schema dependency visible in the method signature and bind repeated
schemas explicitly:

    @transform
    class NormalizeOrders(Transform):
        orders_external = input(OrderRaw)
        orders_internal = input(OrderRaw)
        products = input(Product)

        accepted = output(OrderWithProduct)
        audited = output(OrderWithProduct)

        @transform(
            inputs=[orders_external, products],
            outputs=[accepted, audited],
        )
        def add_product(
            self,
            order: OrderRaw,
            product: Product,
        ) -> tuple[OrderWithProduct, OrderWithProduct]:
            product = join_one(
                product,
                on=(
                    (product.tenant.tenant_id == order.tenant.tenant_id)
                    & (product.id == order.product_id)
                ),
                how=Join.LEFT,
            )
            where(product.id.is_not_null())

            enriched = OrderWithProduct.base(order)(
                product_name=product.name,
            )
            audited_order = OrderWithProduct.base(order)(
                product_name=product.name,
            )
            return enriched, audited_order

The plural decorator arguments map by position. `inputs[0]` binds the first method parameter and selects the driving
DataFrame. Later input bindings expose symbolic relation parameters that can be joined explicitly. `outputs[0]` binds
the first returned schema value, and so on. Existing one-parameter methods and singular `input=` and `output=` options
remain valid shorthands.

A multi-result subtransform has one relational prefix: before hooks, joins, and `where(...)` filters run once. Each
returned schema value is then projected from that same row set into its own result lane. If outputs require different
row predicates, such as accepted and rejected partitions, use separate lane-writing subtransforms as today. This plan
does not hide branching control flow inside tuple return positions.

## Surprises & Discoveries

- Observation: The current compiler rejects every schema-returning method that has more than one non-`self`
  parameter.
  Evidence: `CompileTransform._row_parameter(...)` in
  `src/structure/app/compiler/frontend/logic/actions/CompileTransform.py` requires exactly one row parameter.

- Observation: Input ambiguity handling already exists, but only for the single current-row source.
  Evidence: `CompileTransform._input_for_schema(...)` finds one declared input by schema and recommends
  `@transform(input=that_input)` when several declarations match.

- Observation: Multiple public output lanes already exist, but one method can write only one lane.
  Evidence: `StepPlan` has singular `output_schema`, `projection`, and `output_lane` fields, while
  `tests/specifications/multiple-outputs/test_v1_multiple_transform_outputs.py` models each lane write as a separate
  method.

- Observation: Current lookup syntax is attached to `InputScope`, so an explicitly passed schema parameter needs
  either method behavior on that proxy or a free function.
  Evidence: `src/structure/app/dsl/logic/model/expr/InputScope.py` owns `InputScope.join_one(...)`; no public
  free-standing `join_one(...)` is exported from `src/structure/__init__.py`.

- Observation: The online runner obtains right-side join frames only from original invocation inputs.
  Evidence: `RunOnlinePySparkTransform._step(...)` uses `inputs[join.input_name]`. Supporting a schema parameter bound
  to an earlier output lane therefore requires join sources to be frame references, not only input names.

- Observation: The generated and online paths already preserve named frames by source step, which is a strong base for
  multi-result execution.
  Evidence: both `RenderPySparkTransformModule` and `RunOnlinePySparkTransform` maintain a source/frame mapping while
  walking source-order recipes.

## Decision Log

- Decision: Add `inputs=` and `outputs=` as ordered plural counterparts to method-level `input=` and `output=`.
  Rationale: Positional mapping is visible, deterministic, and works when several declarations share one schema.
  Sets, dictionaries keyed by schema, and name-based magic would either lose order or remain ambiguous.
  Date/Author: 2026-06-23 / Codex.

- Decision: Preserve singular decorator options as exact one-item shorthands and reject simultaneous singular and
  plural forms.
  Rationale: Existing source remains valid, while conflicting forms such as `input=x, inputs=[x]` fail early instead
  of creating precedence rules.
  Date/Author: 2026-06-23 / Codex.

- Decision: Permit omitted plural bindings when every parameter or result can be matched uniquely by schema; require
  `inputs=[...]` or `outputs=[...]` only when inference has zero or several candidates.
  Rationale: The plural options are ambiguity resolvers, not ceremony required for every multi-schema method.
  Repeated schemas, including the motivating two `OrderRaw` inputs or two `OrderWithProduct` outputs, become explicit.
  Date/Author: 2026-06-23 / Codex.

- Decision: Treat the first schema parameter as the driving relation and all later schema parameters as explicit
  relation dependencies.
  Rationale: Spark operations need one left/current DataFrame. Making the first binding the driver mirrors ordinary
  function reading order and avoids an additional `from=` option.
  Date/Author: 2026-06-23 / Codex.

- Decision: Support both declared inputs and already available output lanes as parameter bindings.
  Rationale: The singular `input=` option already accepts either `input(...)` or `output(...)` declarations. The plural
  form should compose with existing lane flow rather than becoming a weaker special case. Internal join recipes must
  identify a general frame source so online and generated execution can join either kind.
  Date/Author: 2026-06-23 / Codex.

- Decision: Add public `join_one(relation, *, on, how=Join.LEFT, hint=None)` and retain
  `self.products.join_one(...)` as a compatibility spelling that delegates to the same operation.
  Rationale: A parameter named `product_source` should read as an ordinary function argument, not require access back
  through `self`. One implementation path prevents the method and function forms from drifting.
  Date/Author: 2026-06-23 / Codex.

- Decision: Use standard Python tuple annotations and tuple values for multiple results.
  Rationale: `-> OrderWithProduct, OrderWithProduct` is not valid Python annotation syntax. The supported form is
  `-> tuple[OrderWithProduct, OrderWithProduct]`, and the method returns a tuple of schema instances in the same order.
  Date/Author: 2026-06-23 / Codex.

- Decision: Execute the relational prefix once, then project each returned schema independently.
  Rationale: Duplicating one symbolic method into several complete steps would repeat joins and filters in generated and
  online plans. A shared prefix preserves source intent and avoids accidental duplicate Spark work.
  Date/Author: 2026-06-23 / Codex.

- Decision: Apply `where(...)` predicates to every result from a multi-result method.
  Rationale: `where(...)` is currently a subtransform-level operation. Associating calls with arbitrary tuple positions
  would require new syntax and non-obvious ordering semantics. Different row partitions remain explicit separate
  subtransforms.
  Date/Author: 2026-06-23 / Codex.

- Decision: Preserve after hooks for multi-result subtransforms by selecting their input lane with `df=`.
  Rationale: A before hook has one unambiguous driving DataFrame. An after hook remains a one-DataFrame hook when
  `@after(add_product, lane=accepted)` binds it to one returned lane. Single-result after hooks use the same
  explicit `lane=` selector.
  Date/Author: 2026-06-23 / Codex.

- Decision: Keep a raw relation parameter unusable in projection or `where(...)` until it has been joined; use the
  `RowScope` returned by `join_one(...)` for joined fields.
  Rationale: Reading an unrelated DataFrame column without a relational operation has no valid Spark meaning. Failing
  during compilation gives the user a direct correction instead of emitting an unresolved column.
  Date/Author: 2026-06-23 / Codex.

## Outcomes & Retrospective

The feature is implemented end to end. Ordered `inputs=[...]` and `outputs=[...]` resolve repeated-schema ambiguity,
while singular forms remain compatible. `join_one(product, ...)` supports ordinary parameter composition and permits
the concise `product = join_one(product, ...)` spelling. Multi-result steps lower one shared join/filter prefix and
project directly into output-named frames such as `accepted` and `audited`. Result-specific after hooks use
`@after(method, lane=output_declaration)`.

The full repository build passed: formatting, linting, typing, 118 tests, source distribution, and wheel. The live
PySpark matrix remains environmentally unverified because Docker Desktop was not running. A dedicated integration
scenario is present and will run in both configured backend lanes when `make integration` is rerun with Docker.

## Context and Orientation

The public authoring surface lives under `src/structure/app/dsl/`. A transform class declares named inputs with
`input(Schema)`, named outputs with `output(Schema)`, and public schema-returning methods as subtransforms. The
`@transform` decorator serves two roles: on classes it marks transform definitions; on methods it optionally binds one
input or output lane. `src/structure/app/dsl/logic/model/transforms/transform_api.py` validates and stores that
metadata.

`src/structure/app/compiler/frontend/logic/actions/CompileTransform.py` discovers methods in class source order,
resolves their type hints, creates one symbolic `RowScope`, invokes the method, validates its returned schema instance,
and builds `StepPlan`. A lane is a named internal DataFrame stream. The default lane is `df`; field-declared outputs
such as `accepted` and `rejected` are additional lanes. A method can consume only a declared input or lane available
earlier in source order.

Symbolic execution does not process real rows. `RowScope` and `InputScope` produce expression objects when fields are
accessed. `CompileContext` captures `where(...)` filters and `join_one(...)` joins while the user's method runs.
`StepPlan` in `src/structure/app/compiler/ir/logic/model/StepPlan.py` currently stores one source, one input schema, one
projection, and one output lane.

`LowerPySparkPlan` converts compiler IR to PySpark-neutral recipe records. `RunOnlinePySparkTransform` interprets those
recipes with live PySpark DataFrames, while `RenderPySparkTransformModule` writes equivalent generated source. Both
paths must remain semantically identical. Compiler traceability records the same step, join, filter, projection, and
validation relationships for explain and generated traceability output.

This plan uses two new internal terms:

An input binding connects one method parameter to one declared input or available output lane. It records the
parameter name, schema, source frame, symbolic scope name, ordinal, and whether it is the driving relation.

A step result connects one returned schema value to one destination lane. It records the result ordinal, output
schema, projection, lane name, and deterministic frame identifier. A single-result method has one step result and
retains compatibility accessors for code that expects singular step properties during migration.

Input inference should proceed in parameter order. For the first parameter, prefer the current `df` lane when its

class input descriptors through `self` for compatibility.

Update lane bookkeeping after symbolic execution. Every result receives a deterministic frame identifier; use the

Extend PySpark target recipes with the same structure. A step recipe should carry one driving frame, shared hooks,

In online execution, resolve every join right side from the complete `frames` mapping. Execute before hooks once, alias

Update traceability and explain data. A multi-result method should have one step dependency, one set of join and filter

## Idempotence and Recovery

Specification and test additions are additive and safe to repeat. Keep the old singular metadata and recipe accessors
until all production callers have migrated, then remove only compatibility code that has no remaining repository use.
This permits compiler, target, and runtime work to land incrementally while existing transforms stay green.

Do not represent one multi-result method as several fully repeated steps even as a temporary shortcut; that can produce
duplicate joins in real Spark plans and establish the wrong semantic contract. If plural target lowering is incomplete,
keep the compiler milestone green and fail target lowering with a focused unsupported-feature diagnostic until the
shared-prefix recipe is implemented.

If generated and online outputs diverge, use the backend-neutral step/result recipe as the authority. Compare the
generated source and online interpreter operation order, correct the consumer that differs, and add a parity regression
before proceeding.

If support for joining earlier output lanes proves incompatible with the current frame model, do not silently narrow
`inputs=` to original declarations. Record the discovery here, introduce a typed general frame reference in IR, and
