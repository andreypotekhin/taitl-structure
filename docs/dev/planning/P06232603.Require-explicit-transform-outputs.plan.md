## Purpose / Big Picture

Every Structure transform will declare its public result through one or more class fields created with `output(...)`.
Users will no longer need the class-level `to=` decorator option, and the same declaration model will cover both
single-output and multiple-output transforms. A single declared output remains concise because the final canonical
DataFrame lane automatically satisfies it when the schemas match.

## Decision Log

- Decision: Require output declarations before symbolic execution begins.
  Rationale: A missing public result contract is a transform-level structural error and should fail before method bodies
  are interpreted.
  Date/Author: 2026-06-23 / Codex.

- Decision: Keep automatic final-lane binding for exactly one declared output.
  Rationale: Explicitness is needed at the public contract boundary, not as repetitive method metadata for a linear
  single-output pipeline.
  Date/Author: 2026-06-23 / Codex.

- Decision: Keep `result.df` as a convenience alias for any sole named output.
  Rationale: Existing runtime ergonomics remain useful while the declared output name becomes authoritative.
  Date/Author: 2026-06-23 / Codex.

## Context and Orientation

`src/structure/app/dsl/logic/model/transforms/transform_api.py` implements the overloaded `@transform` decorator and
the `input(...)` and `output(...)` declaration functions. Class decorator options are stored on the transform class.

`src/structure/app/compiler/frontend/logic/actions/CompileTransform.py` turns a decorated class into `TransformPlan`
intermediate representation. Its `_outputs` method currently supports three forms: class-level `to=Schema`,
field-declared outputs, and implicit inference from the final compiled step.

`OutputDeclaration` is a descriptor collected by `Transform.__init_subclass__`. A lane is a named DataFrame stream.
The canonical linear lane is named `df`; explicitly branched output lanes use their `output(...)` field names.

## Plan of Work

First remove `to` from the class decorator allow-list. Add an early compiler check that a transform class owns at least
one collected output declaration, with a diagnostic that shows `name = output(Schema)` as the remedy. Simplify
`_outputs` by deleting both class-level and implicit-output branches while preserving automatic binding from `df` for
one declaration.

Then update all executable transforms in source fixtures and tests. Each linear transform receives one semantically
named output field matching its final schema. Existing multi-output transforms retain their declarations. Tests will
cover rejection of `to=`, rejection of a missing output declaration, sole-output automatic binding, and multiple-output
lane binding.

Finally revise public specifications and examples so `output(...)` is the only transform result declaration. Run the
focused specification tests, the full pytest suite, and `make build`. Record observed results here and move the plan to
the done directory.

## Validation and Acceptance

This transform must compile and expose both `result.df` and `result.published`:

    @transform
    class PublishOrders(Transform):
        orders = input(OrderRaw)
        published = output(OrderPublished)

        def publish(self, order: OrderRaw) -> OrderPublished:
            return OrderPublished.base(order)()

A transform with no `output(...)` field must fail with `DSL-E0402` and tell the author to declare
`name = output(Schema)`. `@transform(to=Schema)` must fail as an unknown class option. Multiple declared outputs must
continue to require explicit method-level lane production.

## Interfaces and Dependencies

The public result declaration is:

    output(schema: type[Structure]) -> OutputDeclaration

The class decorator accepts only `validate_intermediate` and `streaming_compatible`. Method-level `@transform` continues
to accept `input`, `inputs`, `output`, and `outputs`.
