# Make Online Execution the v1 Default

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and
`Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows `docs/dev/auto/Plans.md`. It is self-contained so a contributor can design, document, and

## Purpose / Big Picture

Structure should let users run schema-driven PySpark transforms directly from their application code without committing
generated PySpark files. After this work, a developer can create a lightweight `StructureSession`, construct a transform
invocation with input DataFrames, and execute it online:

    session = StructureSession(spark=spark, ctx=ctx)

    result = EnrichOrders(
        orders=orders_df,
        customers=customers_df,
        products=products_df,
    ).run(session)

The transform instance is a deferred invocation: constructing it binds the named input DataFrames, but does not start
compilation or Spark work. Calling `.run(session)` asks the session to choose the configured runtime runner and execute
the transform. Generated PySpark remains useful for provenance, review, debugging, and generated-mode projects, but it
is not the default operational artifact in v1.

## Decision Log

- Decision: Make online execution the v1 default through `execution_mode = "online"`.
  Rationale: The project has no legacy generated-code users, and the desired product experience is direct runtime use
  of Structure transforms without committing generated PySpark.
  Date/Author: 2026-06-19 / Codex

- Decision: Support `execution_mode = "generated"` as the optional alternate mode.
  Rationale: Generated PySpark remains valuable for provenance, review, debugging, CI diff checks, and teams that want
  checked-in generated artifacts.
  Date/Author: 2026-06-19 / Codex

- Decision: Treat transform instances as deferred invocations.
  Rationale: `EnrichOrders(orders=orders_df, customers=customers_df).run(session)` lets users bind inputs when
  convenient and defer execution until a runtime session is available. It also avoids passing Spark or runtime config
  into the transform constructor.
  Date/Author: 2026-06-19 / Codex

- Decision: Make `StructureSession` the keeper of runtime configuration and runner selection.
  Rationale: Session-owned configuration gives Structure one place to hold Spark, optional `ctx`, resolved Structure
  configuration, target backend, execution mode, compiled-plan cache, and future backend choices.
  Date/Author: 2026-06-19 / Codex

- Decision: Keep `StructureSession` lightweight by delegating execution to runner components.
  Rationale: A session should coordinate runtime state, not become a large execution framework. `OnlinePySparkRunner`,
  `GeneratedPySparkRunner`, and `RuntimeTargetRegistry` keep responsibilities focused.
  Date/Author: 2026-06-19 / Codex

- Decision: Lower online execution from compiler IR directly to live PySpark operations instead of executing generated
  Python source text.
  Rationale: Executing generated text at runtime would add import-path, source-string, and security complexity. IR is
  the semantic contract, so online and generated execution should share IR semantics while using different consumers.
  Date/Author: 2026-06-19 / Codex

- Decision: Preserve Spark-free compiler commands.
  Rationale: Online execution is a runtime boundary. `structure check`, `structure compile`, and
  `structure compile --fail-on-diff` must remain fast and usable without PySpark, Java, SparkSession, Spark startup, or
  a Spark cluster.
  Date/Author: 2026-06-19 / Codex

## Context and Orientation

Structure lets users define schema classes and transform classes in ordinary Python source. The compiler discovers
those classes, symbolically executes schema-returning methods, builds backend-neutral intermediate representation
called IR, checks that the transform can be expressed as optimizer-visible Spark operations, and emits PySpark code.

The current v1 docs present generated PySpark classes as the main runtime surface. This plan changes that product
model. The compiler frontend remains essential, but the IR now has two v1 consumers:

- the generated PySpark emitter, which writes deterministic Python files for provenance, review, and generated-mode
  execution;
- the online PySpark runner, which executes the same transform semantics directly at runtime without writing generated
  files.

Terms used in this plan:

- Online execution: running a Structure transform directly from source metadata and compiler IR at application runtime,
  without importing checked-in generated transform classes.
- Generated execution: running a checked-in generated PySpark class such as `EnrichOrdersGenerated`.
- Transform invocation: an instance of a transform class that stores named input DataFrames and can later be run through
  a `StructureSession`.
- `StructureSession`: the public runtime object that owns Spark, optional context, resolved Structure config, target
  backend selection, and runner delegation.
- Runner: an internal component that executes a transform invocation for a specific execution mode and backend.
- IR: the compiler's intermediate representation, such as `TransformPlan`, `StepPlan`, operations, hooks, validation,
  and expression nodes.

Implement the config model. Add `execution_mode` with default `online`. Validate allowed values. Make config resolution
use the existing precedence model: CLI flags, `[tool.structure]` in `pyproject.toml`, `structure.toml`, then defaults.
If CLI flags exist for runtime mode, they must override file config. The config object should expose both
`execution_mode` and target backend values to `StructureSession`.

Implement the public runtime API. `StructureSession` should accept keyword-only constructor arguments:

    StructureSession(*, spark, ctx=None, config=None)

The `spark` object is required for PySpark execution. `ctx` is optional and is passed to hooks. `config` may be either a
resolved Structure config object or a mapping accepted by the config loader. If `config` is omitted, the session loads
project config using the normal resolution rules from the current working directory.

Implement transform input binding. `Transform.__init__(**inputs)` stores declared runtime inputs on the instance
without running compilation or Spark actions. It must reject positional arguments. It must reject input names that are
not declared with `input(...)`. It must reject missing declared inputs at run time with a structured diagnostic. It
should permit construction with a partial input set only if the implementation deliberately supports late binding; for
v1, prefer requiring all declared inputs by the time `.run(session)` executes.

Implement:

    def Transform.run(self, session):
        return session.run(self)

If user source defines a public schema-returning method named `run`, the compiler must reject it with a diagnostic
explaining that `run` is reserved for online execution and suggesting a method name such as `execute`, `normalize`, or
`publish`.

Implement runtime runner selection. Add a registry that maps:

    ("online", "pyspark") -> OnlinePySparkRunner
    ("generated", "pyspark") -> GeneratedPySparkRunner

`StructureSession.run(transform)` should ask the registry for the runner matching the resolved config, then delegate.
The session should hold a small in-memory cache keyed by transform class, source fingerprint when available, resolved
config fingerprint, Structure version, execution mode, target backend, and target PySpark range. The first
implementation may be conservative and compile on every run, but the API should leave room for caching without changing
user code.

Implement `OnlinePySparkRunner`. It should compile the transform class to `TransformPlan` using the same compiler
frontend used by `structure check` and `structure compile`, then execute the plan directly with live PySpark calls. It
must not write generated files and must not execute generated Python source text. The execution order must match
generated code:

1. Validate all declared input DataFrames.
2. Create `HookInputs` only when at least one hook declares `pass_inputs=True`.
3. For each source-ordered step, run `@before` hooks, compiled filters, joins, projections, `@after` hooks, and
   configured schema validation.
4. Apply hook `schema_mode` and `project_output` rules.
5. Validate and return the final DataFrame.

Factor PySpark lowering carefully. If practical, extract a shared semantic lowering layer so generated emission and
online execution cannot drift. Text-generation details such as import ordering remain in the generated emitter.
Semantic decisions such as expression lowering, join aliasing, validation placement, hook order, and projection shape
must be shared or covered by parity tests.

Implement `GeneratedPySparkRunner`. It should import the generated class for the transform invocation, instantiate it
with `spark=session.spark` and `ctx=session.ctx`, and call `run(**transform_inputs)`. If generated files are missing,
stale, or not importable, fail with a diagnostic that tells the user either to run `structure compile` or switch to
`execution_mode = "online"`.

Add diagnostics. Runtime compile failures should use the same structured compiler diagnostics as `structure check`.
Runtime execution diagnostics should include transform class name, execution mode, target backend, input name when
applicable, problem, suggested fix, and a link to `docs/specifications/OnlineExecution.md` or
`docs/Configuration.md`.

Documentation edits are additive or replacement edits and can be repeated safely. If a doc update creates conflicting

## Interfaces and Dependencies

Public API to expose from `structure`:

    class StructureSession:
        def __init__(self, *, spark, ctx=None, config=None): ...
        def run(self, transform): ...

    class Transform:
        def __init__(self, **inputs): ...
        def run(self, session): ...

Internal interfaces:

    class RuntimeTargetRegistry:
        def runner_for(self, *, execution_mode, target_backend): ...

    class OnlinePySparkRunner:
        def run(self, *, session, transform): ...

    class GeneratedPySparkRunner:
        def run(self, *, session, transform): ...

The online runner depends on the compiler frontend and PySpark runtime. The generated runner depends on generated
modules being importable. Compiler commands must not depend on PySpark.

`StructureSession` should not start or configure Spark. The caller supplies an existing Spark session. Spark SQL
assumptions from config may be validated or reported by runtime diagnostics, but v1 should not silently mutate Spark
session settings.
