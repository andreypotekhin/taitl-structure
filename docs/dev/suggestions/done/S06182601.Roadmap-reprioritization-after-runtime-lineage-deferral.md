### S06182601 Roadmap Reprioritization After Runtime Lineage Deferral

Runtime LDJSON lineage is now correctly deferred beyond v4, which leaves the active roadmap focused on Structure's
compiler-first value: typed source transforms, clean generated PySpark, schema validation, compiler provenance, static
dataflow, and strong diagnostics. The recommended adjustment is to spend the freed attention on end-user adoption
features in v1, keep v2 centered on richer Spark algebra and manual optimization, and split v3 into streaming
orchestration while moving backend expansion to v4.

- Keep v1 focused on the compiler core, but add diagnostic codes with documentation links, a `structure doctor`
  environment/configuration check, and polished starter examples. These are not glamorous, but they make the
  library feel intelligent: when setup, import safety, PySpark targeting, generated-code drift, or validation fails,
  Structure should point the user to the exact fix and relevant documentation.
- Treat compact compiler provenance and compact static dataflow as v1, but move richer field-level explain views,
  dependency graph exports, and visualization-friendly output to v2. v1 needs enough lineage to explain generated code
  and diagnostics; v2 can make lineage analytical.
- Move production incremental compilation out of the v1 must-have list and into v2, while preserving v1 architecture
  hooks for source hashing and deterministic generated output. v1 should ship with reliable full-project compile,
  `check`, and `compile --fail-on-diff`; `compile --changed-only`, cache invalidation policies, and cache diagnostics
  are better v2 work because they add subtle correctness risk.
- Keep `join_one(...)` in v1, including composite equi-keys, `Join.LEFT`, `Join.INNER`, broadcast hinting, right-side
  projection, and duplicate-right-side warnings. Move `join_many(...)`, semi/anti joins, non-equi joins, and richer
  join strategy controls to v2. Row-multiplying semantics are useful, but they complicate validation, lineage, and user
  expectations enough to deserve a separate stage.
- Keep v1 streaming compatibility narrow: generated transforms may accept a streaming current DataFrame only when the
  compiled operations are known to be Spark Structured Streaming-compatible. Do not add streaming source/sink lifecycle
  concepts before v3. Add a visible compile report flag that says whether a transform is streaming-compatible and why.
- Prioritize v2 in this order: windowing and deduplication helpers first, typed aggregations second, higher-order
  array/map expressions third, then manual optimization directives. End users commonly need latest-record selection,
  ranking, and dedupe before they need grouping sets or elaborate persistence controls.
- Add repartition/coalesce hints explicitly to the v2 public roadmap, because they already appear in the project
  management roadmap and are a natural companion to caching and persistence directives. Keep all such controls explicit
  in source and obvious in generated code.
- Split later roadmap stages: v3 full streaming orchestration (`readStream`, `writeStream`, triggers, checkpoints,
  watermarks, output modes, and state policies) and v4 Spark Connect. Spark Connect should not compete with streaming
  semantics unless it can be added behind the existing emitter boundary without changing the public DSL or generated
  class contract.
- Add an end-user feature not yet visible enough in the roadmap: generated documentation artifacts for schemas and
  transforms. A command such as `structure docs` could produce Markdown or JSON showing schemas, fields, nullability,
  inputs, outputs, hooks, validation policy, and lineage summary. This should be v2 unless trivial pieces fall out of
  v1 compiler metadata, because it reuses the same IR and lineage without expanding runtime responsibility.
- Add another user-facing feature for v2: a pytest helper or plugin that runs `structure check`, verifies generated code
  freshness, and offers small snapshot helpers for generated PySpark. Build integration exists in v1, but test
  ergonomics will strongly affect adoption by teams that want generated code committed and reviewed.

Suggested stage shape:

- v1 keeps compiler core, schemas, projection/filtering, hooks, `join_one`, validation, compact compiler lineage,
  compact static dataflow, `check`, `compile`, `explain`, `--fail-on-diff`, streaming compatibility reporting,
  diagnostic documentation links, and a setup doctor. Move production incremental compile, `join_many`, and rich
  lineage views later.
- v2 adds windowing, dedupe, aggregations, higher-order array/map functions, manual optimization hints,
  repartition/coalesce, richer explain/lineage, a pytest helper, and generated docs. Keep streaming lifecycle and
  Spark Connect out.
- v3 adds streaming source/sink orchestration, triggers, checkpoints, watermarks, output modes, and state policies.
- v4 adds Spark Connect if the core contract is stable. Runtime LDJSON lineage remains beyond v4.

The north star should be ruthless: v1 proves that Structure can replace hand-maintained PySpark boilerplate with a
strict, readable compiler workflow. v2 makes that workflow useful for mainstream analytical pipelines. v3 takes
ownership of job lifecycle concerns only after the transform compiler has earned trust. v4 adds Spark Connect only
after the ordinary PySpark and streaming contracts are stable.
